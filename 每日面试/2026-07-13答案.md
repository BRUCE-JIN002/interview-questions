# 2026-07-13 全栈偏前端每日面试参考答案

## 1. Proxy 与 Reflect：响应式系统的代理不变量

**参考答案：** `Reflect` 以默认对象语义执行操作，正确传递 `receiver`，并返回 `set` 是否成功。Proxy 不能谎报不可配置属性：例如目标对象的不可写、不可配置数据属性已有值，`set` trap 不能返回 `true` 并假装改写成功，否则会抛 `TypeError`。响应式实现应先追踪依赖，再用 `Reflect.set`；新增/修改、数组长度等应做精确触发。

**关键考点：** receiver、严格模式 `set` 返回值、不可配置属性、依赖追踪与触发分离。

**追问：** 为什么 getter 里直接 `target[key]` 会在继承访问器场景出错？

## 2. TypeScript `satisfies`、`as const` 与类型断言

**参考答案：**

```ts
type Route = { path: `/${string}`; permission: 'order:read' | 'order:write' };
const routes = [
  { path: '/orders', permission: 'order:read' },
] as const satisfies readonly Route[];
```

`satisfies` 做形状检查但保留推断出的字面量类型；`as const` 使值只读并收窄字面量；`as T` 是强制断言，可能掩盖不兼容数据，不能替代校验。

**关键考点：** 上下文类型、字面量保留、只读、断言逃逸。

**追问：** 若允许扩展字段，如何让 `Route` 同时检查必填字段？

## 3. ES2024 `Object.groupBy` 与数据分组边界

**参考答案：** `Object.groupBy` 将键转为属性键，适合字符串/symbol 分类；`Map.groupBy` 保留对象键身份。基础实现使用无原型对象，避免 `__proto__` 污染。

```ts
function groupBy<T>(items: Iterable<T>, keyOf: (x: T, i: number) => PropertyKey) {
  const out: Record<PropertyKey, T[]> = Object.create(null);
  let i = 0;
  for (const item of items) {
    const key = keyOf(item, i++);
    (out[key] ??= []).push(item);
  }
  return out;
}
```

`for...of` 遵循迭代语义；数组空洞是否跳过需明确与原生方法的兼容目标。

**关键考点：** PropertyKey、Map 对象键、原型污染、迭代器。

## 4. 浏览器渲染：避免强制同步布局

**参考答案：** 同一任务中先写几何样式再读布局信息会迫使浏览器提前 style/layout，循环后形成 layout thrashing。拖拽优先用 `transform`，将读操作集中在写之前；每帧只在 `requestAnimationFrame` 写一次，必要时缓存尺寸或用 `ResizeObserver` 更新。用 Performance 面板确认 Layout 时长、长任务和 INP 改善。

**关键考点：** 渲染流水线、读写分离、合成层、INP。

**追问：** `getBoundingClientRect`、`scrollTop` 哪些读取可能触发布局？

## 5. HTTP 缓存：`stale-while-revalidate` 的页面体验设计

**参考答案：** 内容列表可设较短 `max-age` 加 `stale-while-revalidate`，过期后先返回旧副本并后台验证；短期源站故障可用有限 `stale-if-error`。结算价格使用 `no-store` 或极短缓存并在提交前由服务端以权威价格重算，不能只信页面值。前端对列表可标注“刚刚更新/刷新中”，对价格则显式刷新或阻止过期提交。

**关键考点：** 缓存可容忍性按资源分级、服务端权威校验、陈旧数据可见性。

## 6. React 列表 key、状态保留与重置

**参考答案：** React 在相同父级下按组件类型和 key 对齐 fiber；用 index 做 key 时，筛选/插入改变位置，局部 state 会复用到另一订单。应使用不可变业务 ID；订单变更时需要重置编辑器，可让编辑组件的 key 包含订单 ID，或在受控状态中显式按 ID 初始化。不要用随机 key，它会导致每次重挂载。

**关键考点：** reconciliation、稳定身份、受控/非受控状态。

## 7. Vue 3 `effectScope` 在可复用逻辑中的资源回收

**参考答案：** 将模块内响应式副作用置入 scope；停止时统一 `scope.stop()`，`watch` 会停止。

```ts
const scope = effectScope();
const stop = scope.run(() => watch(query, fetchData))!;
function dispose() { stop(); socket.close(); clearInterval(timer); scope.stop(); }
```

WebSocket、timer、DOM listener、进行中的 fetch 不属于 Vue reactive effect，仍需分别关闭、清计时器、移除监听和 abort。

**关键考点：** 组件 scope 与 detached scope、watch 清理、外部资源所有权。

## 8. 手写：支持组合取消的 `AbortSignal.any` polyfill

**参考答案：** 空数组返回不会 abort 的 signal；先扫描已 abort 信号，再为其余信号加一次性监听。abort 后移除所有监听，保留第一个 reason。

```ts
function abortSignalAny(signals: AbortSignal[]) {
  const c = new AbortController();
  if (!signals.length) return c.signal;
  const cleanup = () => signals.forEach(s => s.removeEventListener('abort', onAbort));
  const onAbort = (e: Event) => { if (!c.signal.aborted) c.abort((e.target as AbortSignal).reason); cleanup(); };
  for (const s of signals) {
    if (s.aborted) { c.abort(s.reason); return c.signal; }
  }
  signals.forEach(s => s.addEventListener('abort', onAbort, { once: true }));
  return c.signal;
}
```

**关键考点：** 首个 reason、同步已取消、once 不等于其他信号监听自动释放。

## 9. 算法：编辑距离

**参考答案：** `dp[i][j]` 表示 `word1` 前 `i` 个字符转为 `word2` 前 `j` 个字符的最少操作。相同字符继承左上；不同字符取删除 `dp[i-1][j]`、插入 `dp[i][j-1]`、替换 `dp[i-1][j-1]` 的最小值加一。二维时间 `O(mn)`、空间 `O(mn)`；一维数组保留上轮左上值即可降为 `O(n)`。

**关键考点：** 边界 `dp[i][0]=i`、状态含义、滚动数组。

## 10. Next.js：Server Action 的鉴权、校验与重放防护

**参考答案：** Action 内重新读取 session，按服务端订单归属校验资源权限，以 schema 校验并规范化输入。提交携带幂等键，服务端用唯一约束/事务记录请求结果；数据库更新使用当前版本或条件更新。客户端禁用仅改善体验，不能作为安全控制。Action 只暴露最小输入，避免把敏感字段信任为表单值；审计用户、资源和结果。

**关键考点：** server-side authorization、IDOR、schema、幂等、乐观锁。

## 11. Node.js：背压下的 CSV 导出接口

**参考答案：** 数据库用 keyset cursor/stream 分批拉取，不一次性 `await` 全量数组；将行转换为 CSV 的 Transform 与 `res` 用 `pipeline` 串联，`write()` 返回 false 时等待 drain 形成背压。监听请求 close/abort 并取消 DB cursor 和 pipeline；导出任务记录 traceId、行数、耗时、失败原因。超大或可重试导出可改为后台任务写对象存储。

**关键考点：** `stream.pipeline`、背压、取消传播、资源上限。

## 12. NestJS：Webhook 验签、幂等与异步处理

**参考答案：** 在验签路由保留 raw body，用原始字节与时间戳计算 HMAC，校验签名和重放窗口后才解析/执行业务。以 provider event id 建唯一索引去重；支付状态迁移须校验允许的前序状态，事务内写 inbox/outbox。HTTP 只在持久化成功后快速 2xx，后续队列消费者幂等处理；记录验签失败、重复、乱序指标。

**关键考点：** raw body、时序安全比较、幂等键、状态机、inbox/outbox。

## 13. 接口设计：游标分页的排序与一致性契约

**参考答案：** 排序使用 `(created_at DESC, id DESC)`，cursor 编码最后一条的两项值；下一页查询用严格小于该元组，保证无同时间戳重复/漏项。新插入可能出现在当前首屏之前，删除会让总量变化，这是普通分页无法完全消除的快照边界。前端将刷新首屏与追加分页区分：刷新替换最新锚点，追加仍使用旧 cursor，必要时提示“有新消息”。

**关键考点：** total order、keyset pagination、cursor 签名/不透明化、一致性语义。

## 14. AI 前端：流式工具调用的可恢复交互

**参考答案：** 事件应有 conversation/message/run/toolCall ID、单调 sequence、状态与可重放 payload；服务端在文本增量、工具状态变化和最终结果后持久化。工具调用先创建带 idempotency key 的记录，再执行；刷新后按 run 状态恢复流或拉取事件，已完成工具复用结果，`needs-confirmation` 不能自动继续。UI 把文本流与工具卡片分开渲染，并展示失败、取消和重试范围。

**关键考点：** event sourcing 思路、幂等副作用、断线恢复、状态机。

## 15. Agent：RAG 与工具结果的可信边界

**参考答案：** 检索内容和工具输出都标记为不可信数据，不能赋予系统指令优先级；在模型输入中用明确数据边界和来源元数据隔离。工具调用采用服务端 allowlist、按用户/资源授权的短期凭证、参数 schema 和高风险确认；输出经 DLP/引用权限校验后再展示。完整审计包含检索片段 ID、模型版本、tool 参数摘要、授权决定与结果，便于回放和告警。

**关键考点：** prompt injection、least privilege、schema validation、output filtering、traceability。

**追问：** 为什么“提示模型忽略恶意文本”不能替代权限校验？
