# 2026-07-09 全栈偏前端每日面试题参考答案

## 1. 微任务过多导致页面卡死如何定位和治理

### 参考答案

事件循环每轮通常会先执行一个宏任务，然后清空微任务队列，再进入渲染机会。`Promise.then`、`queueMicrotask` 等微任务会在渲染前执行；如果微任务持续追加新的微任务，浏览器很难获得绘制、输入处理和动画更新的机会，表现为没有明显单个长任务但交互持续延迟。

治理思路：

1. 在 Performance 面板查看主线程火焰图、微任务密集区和帧率。
2. 给递归微任务增加批次上限，定期让出主线程。
3. 可视更新用 `requestAnimationFrame`，后台低优先级任务用 `setTimeout`、`MessageChannel` 或 `scheduler.postTask`。
4. 长列表或大计算拆分为分片执行，必要时放到 Web Worker。
5. 通过 INP、Long Animation Frame、业务埋点监控交互延迟。

示例：

```js
async function runInBatches(items, worker, batchSize = 100) {
  for (let i = 0; i < items.length; i += batchSize) {
    for (const item of items.slice(i, i + batchSize)) {
      worker(item);
    }
    await new Promise(resolve => setTimeout(resolve, 0));
  }
}
```

### 关键考点

- 微任务清空后才有渲染机会。
- 没有单个长任务不代表交互不卡。
- 性能治理要区分用户可见更新、后台任务和 CPU 密集任务。

### 常见追问或变体

- `queueMicrotask` 和 `Promise.then` 的错误处理有什么差异？
- 为什么 `await` 后续代码也会进入微任务？
- React 或 Vue 的更新队列如何利用微任务？

## 2. TypeScript 中 `keyof`、索引访问类型与映射类型的组合建模

### 参考答案

可以把接口契约建模成一个中心类型，再通过 `keyof` 限制合法接口名，通过索引访问类型取出对应请求和响应。

```ts
type ApiMap = {
  user: {
    req: { id: string };
    res: { id: string; name: string };
  };
  order: {
    req: { orderId: string };
    res: { orderId: string; status: "paid" | "pending" };
  };
};

type ApiName = keyof ApiMap;
type ApiReq<K extends ApiName> = ApiMap[K]["req"];
type ApiRes<K extends ApiName> = ApiMap[K]["res"];

declare function request<K extends ApiName>(
  name: K,
  payload: ApiReq<K>
): Promise<ApiRes<K>>;

request("user", { id: "u1" }).then(res => res.name);
```

`keyof` 把对象类型的键转成联合类型；`T[K]` 根据键取字段类型；映射类型适合批量生成客户端方法。

```ts
type Client<T extends Record<string, { req: unknown; res: unknown }>> = {
  [K in keyof T]: (payload: T[K]["req"]) => Promise<T[K]["res"]>;
};
```

### 关键考点

- 类型中心化比散落的函数重载更可维护。
- 泛型参数 `K` 要贯穿接口名、请求和响应。
- 映射类型适合从契约生成调用端类型。

### 常见追问或变体

- 如何让接口名支持 `/user/:id` 这类模板字符串？
- 如何从 OpenAPI 生成类似类型？
- 为什么 `K extends string` 不如 `K extends keyof ApiMap` 精确？

## 3. 手写支持最大重试次数、指数退避和抖动的请求重试函数

### 参考答案

重试只适合临时性失败，例如网络抖动、`429`、`503`、网关超时；业务校验失败、鉴权失败、参数错误不应盲目重试。

```js
const sleep = ms => new Promise(resolve => setTimeout(resolve, ms));

async function retry(fn, options = {}) {
  const {
    retries = 3,
    baseDelay = 200,
    maxDelay = 3000,
    shouldRetry = () => true,
  } = options;

  let lastError;

  for (let attempt = 0; attempt <= retries; attempt++) {
    try {
      return await fn({ attempt });
    } catch (error) {
      lastError = error;
      if (attempt === retries || !shouldRetry(error, attempt)) {
        throw error;
      }

      const exponential = Math.min(maxDelay, baseDelay * 2 ** attempt);
      const jitter = Math.random() * exponential * 0.3;
      await sleep(exponential + jitter);
    }
  }

  throw lastError;
}
```

生产环境还要配合请求超时、取消、幂等键、全局重试预算和服务端限流响应头。

### 关键考点

- 指数退避降低持续打满上游的风险。
- 随机抖动避免客户端同时重试。
- 非幂等写操作需要幂等键或明确禁止自动重试。

### 常见追问或变体

- 如何支持 `AbortSignal`？
- 如何读取 `Retry-After`？
- 如何统计重试次数和最终失败原因？

## 4. CSS Container Queries 与响应式组件设计

### 参考答案

媒体查询根据视口变化，Container Queries 根据容器尺寸或样式变化。组件如果被复用到不同布局区域，容器查询更贴近组件自身可用空间。

```css
.product-card-wrapper {
  container-type: inline-size;
}

.product-card {
  display: grid;
  gap: 12px;
}

@container (min-width: 420px) {
  .product-card {
    grid-template-columns: 120px 1fr;
  }
}
```

设计时应把“组件外部布局”和“组件内部形态”分开：外部决定宽度和位置，内部通过容器查询切换图片比例、信息密度、按钮排列。避免在太多嵌套节点上开启容器，减少样式复杂度和调试成本。

### 关键考点

- 组件复用场景优先考虑容器尺寸。
- 容器查询需要显式建立 containment。
- 它不是媒体查询替代品，页面级断点仍然有价值。

### 常见追问或变体

## 5. React Context 频繁更新引发大面积重渲染

### 参考答案

Context 的 `value` 一旦变成新引用，所有消费该 Context 的组件都会重新渲染。把用户、权限、主题和筛选条件放在同一个 Context 中，会让高频状态污染低频状态。

治理方案：

1. 按变化频率和业务边界拆分 Context。
2. `value` 使用 `useMemo` 保持引用稳定。
3. 高频、细粒度状态放到外部 store，通过 `useSyncExternalStore` 或状态库选择器订阅。
4. 把 action 和 state 分开，避免只使用 dispatch 的组件被 state 更新牵连。
5. 对昂贵子树使用组件拆分和 `memo`，但不要把 `memo` 当根因修复。

```tsx
const UserContext = createContext(null);
const FilterContext = createContext(null);

function UserProvider({ user, children }) {
  const value = useMemo(() => ({ user }), [user]);
  return <UserContext.Provider value={value}>{children}</UserContext.Provider>;
}
```

### 关键考点

- Context 适合低频全局数据，不适合所有状态。
- 变化频率是拆分 Context 的重要标准。
- 选择器订阅能减少无关字段变化带来的渲染。

### 常见追问或变体

- 为什么 `React.memo` 挡不住 Context 更新？
- `useSyncExternalStore` 解决了什么一致性问题？
- 权限数据应该放 Context、路由层还是服务端？

## 6. Vue 3 `flush` 时机与 DOM 读取一致性

### 参考答案

Vue 默认 watcher 通常在组件 DOM 更新前执行，此时读取 DOM 可能拿到旧值。

- `flush: "pre"`：默认时机，适合派生状态、请求参数、普通副作用。
- `flush: "post"`：组件 DOM 更新后执行，适合读取布局、滚动定位、动画测量。
- `flush: "sync"`：同步执行，适合极少数需要立即响应的逻辑，但容易造成重复触发和性能问题。

```ts
watch(
  () => props.open,
  async open => {
    if (!open) return;
    await nextTick();
    const rect = panelRef.value?.getBoundingClientRect();
    updatePosition(rect);
  },
  { flush: "post" }
);
```

表单校验一般用默认或 post，取决于是否依赖 DOM；动画测量用 post；日志埋点如果不依赖 DOM，默认即可。

### 关键考点

- watcher 执行时机和 DOM 更新批处理有关。
- 读取 DOM 尺寸通常需要 `post` 或 `nextTick`。
- `sync` 要谨慎，可能破坏批处理收益。

### 常见追问或变体

- `watchEffect` 与 `watchPostEffect` 的区别？
- `nextTick` 解决的是哪一层异步？
- 如何清理 watcher 中的异步请求？

## 7. Fetch Priority 与关键资源加载优化

### 参考答案

浏览器会根据资源类型、发现时机和上下文推断优先级。LCP 图片如果在 JS 渲染后才出现，即使体积不大也会被晚发现。`preload` 解决提前发现，`fetchpriority="high"` 提示优先级，二者可以配合但不能滥用。

```html
<link rel="preload" as="image" href="/hero.webp" fetchpriority="high">
<img src="/hero.webp" fetchpriority="high" width="1200" height="640" alt="">
```

优化方案：

1. 首屏 LCP 图直接出现在 HTML 中，避免依赖晚执行 JS。
2. 只给一个最关键 LCP 资源高优先级。
3. 非首屏图片继续懒加载。
4. 字体、CSS、首屏图按实际瀑布图调优，避免所有资源都 high。
5. 用 RUM 验证 LCP、TTFB、资源加载时间变化。

### 关键考点

- `preload` 主要解决发现早晚，`fetchpriority` 影响相对优先级。
- 优先级提示不是强制命令。
- 滥用 high 会挤占 CSS、字体等关键资源。

### 常见追问或变体

- `loading="lazy"` 和 `fetchpriority="high"` 同时使用是否合理？
- CSS 背景图如何优化 LCP？
- HTTP/2 或 HTTP/3 下资源优先级还重要吗？

## 8. 手写事件总线的 once、off 与异常隔离

### 参考答案

关键是保存原始回调和包装回调的关系，`emit` 时复制监听器快照，避免监听器增删影响当前派发。

```js
class EventBus {
  #events = new Map();

  on(type, fn) {
    if (!this.#events.has(type)) this.#events.set(type, new Set());
    this.#events.get(type).add(fn);
    return () => this.off(type, fn);
  }

  once(type, fn) {
    const wrapper = (...args) => {
      this.off(type, wrapper);
      return fn(...args);
    };
    wrapper.raw = fn;
    return this.on(type, wrapper);
  }

  off(type, fn) {
    const set = this.#events.get(type);
    if (!set) return;
    for (const item of set) {
      if (item === fn || item.raw === fn) set.delete(item);
    }
    if (set.size === 0) this.#events.delete(type);
  }

  emit(type, ...args) {
    const listeners = [...(this.#events.get(type) || [])];
    const errors = [];
    for (const listener of listeners) {
      try {
        listener(...args);
      } catch (error) {
        errors.push(error);
      }
    }
    if (errors.length) queueMicrotask(() => { throw new AggregateError(errors); });
  }
}
```

### 关键考点

- `once` 需要处理包装函数与原函数的映射。
- `emit` 使用快照保证派发稳定性。
- 异常隔离不能吞错，应有统一上报策略。

### 常见追问或变体

- 如何支持通配符事件？
- 如何限制最大监听器数量防止泄漏？
- 同步 emit 和异步 emit 如何取舍？

## 9. 算法：两个有序数组的中位数

### 参考答案

对较短数组二分，找到分割线，使左半部分元素数量等于右半部分，且 `left1 <= right2`、`left2 <= right1`。

```js
function findMedianSortedArrays(nums1, nums2) {
  if (nums1.length > nums2.length) {
    return findMedianSortedArrays(nums2, nums1);
  }

  const m = nums1.length;
  const n = nums2.length;
  let left = 0;
  let right = m;
  const half = Math.floor((m + n + 1) / 2);

  while (left <= right) {
    const i = Math.floor((left + right) / 2);
    const j = half - i;

    const left1 = i === 0 ? -Infinity : nums1[i - 1];
    const right1 = i === m ? Infinity : nums1[i];
    const left2 = j === 0 ? -Infinity : nums2[j - 1];
    const right2 = j === n ? Infinity : nums2[j];

    if (left1 <= right2 && left2 <= right1) {
      if ((m + n) % 2 === 1) return Math.max(left1, left2);
      return (Math.max(left1, left2) + Math.min(right1, right2)) / 2;
    }

    if (left1 > right2) right = i - 1;
    else left = i + 1;
  }
}
```

复杂度是 `O(log(min(m,n)))`，空间复杂度 `O(1)`。

### 关键考点

- 只需要二分较短数组。
- 分割线条件比真实合并更高效。
- 边界用正负无穷简化空侧处理。

### 常见追问或变体

- 如何扩展为两个有序数组第 K 小？
- 为什么不能直接二分值域？
- 空数组和重复元素如何处理？

## 10. Next.js Route Segment Config 与动态渲染边界

### 参考答案

Route Segment Config 用于控制某个 layout、page 或 route handler 的渲染和缓存行为。常见配置包括：

- `dynamic`：控制是否强制动态或静态。
- `revalidate`：设置 ISR 重新验证时间。
- `fetchCache`：控制 fetch 默认缓存策略。
- `runtime`：选择 Node.js 或 Edge Runtime。

详情页应拆成稳定的商品内容和用户态内容。商品标题、图片、描述可以静态渲染或按标签失效；登录态价格、库存提醒、会员权益放到动态子组件或客户端请求中，避免一个用户态读取把整页拖成动态。

```tsx
export const revalidate = 300;

export default async function ProductPage({ params }) {
  const product = await getProduct(params.id);
  return (
    <>
      <ProductStatic product={product} />
      <UserPrice productId={params.id} />
    </>
  );
}
```

如果价格必须服务端鉴权读取，应明确该局部动态边界，避免缓存混入用户数据。

### 关键考点

- 动态函数和不缓存 fetch 会影响渲染模式。
- 静态内容与用户态内容要拆边界。
- 缓存策略必须和数据隐私匹配。

## 11. Node.js HTTP Agent、连接复用与上游雪崩

### 参考答案

HTTP Agent 负责管理客户端连接池。开启 keep-alive 可以复用 TCP 连接，降低握手成本；但 `maxSockets` 过大可能压垮上游，过小会在本服务内排队。空闲连接过久可能被上游关闭，复用时触发 `ECONNRESET`。

治理策略：

1. 为不同上游配置独立 Agent、超时和并发上限。
2. 设置合理的连接、请求、响应和整体 deadline。
3. 空闲连接生命周期小于上游 keep-alive 超时。
4. 对非关键上游做熔断、降级和请求合并。
5. 监控连接池排队、重试、超时、状态码和上游延迟。

```js
import http from "node:http";

const agent = new http.Agent({
  keepAlive: true,
  maxSockets: 100,
  maxFreeSockets: 20,
  timeout: 30_000,
});
```

### 关键考点

- 连接复用提高效率，但需要容量上限。
- 超时要分层，不能只依赖默认值。
- BFF 要保护上游，也要保护自己。

### 常见追问或变体

- `ECONNRESET` 一定是服务端 bug 吗？
- 如何设计重试和熔断的顺序？
- HTTP/2 连接复用会改变哪些参数？

## 12. NestJS 自定义参数装饰器与用户上下文传递

### 参考答案

Guard 负责鉴权并把解析后的用户上下文挂到 request；自定义参数装饰器负责从 request 中读取上下文，Controller 不再手写解析 token。

```ts
import { createParamDecorator, ExecutionContext } from "@nestjs/common";

export const CurrentUser = createParamDecorator(
  (field: string | undefined, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest();
    const user = request.user;
    return field ? user?.[field] : user;
  }
);
```

```ts
@Get("profile")
getProfile(@CurrentUser() user: UserContext) {
  return this.userService.getProfile(user.id);
}
```

业务层建议接收明确的 `UserContext` 或 `tenantId`，不要依赖 Express/Fastify request。测试时分别覆盖 Guard 注入、装饰器读取和 Controller 参数传递。

### 关键考点

- Guard 负责安全判断，Decorator 负责参数提取。
- 业务服务不应绑定 HTTP 框架对象。
- 用户上下文结构要稳定且可测试。

## 13. 数据库软删除、唯一约束与历史数据查询

### 参考答案

目标是“未删除用户邮箱唯一”，但保留历史删除记录。PostgreSQL 可以使用部分唯一索引：

```sql
CREATE UNIQUE INDEX users_email_active_uk
ON users (email)
WHERE deleted_at IS NULL;
```

MySQL 不支持同样形式的部分索引，常见方案是增加生成列或把唯一键扩展为 `(email, active_flag)`。

```sql
ALTER TABLE users
ADD active_email VARCHAR(255)
  GENERATED ALWAYS AS (
    CASE WHEN deleted_at IS NULL THEN email ELSE NULL END
  ) STORED,
ADD UNIQUE KEY users_active_email_uk (active_email);
```

查询历史记录时保留普通索引，例如 `(email, deleted_at)`。业务层所有默认查询都应带 `deleted_at IS NULL`，管理后台或审计接口显式允许查询历史。

### 关键考点

- 软删除会影响唯一约束语义。
- PostgreSQL 部分索引表达力更直接。
- MySQL 可以利用唯一索引允许多个 NULL 的特性。

### 常见追问或变体

- 恢复软删除数据时如何处理邮箱冲突？
- 软删除是否适合所有业务？
- ORM 的全局过滤条件有哪些风险？

## 14. AI 聊天前端的消息分片、撤销与重新生成

### 参考答案

不要只用线性数组覆盖消息。更稳妥的模型是“会话树 + 当前分支指针”。每条消息有稳定 id、parentId、role、status、chunks、version 和错误信息。

```ts
type Message = {
  id: string;
  parentId: string | null;
  role: "user" | "assistant" | "tool";
  status: "pending" | "streaming" | "done" | "error" | "aborted";
  content: string;
  chunks: string[];
  version: number;
};
```

重新生成时，从目标用户消息创建新的 assistant 子节点，而不是覆盖旧回答。编辑历史问题时，从该节点生成新分支，保留旧分支供回看。流式输出按 messageId 和 version 追加，收到过期 chunk 时丢弃。

状态机：

1. `pending`：请求已创建。
2. `streaming`：持续接收分片。
3. `done`：完整结束。
4. `aborted`：用户停止。
5. `error`：网络或模型失败，可重试。

### 关键考点

- 消息要有稳定 id 和版本，避免流式乱序覆盖。
- 重新生成是创建新分支，不是原地改历史。
- 停止、失败、重试都应是明确状态。

### 常见追问或变体

- 如何持久化会话树？
- 多模型并行回答如何组织 UI？
- 工具调用消息是否展示给用户？

## 15. Agent 工具调用的权限票据与审计链路

### 参考答案

Agent 工具权限应由宿主系统授予，不由模型自行决定。每次工具调用携带短期权限票据，票据包含用户、会话、工具名、允许动作、参数范围、过期时间和 requestId。

设计要点：

1. 工具注册时声明只读或写操作、敏感字段、审批要求。
2. 只读工具和写工具使用不同权限域，不能通过参数切换动作。
3. 写操作前展示人类可读确认页，确认内容来自后端规范化参数。
4. 敏感参数在日志中脱敏，但保留可追踪哈希或引用 id。
5. 审计日志记录模型建议、用户确认、工具输入摘要、工具结果和操作者。

伪代码：

```ts
type ToolTicket = {
  userId: string;
  sessionId: string;
  tool: "order.read" | "address.update" | "refund.request";
  scope: "read" | "write";
  expiresAt: number;
  requestId: string;
};
```

防止越权的关键是服务端按票据和工具元数据校验，不能相信模型输出的工具名、scope 或自然语言解释。

### 关键考点

- 模型只能提出调用意图，授权由系统完成。
- 只读和写操作要从工具定义层隔离。
- 审计链路要能复盘“谁在何时基于什么确认做了什么”。
