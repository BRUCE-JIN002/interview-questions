# 2026-06-30 全栈偏前端面试题答案

## 1. JavaScript 显式资源管理与异步释放

### 参考答案

`using` 声明的对象需实现 `Symbol.dispose`，离开当前词法作用域时自动同步释放；`await using` 还支持 `Symbol.asyncDispose`，适合异步关闭。多个资源按声明的逆序释放，类似栈。它把“获取资源”和“保证释放”绑定在语法结构中，减少多层 `try...finally` 的嵌套与遗漏，但不替代业务级错误处理。

前端可用于临时事件订阅、对象 URL 或锁；Node.js 可用于文件句柄、数据库连接或临时目录。业务异常与释放异常同时发生时，需要关注 `SuppressedError`：主错误保留，清理错误作为被抑制错误附带。

### 关键考点

- 词法作用域、逆序释放、同步与异步资源
- 资源对象必须遵循 disposable 协议
- 清理也可能失败，不能吞掉原始业务异常

### 常见追问或变体

- `using` 与 GC 有什么区别？
- 如何把旧式 `close()` API 包装成 disposable 对象？

## 2. TypeScript 函数类型的协变、逆变与双向协变

### 参考答案

返回值通常协变：能返回 `Dog` 的函数可以用于要求返回 `Animal` 的位置。参数在严格检查下逆变：调用方若允许传任意 `Animal`，实现就不能只接受 `Dog`，否则收到 `Cat` 会在运行时出错。因此 `(dog: Dog) => void` 不能赋给 `(animal: Animal) => void`。

`strictFunctionTypes` 对函数属性执行更严格的参数检查，但为兼容常见泛型容器，方法语法存在双向协变式的宽松行为。设计回调 API 时应准确表达输入边界，不应靠方法语法绕过安全检查。

### 关键考点

- 返回值协变、参数逆变
- 函数属性与方法声明的检查差异
- 类型兼容性最终要对应真实调用方向

### 常见追问或变体

- `readonly` 容器为何更容易安全协变？
- `unknown`、`never` 在函数参数中分别有什么意义？

## 3. 手写 `Promise.any`

### 参考答案

```ts
function promiseAny<T>(iterable: Iterable<T | PromiseLike<T>>): Promise<T> {
  return new Promise((resolve, reject) => {
    const errors: unknown[] = [];
    let total = 0;
    let rejected = 0;

    for (const item of iterable) {
      const index = total++;
      Promise.resolve(item).then(resolve, error => {
        errors[index] = error;
        rejected++;
        if (rejected === total) {
          reject(new AggregateError(errors, 'All promises were rejected'));
        }
      });
    }

    if (total === 0) {
      reject(new AggregateError([], 'All promises were rejected'));
    }
  });
}
```

结果数组按输入索引写入，因此不是按拒绝完成顺序排列。时间和额外空间均为 `O(n)`。普通值经 `Promise.resolve` 归一化；空输入立即 rejected。

### 关键考点

- 任一 fulfilled 即短路
- 全部 rejected 才拒绝，原因保持输入顺序
- 空输入和 thenable 归一化

### 常见追问或变体

- 与 `Promise.race`、`Promise.allSettled` 的差异是什么？
- 迭代器本身抛错时，如何做到与规范行为更接近？

## 4. React Server Components 的安全边界

### 参考答案

Server Component 的模块代码及其服务器依赖不会作为客户端 JavaScript 打包，但它返回给客户端的序列化结果会跨越 RSC 边界；传给 Client Component 的 props、错误消息、缓存结果和日志关联信息都可能暴露。服务端执行不等于自动安全。

每次数据读取都应在服务端基于当前身份做授权；只构造白名单 DTO，不把 ORM 实体直接下发；密钥只在服务器模块使用；共享缓存键必须包含会影响结果的租户、用户或权限维度，或只缓存公共数据。日志应脱敏。应通过静态检查、DTO 测试和不同身份的集成测试验证无越权。

### 关键考点

- 代码不下发与数据不下发是两回事
- 鉴权必须靠近数据访问，不能只靠页面入口
- 用户相关数据不能进入错误的共享缓存

### 常见追问或变体

- Server Action 是否可以信任 Client Component 传入的用户 ID？
- 如何使用 taint API 辅助发现敏感对象越界？

## 5. Vue 3 响应式解构与 `toRef` / `toRefs`

### 参考答案

`reactive` 依赖代理的属性读取和写入完成追踪。`const { count } = state` 只复制了当前基本值，之后读取不再经过代理；若解构值是已代理的嵌套对象，修改其内部字段仍会经过该对象的代理，所以看起来仍有响应性。

`toRef(state, 'count')` 创建与单个属性双向关联的 ref；`toRefs(state)` 批量转换可枚举属性，适合组合式函数暴露一组可解构状态；`computed` 适合派生值或需要受控 getter/setter 的接口。不要为了“保险”无差别使用 `toRefs`。

### 关键考点

- 响应性来自代理访问，不来自变量名
- 基本值复制与代理对象引用的差异
- 可写源状态和派生状态应区分

### 常见追问或变体

- props 解构在不同 Vue 编译器版本中有什么差异？
- `toRef(() => props.foo)` 与 `computed(() => props.foo)` 如何选择？

## 6. View Transition API 与无障碍降级

### 参考答案

同文档转场通过 `document.startViewTransition(updateCallback)`：浏览器捕获旧快照，回调中同步完成决定新视图的 DOM 状态更新，再捕获新快照并执行伪元素动画。共享元素需稳定且唯一的 `view-transition-name`。跨文档转场则要求同源页面选择启用，并通过页面导航连接前后快照。

导航应有递增 token 或可取消任务，旧请求完成时不能覆盖新页面。API 不可用时直接更新 DOM；`prefers-reduced-motion: reduce` 下关闭或显著缩短动画。转场是渐进增强，内容正确性和焦点管理不能依赖动画。

### 关键考点

- 快照、DOM 更新、动画三个阶段
- 唯一共享元素名称与并发导航治理
- 无障碍、焦点与渐进增强

### 常见追问或变体

- 异步数据应在 update callback 前还是内部获取？
- 大面积快照动画有哪些性能成本？

## 7. WebAuthn / Passkey 登录流程与威胁模型

### 参考答案

注册时服务端生成一次性 challenge 和 RP 配置；前端调用 `navigator.credentials.create`；认证器为 RP 创建密钥对并返回公钥凭证，服务端校验 challenge、origin、RP ID、用户验证标志及证明后保存公钥。登录时服务端再发 challenge，认证器用私钥签名，服务端用已存公钥验证。

challenge 防重放；RP ID 与 origin 绑定抵抗钓鱼；用户验证证明本地 PIN/生物验证已完成；签名计数器可作为克隆凭证的风险信号，但不能作为唯一判断。认证成功只能由持有账户公钥和 challenge 状态的服务端确认。接口返回应避免暴露账号是否存在，并让 challenge 短期、一次性消费。

### 关键考点

- 私钥不离开认证器，服务端保存公钥
- 服务端校验 challenge、origin、RP ID 和签名
- 账户枚举、重放与恢复流程同样重要

### 常见追问或变体

- Discoverable Credential 如何支持无用户名登录？
- Passkey 同步与传统硬件安全密钥的风险有何不同？

## 8. 用有限状态机治理复杂表单

### 参考答案

可建模为 `editing → validating → uploading → submitting → underReview → approved/rejected`，并把 `saveDraft`、`validationPassed`、`uploadFailed`、`reviewRejected` 等作为事件。重试次数、草稿 ID、上传进度属于上下文，不应全部膨胀为状态。

状态机只接受当前状态允许的事件，从结构上消除“已批准但仍在上传”等非法组合。每个异步调用关联 invocation ID 或 `AbortSignal`，离开状态后忽略旧结果。服务端保存业务状态和版本号，前端刷新后以服务端快照恢复；本地只保存未提交草稿。代价是前期建模和团队学习成本，因此简单线性表单无需引入。

### 关键考点

- 状态、事件和上下文的边界
- 取消或忽略过期异步结果
- 服务端状态是跨刷新恢复的权威来源

### 常见追问或变体

- 并行状态如何表达上传与草稿保存同时进行？
- 如何测试状态机的完整转移覆盖？

## 9. 算法：每日温度与单调栈

### 参考答案

```ts
function dailyTemperatures(temperatures: number[]): number[] {
  const answer = Array(temperatures.length).fill(0);
  const stack: number[] = [];

  for (let i = 0; i < temperatures.length; i++) {
    while (
      stack.length &&
      temperatures[i] > temperatures[stack[stack.length - 1]]
    ) {
      const previous = stack.pop()!;
      answer[previous] = i - previous;
    }
    stack.push(i);
  }

  return answer;
}
```

栈保存尚未找到更高温度的下标，并保持对应温度单调不增。当前温度更高时，连续弹出并计算距离。每个下标只入栈、出栈一次，因此时间 `O(n)`，空间 `O(n)`。

### 关键考点

- 栈存下标而非只存温度
- 单调性对应“下一个更大元素”
- 摊还复杂度分析

### 常见追问或变体

- 如何求循环数组中的下一个更大元素？
- 如果要求前一个更高温度，应如何调整遍历方向？

## 10. Next.js 按标签失效与并发更新一致性

### 参考答案

公共商品详情可按 `product:{id}`，分类列表按 `category:{id}`，私人购物车按用户隔离且通常不做跨用户共享缓存。商品更新后失效商品标签及受影响分类标签；后台更新可用允许陈旧再验证语义的 `revalidateTag`，用户提交后需要立刻读到新值时优先使用 `updateTag`；页面结构或无法精确枚举依赖时再用 `revalidatePath`。

标签过粗会造成缓存雪崩，过细则增加维护成本。并发写入必须由数据库版本、事务或乐观锁解决，缓存失效不能提供写一致性。写成功后再失效，并确保缓存键包含语言、区域、权限等影响输出的维度。

### 关键考点

- 标签围绕数据实体和依赖关系设计
- 读己之写与 stale-while-revalidate 语义不同
- 数据库负责写一致性，缓存负责读取性能

### 常见追问或变体

- 失效失败时如何通过 outbox 或重试补偿？
- 如何监控缓存命中率与失效风暴？

## 11. Node.js 线上内存泄漏定位

### 参考答案

先同时观察 RSS、`heapUsed`、`external`、`arrayBuffers`、GC 频率、句柄数和流量。`heapUsed` 稳定而 RSS 上涨，重点怀疑 Buffer/ArrayBuffer、原生扩展、线程栈、内存碎片或 allocator 未归还操作系统，而不只查 JS 堆。

在可控流量下复现并对比多份 Heap Snapshot 的 retained size；用 Allocation Profile 找持续分配路径；检查无界缓存、闭包、定时器、EventEmitter listener、未释放流和外部内存。原生侧可结合诊断报告、系统采样器和 allocator 指标。Heap Snapshot 会暂停主线程且可能临时需要接近双倍堆内存，应在副本、摘流实例或内存余量充足时采集。

### 关键考点

- RSS 不等于 V8 heap
- 通过趋势和差分快照找“持续保留”
- 诊断动作本身可能触发生产事故

### 常见追问或变体

- `--max-old-space-size` 为什么通常不是修复？
- 如何判断是泄漏还是高峰后内存未归还？

## 12. NestJS 多租户数据隔离

### 参考答案

租户 ID 必须来自已验证 token、session 或服务端 API key 映射，而不是请求 body/header 中任意值。Guard 验证身份与租户成员关系，把不可变的租户上下文放入请求级上下文；Service/Repository 从上下文读取，ORM 通过强制 scope、查询扩展或数据库 Row-Level Security 自动追加 `tenant_id`。

唯一索引通常应包含 `tenant_id`。审计日志记录操作者、租户、资源和 trace ID。后台任务的消息体携带已签名或服务端生成的租户标识，消费者重新加载授权上下文。测试需覆盖跨租户 ID 猜测、关联查询、批量操作和管理员例外路径；高风险场景优先使用数据库 RLS 作为纵深防御。

### 关键考点

- 租户上下文必须来自可信身份
- 隔离应在数据访问层默认生效
- Web 请求、后台任务和审计链路都要携带租户

### 常见追问或变体

- 请求级 provider 有什么性能成本？
- 超级管理员跨租户查询如何显式授权和审计？

## 13. GraphQL N+1 与请求级 DataLoader

### 参考答案

Resolver 不逐篇查作者，而是调用 `authorLoader.load(authorId)`。同一事件循环批次内的 key 被合并为一次查询，批处理函数返回与输入 key 等长、同顺序的结果数组。数据库返回顺序通常不可靠，需要先建 `Map` 再按 keys 映射。

实例必须请求级，避免一个用户查询到的数据被下一用户从缓存复用。缓存 key 应包含会影响结果的租户、权限或语言，或为不同权限创建不同 loader。单个 key 的失败应放在对应位置，不应让无关结果全部失败；复合 key 需要稳定序列化或自定义 cache key。

### 关键考点

- batching 解决查询次数，request cache 解决重复读取
- 输出必须与输入 key 一一对齐
- 缓存边界必须与授权边界一致

### 常见追问或变体

- DataLoader 为什么不等于 Redis？
- mutation 后如何清除或更新当前请求内的 loader cache？

## 14. RAG 流式回答中的引用一致性

### 参考答案

协议应使用稳定的 `answer_id`、单调递增 `seq` 和稳定 `source_id`。可拆成 `source.upsert`、`text.delta`、`citation.add`、`citation.revoke`、`answer.completed`、`error` 事件；引用关联文本的结构化 span 或生成单元，而不是只靠 Markdown 中易漂移的 `[1]`。

前端允许引用先显示占位，来源元数据晚到后补全；断线按 last event ID 恢复并去重。完成事件携带最终引用映射和校验结果，客户端据此纠偏。不存在的引用不能渲染成可信链接；撤回来源时标记受影响内容而非静默替换。点击引用展示原文片段、文档信息和检索时间。

### 关键考点

- 稳定 ID、顺序号、幂等重放
- 增量展示与最终权威快照分层
- 模型输出不能直接成为可信引用关系

### 常见追问或变体

- 如何处理同一文本片段对应多个来源？
- 如何评估 citation correctness 和 citation completeness？

## 15. Agent 的中断、恢复与副作用幂等

### 参考答案

checkpoint 至少保存运行 ID、步骤状态、规范化输入、工具调用 ID、结果摘要、审批状态、上下文版本和下一步指针。工具调用采用 `planned → started → succeeded/failed` 状态，并把副作用记录持久化。恢复时不能仅因“started”就重试，应先按幂等键查询外部系统确认是否已成功。

创建订单使用由业务请求稳定派生的幂等键，服务端以唯一约束保证同键只产生一个订单；通知也使用独立键或 outbox。checkpoint 与副作用日志最好通过事务或 transactional outbox 衔接。恢复前重新验证用户授权、凭证有效性、库存、价格和审批，因为外部状态及权限可能已变化；不能盲目重放旧计划。

### 关键考点

- durable execution 不能只保存对话文本
- exactly-once 通常通过幂等效果而非网络传输保证
- 恢复是重新校验后的续跑，不是无条件重放

### 常见追问或变体

- 不支持幂等键的第三方工具如何补偿？
- 哪些工具调用恢复时必须再次人工确认？
