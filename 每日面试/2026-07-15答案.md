# 2026-07-15 全栈偏前端每日面试参考答案

## 1. JavaScript：`Atomics.waitAsync` 与共享内存异步协作

**参考答案：** `Atomics.wait` 会阻塞当前 agent，主线程通常不能使用；`waitAsync` 立即返回 `{ async, value }`，有等待时 `value` 是会完成为 `"ok"`、`"not-equal"` 或 `"timed-out"` 的 Promise。共享的 `Int32Array` 槽位存状态/版本号：先读版本，只有仍等于预期版本才 `waitAsync`，生产者先写结果再 `Atomics.store` 新版本并 `Atomics.notify`。版本检查避免“通知发生在开始等待前”而永久错过。超时后重新读取状态决定重试或报错；取消应由应用层 `AbortSignal` 结束等待结果的消费，且所有共享内存必须只在跨源隔离的可信环境使用。

**关键考点：** 原子读写、happens-before、版本号、主线程不可阻塞、超时。

**追问：** 为什么通知不能作为唯一状态，而应有可读取的状态值？

## 2. TypeScript：显式方差标注与回调 API 设计

**参考答案：** 若 `Dog extends Animal`，只返回 `T` 的 `Producer<Dog>` 可当作 `Producer<Animal>`（协变）；只接收 `T` 的 `Consumer<Animal>` 可当作 `Consumer<Dog>`（逆变）。类型参数同时读写通常应不变。`interface Producer<out T> { get(): T }`、`interface Consumer<in T> { put(value: T): void }` 可表达已知的泛型声明意图，但它们只适用于相同泛型类型之间的实例化比较，TypeScript 仍按结构兼容性检查；不匹配的真实成员用法会报错，不能靠标注把不安全 API 变安全。回调参数在 `strictFunctionTypes` 下优先用函数属性而非方法语法，以获得更严格的逆变检查。

**关键考点：** 协变、逆变、不变、结构类型、`strictFunctionTypes`。

## 3. ES6+：异步生成器的背压与提前关闭

**参考答案：** 生成器只在消费者请求下一项时执行到下一个 `yield`，因此自然避免无限预取；每次循环体完成才会触发下一次 `next()`。在 `finally` 中关联 `AbortController`，使 `break` 触发 iterator 的 `return()` 后取消尚未完成的请求。

```ts
async function* pages<T>(fetchPage: (cursor?: string, signal?: AbortSignal) => Promise<{ items: T[]; next?: string }>) {
  const controller = new AbortController();
  let cursor: string | undefined;
  try {
    do {
      const page = await fetchPage(cursor, controller.signal);
      for (const item of page.items) yield item;
      cursor = page.next;
    } while (cursor);
  } finally {
    controller.abort();
  }
}
```

请求失败会使迭代抛错给消费方；不要吞错后无限重试，重试预算应由调用方或明确策略控制。

**关键考点：** 拉取式背压、`return()`、`finally`、`AbortSignal`。

## 4. 浏览器：长任务切片与 INP 优化

**参考答案：** 先记录真实点击，用 Performance 的 Interaction/long task 标记、CPU profile 和 `PerformanceObserver('longtask')` 判断是计算、布局还是渲染占用主线程。`setTimeout(0)` 可粗粒度让出一轮任务；`scheduler.yield()` 语义更直接且可渐进增强；`requestIdleCallback` 适合不紧急后台工作，不能承诺及时完成；纯计算量大或可序列化时移至 Worker，主线程只负责输入和渲染。Promise continuation 属于微任务，持续排队会在浏览器处理渲染和下一次输入前全部执行，反而恶化 INP。切片后应复测 p75 INP，并避免每片触发大量 DOM 读写。

**关键考点：** Long Task、宏任务/微任务、yield、Worker、p75 INP。

## 5. 网络：HTTP 条件请求与并发编辑冲突

**参考答案：** `GET` 返回内容及代表版本的强 `ETag`；保存时发送 `If-Match: "etag"`。服务端仅当当前 ETag 匹配才写入并生成新 ETag，否则返回 `412` 和可安全获取的最新版本入口。前端保留基线、用户草稿和服务器最新版本，展示字段级 diff，让用户选择覆盖、合并或放弃，再以新 ETag 提交。客户端时间可被篡改、时钟不同且不能表示服务端所有写入，因此不是并发控制依据。

**关键考点：** 强 ETag、`If-Match`、412、三方合并、服务端版本权威。

**追问：** `If-None-Match` 更常用于哪类语义？

## 6. React：View Transition 与并发导航的状态边界

**参考答案：** 把会导致路由内容切换的更新放入 `startTransition`，并让可过渡的区域使用 `<ViewTransition>` 或浏览器 API；渲染准备好后再提交视觉过渡，输入状态仍应立即更新。不要把输入框值、鉴权结果、请求中的写操作、实时库存等可变事实冻结成动画快照；应保持状态源独立并处理导航中断。为 `prefers-reduced-motion: reduce` 关闭或缩短动画，保留可见焦点和语义化页面标题变化；低支持度浏览器直接使用普通导航。

**关键考点：** transition priority、可中断渲染、快照边界、无障碍降级。

## 7. Vue 3：`defineExpose` 的组件边界与命令式能力

**参考答案：** 子组件将稳定且必要的命令包成窄接口后暴露：`defineExpose({ openFileDialog, abort, reset })`，父组件通过模板 ref 调用。上传进度、内部文件列表、重试计时器和校验细节仍由子组件管理，父组件通过 props 和事件表达数据契约。直接让父组件改子组件 ref 会绕开校验、并发与清理逻辑，形成双向状态源，后续重构也会破坏调用者。对外暴露的命令应说明调用时机、幂等性和失败结果。

**关键考点：** 封装、模板 ref、单向数据流、最小暴露面。

## 8. 手写：支持订阅取消的 Observable

**参考答案：** 用关闭标记包装 observer；终态和取消都走同一个 `close()`，以保证 teardown 只运行一次。

```ts
type Observer<T> = Partial<{ next(value: T): void; error(err: unknown): void; complete(): void }>;
type Teardown = void | (() => void);

class Observable<T> {
  constructor(private readonly setup: (o: Required<Observer<T>>) => Teardown) {}
  subscribe(observer: Observer<T>) {
    let closed = false;
    let teardown: Teardown;
    const close = () => {
      if (closed) return;
      closed = true;
      if (typeof teardown === 'function') teardown();
    };
    const safe: Required<Observer<T>> = {
      next: value => { if (!closed) observer.next?.(value); },
      error: err => { if (!closed) { observer.error?.(err); close(); } },
      complete: () => { if (!closed) { observer.complete?.(); close(); } },
    };
    try { teardown = this.setup(safe); } catch (err) { safe.error(err); }
    return { unsubscribe: close };
  }
}
```

同步 `setup` 在返回 teardown 前就 `complete()` 是边界情形；工业实现需额外处理延迟取得 teardown。核心是终态幂等、禁止终态后 next、取消释放计时器/监听器。

**关键考点：** teardown、终态、取消幂等、资源释放。

## 9. 算法：最短编辑操作序列

**参考答案：** `dp[i][j]` 表示 `source` 前 `i` 个字符变为 `target` 前 `j` 个字符的最小操作数；首行/列为纯插入/删除。字符相同继承 `dp[i-1][j-1]`，否则取替换、删除、插入三者最小值加一。保存选择或从 `dp[m][n]` 倒推即可得到一组操作，最后反转。只求距离时保留上一行和当前行，空间从 `O(mn)` 降为 `O(n)`；要回溯完整路径一般需完整表或使用 Hirschberg 分治。

**关键考点：** DP 定义、边界初始化、回溯、空间压缩。

## 10. Next.js：`generateMetadata`、数据去重与 SEO 一致性

**参考答案：** 将 `getProduct(id)` 封装为同一渲染请求可去重的稳定数据读取（Next 对相同 `fetch` 有请求记忆；非 fetch 数据源可用 React `cache`），并让 metadata 与页面都调用它，避免各自读取不同版本。商品不存在应 `notFound()`，下架但需保留 SEO 可返回受控 410/替代页并避免虚构标题；数据暂时失败不应缓存错误为长期成功页面。canonical/OG 使用经校验的公开 URL，缓存策略随商品变更频率和发布失效链路决定，正文与 metadata 都携带或记录同一版本用于排障。

**关键考点：** request memoization、React cache、`notFound`、SEO、版本一致性。

## 11. Node.js：ESM Loader 的能力边界与生产风险

**参考答案：** loader hook 介入模块 specifier 的 `resolve` 与源码/格式的 `load`，可重写解析或转换模块。它是运行时关键路径，可能增加冷启动、破坏模块缓存假设、使本地与生产解析不同；远程拉取源码/配置还会引入可用性、完整性和供应链风险。路径别名和源码转换优先交给 TypeScript/bundler 构建期，配置在受控启动期获取并校验/缓存；确需 loader 时固定版本和来源、限制协议/路径、度量启动耗时，并用集成测试覆盖生产运行时。

**关键考点：** resolve/load hooks、冷启动、缓存、供应链、构建期优先。

## 12. NestJS：Outbox 模式与领域事件可靠投递

**参考答案：** 在同一数据库事务内写订单和 outbox 记录，outbox 至少有 `id`、`aggregate_id`、`event_type`、`payload`、`occurred_at`、`status`、`attempts`、`published_at`。独立 publisher 通过 `FOR UPDATE SKIP LOCKED` 批量领取或由 CDC 捕获新增行，投递成功后标记；投递可能重复，因此消息带 event ID，消费者以持久化去重表/业务幂等键处理。失败用退避、最大次数、死信和告警；回放只重发指定事件范围并保留审计，不能把“至少一次”误称为端到端恰好一次。

**关键考点：** 原子落库、at-least-once、`SKIP LOCKED`、消费者幂等、DLQ。

## 13. 数据库：读写分离下的“读己之写”一致性

**参考答案：** 可在写成功后的短窗口把该用户会话读路由到主库；或由写接口返回版本/LSN，后续读携带它，副本未追上则转主库或等待有限时间。也可为会话设置主库粘连，或在前端以已确认的写入结果乐观覆盖详情，但前端状态不能保证刷新、跨端和权限变化后的正确性。支付、库存扣减、权限授予、风控和关键余额必须从权威源读取；普通资料展示可接受可观测的短暂陈旧读。

**关键考点：** replica lag、read-your-writes、LSN、session stickiness、权威读。

## 14. AI/RAG：离线评测集与检索质量归因

**参考答案：** 从真实且脱敏的问题按主题、难度、时效、权限覆盖采样，人工标注可接受答案与支撑它的文档片段/版本；留出测试集，避免把答案泄漏进提示词。检索层测 Recall@k、MRR/nDCG，重排层测正确证据是否进入前位；生成层分别评估是否被证据支持（faithfulness）、是否回答问题，以及引用是否确实指向支撑片段。线上 trace 记录 query 改写、候选 chunk/分数、重排、上下文截断、模型输出和引用映射；无正确 chunk 多为切分/索引/召回，正确 chunk 被排后多为重排，证据在上下文仍乱答则查提示词/模型。

**关键考点：** gold evidence、Recall@k、nDCG、faithfulness、可追踪归因。

## 15. Agent/MCP：OAuth 资源服务器与授权范围

**参考答案：** MCP Server 应作为明确 resource server，发布自己的 resource identifier/audience，并验证 access token 的签发者、签名、过期时间、audience 与所需 scope；不要接受为其他服务签发的 token。scope 应按动作和资源细分，如 `tickets:read`、`tickets:create`，并在服务端把 token 身份映射为实际数据权限，不能只信模型声明。前端确认用于让人理解即将产生的副作用；服务端二次校验负责权限、参数、业务规则和幂等；审计日志记录主体、授权决策、scope、工具名、参数摘要、结果和关联 trace，敏感字段脱敏且不可由模型自行篡改。

**关键考点：** OAuth audience、scope、resource server、defense in depth、审计。

**追问：** 为什么仅靠 UI 上“模型将创建工单”的提示不足以构成授权？
