# 2026-07-01 全栈偏前端面试题参考答案

## 1. `MutationObserver`、Promise、`requestAnimationFrame` 的执行顺序

### 参考答案

点击回调属于一个 task。回调执行完后，浏览器清空微任务队列；Promise reaction 和 MutationObserver notification 都在微任务检查点处理，具体先后取决于它们何时入队，不能只背固定结论。微任务清空后，浏览器才有机会执行渲染步骤及下一帧的 `requestAnimationFrame`。

DOM 修改会安排 observer 通知；若 Promise 微任务继续修改 DOM，通知可能在同一检查点内再次被安排。微任务不断自我追加会让检查点迟迟不能结束，造成渲染饥饿、输入延迟和掉帧。

### 关键考点

- task、微任务检查点、渲染机会不是同一个队列。
- `requestAnimationFrame` 在下一次绘制前执行，不等于“立即执行”。
- 输出题应结合实际入队顺序分析，而不是假设所有浏览器回调都有固定优先级。

## 2. TypeScript `const` 类型参数解决了什么问题

### 参考答案

普通泛型通常把字面量推宽为可变结构；`const` 类型参数会在推断阶段优先保留字面量和只读结构：

```ts
function define<T>(value: T): T {
  return value;
}

function defineConst<const T>(value: T): T {
  return value;
}

const a = define({ routes: ['home', 'about'] });
// { routes: string[] }

const b = defineConst({ routes: ['home', 'about'] });
// { readonly routes: readonly ['home', 'about'] }
```

`as const` 由每个调用方显式书写，并作用于表达式；`const T` 由 API 作者声明推断策略，减少调用噪声。它适合路由、事件名、配置 DSL 等需要保留字面量的 API，但如果实现或调用方需要修改值，就不应强行使用只读推断。

### 关键考点

- 它改变推断偏好，不是运行时冻结。
- 约束必须允许只读值，例如优先写 `readonly string[]`。
- 已存入可变变量再传入时，早已发生的类型拓宽不会被逆转。

### 常见追问或变体

- `const` 声明、`as const`、`readonly` 和 `const T` 分别作用在哪一层？
- 如何从只读元组生成联合类型？

## 3. ES Iterator Helpers 与惰性求值

### 参考答案

`map`、`filter`、`take` 返回新的迭代器 helper，只描述处理流水线；调用 `next()`，或使用 `toArray()`、`for...of`、展开运算符、`reduce` 等终结操作时，数据才按需流过流水线。

```js
const result = Iterator.from(source)
  .filter(x => x.active)
  .map(x => x.id)
  .take(100)
  .toArray();
```

数组链式调用通常为每一步创建中间数组并遍历完整输入；迭代器可以少分配内存、在拿到前 100 个结果后停止，也能表达无限序列。迭代器通常是一次性的，部分消费后再次遍历会从当前位置继续或直接结束；流水线中的异常也会在消费时才抛出。

### 关键考点

- 惰性求值、短路、一次消费。
- Iterator 与 Iterable 的区别。
- 性能收益需要结合数据规模，普通小数组不必为了新 API 改写。

### 常见追问或变体

- 异步数据为什么需要 AsyncIterator，而不是 Iterator Helpers？
- 生成器的 `return()` 在提前结束时有什么作用？

## 4. 用 `AbortSignal` 统一取消多个异步操作

### 参考答案

外部取消、超时和内部控制器可以组合；任一真实失败时调用内部控制器取消兄弟请求。`Promise.all` 只会快速拒绝，并不会自动取消请求。

```ts
async function loadPageData(external: AbortSignal) {
  const internal = new AbortController();
  const timeout = AbortSignal.timeout(8_000);
  const signal = AbortSignal.any([
    external,
    timeout,
    internal.signal,
  ]);

  try {
    const [user, recommendations] = await Promise.all([
      fetch('/api/user', { signal }).then(checkResponse),
      fetch('/api/recommendations', { signal }).then(checkResponse),
    ]);
    return { user, recommendations };
  } catch (error) {
    if (!signal.aborted) internal.abort(error);
    if (signal.aborted) {
      throw signal.reason;
    }
    throw error;
  }
}

function checkResponse(response: Response) {
  if (!response.ok) throw new Error(`HTTP ${response.status}`);
  return response.json();
}
```

生产代码应在调用边界识别取消原因：路由离开通常不展示错误；超时可展示重试；HTTP 或解析错误应正常上报。还要在启动前调用 `signal.throwIfAborted()`，避免已取消任务继续执行。

### 关键考点

- 取消必须由底层 API 接收 signal 才有效。
- `AbortSignal.any()` 组合信号后，来源识别最好依赖明确 reason。
- 失败后的兄弟请求取消是资源治理，不改变原始失败原因。

### 常见追问或变体

- 如何兼容不支持 `AbortSignal.any()` 的环境？
- 如何让重试循环响应取消且不泄漏计时器？

## 5. 第三方 Cookie 限制与存储分区

### 参考答案

嵌入 iframe 的第三方 Cookie 可能默认不可用；localStorage、IndexedDB、BroadcastChannel 等状态可能按“顶层站点 + iframe 来源”分区。因此同一控制台嵌入 A、B 两个客户站点时，可能得到两个隔离存储区，不能再依赖它们共享登录态或广播消息。

兼容设计可以是：

1. 先在控制台自己的顶层页面完成登录，使用短期一次性 code 回到宿主。
2. iframe 与宿主通过校验 `origin` 的 `postMessage` 交换短期、受众受限的会话凭据。
3. 必须访问未分区 Cookie 时，在明确用户手势下请求 Storage Access API，并提供顶层跳转降级。
4. 服务端 Cookie 使用 `Secure`、合适的 `SameSite`，token 不放入 URL、localStorage 或宽泛消息广播。

### 关键考点

- 分区目标是防止跨站跟踪，不只是“Cookie 被禁用”。
- `postMessage` 必须校验精确 origin、消息 Schema 和关联 nonce。
- Storage Access API 依赖浏览器策略与用户授权，不应成为唯一链路。

### 常见追问或变体

- CHIPS/Partitioned Cookie 适合什么场景？
- iframe 被 CSP `frame-ancestors` 或 X-Frame-Options 阻止如何处理？

## 6. React `use` 与 Suspense 的数据读取边界

### 参考答案

`use(promise)` 在渲染阶段读取 Promise：pending 时挂起到最近的 Suspense fallback，fulfilled 时返回值，rejected 时交给错误边界。Effect 请求则发生在提交后，通常先渲染空状态，再用本地 state 驱动二次渲染，且容易形成客户端瀑布。

若组件每次渲染都创建新 Promise，React 每次看到的都是新的 pending 对象，会重复挂起。Promise 应来自带缓存的数据层、框架服务端读取，或由服务端组件创建后作为 prop 传入客户端组件。边界应按用户可感知区域划分；错误边界负责失败 UI，Suspense 只负责等待态。

从服务端传给客户端的 Promise 最终值必须可序列化，也不能把密钥、数据库对象等服务端能力泄露到客户端。

### 关键考点

- `use` 可以条件调用，但仍必须在组件或 Hook 内调用。
- Promise 身份稳定性与请求去重。
- Suspense、Error Boundary 各自处理 pending 与 rejected。

### 常见追问或变体

- 与 TanStack Query 的缓存和 hydration 如何组合？
- Suspense 边界过粗或过细分别有什么问题？

## 7. Vue 3.4+ `defineModel` 的双向绑定契约

### 参考答案

无参数 `defineModel()` 默认声明 `modelValue` prop，并在赋值时发出 `update:modelValue`。命名模型如 `defineModel('title')` 对应 `title` 与 `update:title`，因此可支持多个 `v-model`。

```vue
<script setup lang="ts">
const [title, modifiers] = defineModel<string, 'trim'>('title', {
  set(value) {
    return modifiers.trim ? value.trim() : value;
  },
});
</script>
```

若子组件给模型声明 `default`，而父组件绑定的 ref 初始是 `undefined`，子组件内部可能已有默认值，父组件仍是 `undefined`。组件库应优先要求父层提供唯一数据源，或在契约中明确初始化；不要依靠隐式默认值制造两份状态。

### 关键考点

- `defineModel` 是编译宏，本质仍是 prop + emit。
- 多模型、修饰符、类型参数和转换函数。
- 单向数据流没有消失，子组件不应绕开事件直接修改父状态。

### 常见追问或变体

- `defineModel` 与手写 `computed({ get, set })` 有何差别？
- 对象模型的深层修改为什么可能绕过预期的更新事件？

## 8. 手写支持通配符订阅的 EventEmitter

### 参考答案

用 `Map<string, Set<Handler>>` 保存订阅。`emit` 时复制快照，保证回调内 `off` 不改变当前轮迭代；Set 使同一函数对同一事件重复订阅时保持一份。

```ts
type Handler = (...args: unknown[]) => void;

class EventEmitter {
  private events = new Map<string, Set<Handler>>();

  on(event: string, handler: Handler) {
    let handlers = this.events.get(event);
    if (!handlers) {
      handlers = new Set();
      this.events.set(event, handlers);
    }
    handlers.add(handler);
    return () => this.off(event, handler);
  }

  once(event: string, handler: Handler) {
    const wrapped: Handler = (...args) => {
      this.off(event, wrapped);
      handler(...args);
    };
    return this.on(event, wrapped);
  }

  off(event: string, handler: Handler) {
    const handlers = this.events.get(event);
    handlers?.delete(handler);
    if (handlers?.size === 0) this.events.delete(event);
  }

  emit(event: string, ...args: unknown[]) {
    const direct = [...(this.events.get(event) ?? [])];
    const wildcard = [...(this.events.get('*') ?? [])];
    for (const handler of direct) {
      try { handler(...args); } catch (error) { console.error(error); }
    }
    for (const handler of wildcard) {
      try { handler(event, ...args); } catch (error) { console.error(error); }
    }
  }
}
```

异常策略应由契约决定：这里隔离异常以保证后续监听器执行；基础库也可收集后统一抛出或交给错误回调，不能默默吞掉。

### 关键考点

- 快照迭代、幂等取消、`once` 先取消再执行。
- 通配符回调需接收事件名。
- 返回取消函数有助于组件卸载时清理资源。

### 常见追问或变体

- 如果要求同一函数可重复订阅多次，数据结构如何调整？
- 异步监听器应串行、并行还是不等待？

## 9. 算法：最小覆盖子串

### 参考答案

右指针扩大窗口，当窗口已满足 `t` 的所有字符计数时移动左指针，持续收缩并记录最短答案。用 `formed` 表示已达到目标次数的字符种类数，而不是窗口字符总数。

```ts
function minWindow(s: string, t: string): string {
  if (!t || t.length > s.length) return '';

  const need = new Map<string, number>();
  for (const char of t) need.set(char, (need.get(char) ?? 0) + 1);

  const window = new Map<string, number>();
  let formed = 0;
  let left = 0;
  let bestStart = 0;
  let bestLength = Infinity;

  for (let right = 0; right < s.length; right++) {
    const char = s[right];
    window.set(char, (window.get(char) ?? 0) + 1);
    if (need.has(char) && window.get(char) === need.get(char)) formed++;

    while (formed === need.size) {
      const length = right - left + 1;
      if (length < bestLength) {
        bestStart = left;
        bestLength = length;
      }

      const removed = s[left++];
      if (need.has(removed) && window.get(removed) === need.get(removed)) {
        formed--;
      }
      window.set(removed, window.get(removed)! - 1);
    }
  }

  return bestLength === Infinity
    ? ''
    : s.slice(bestStart, bestStart + bestLength);
}
```

### 关键考点

- 重复字符必须按次数满足。
- 左右指针各至多移动 `n` 次，时间复杂度 `O(|s| + |t|)`。
- 空间复杂度为字符集规模，或写作 `O(|Σ|)`。

### 常见追问或变体

- 返回所有同长度最小窗口如何修改？
- Unicode grapheme cluster 不能简单按 UTF-16 code unit 处理。

## 10. Next.js Partial Prerendering 的静态壳与动态岛边界

### 参考答案

商品名称、描述、图片可构成预渲染静态壳；会员价读取 Cookie，实时库存读取动态数据，应各自放入 Suspense 边界。构建时先生成壳与 fallback，请求时再把动态区域流式填入。

动态区域应尽量小。读取 `cookies()`、`headers()` 等请求数据，或使用动态数据源，会使相应边界进入动态渲染。不能把带用户身份的会员价放进跨用户共享缓存；缓存 key 必须完整包含隔离维度，通常更安全的是不缓存个性化结果。

验证时同时检查构建输出、响应首字节、HTML 初始壳、动态 chunk 到达顺序，并以不同用户请求确认无数据串漏。Partial Prerendering/Cache Components 的具体开关和稳定性依 Next.js 版本确认，不能照搬旧版配置。

### 关键考点

- Suspense 边界决定静态壳和动态洞的粒度。
- 共享缓存与用户数据隔离。
- PPR 优化感知性能，不减少动态后端本身的成本。

### 常见追问或变体

- 动态库存失败时如何局部降级？
- CDN 缓存 HTML 时如何避免 Cookie 导致缓存污染？

## 11. Node.js Permission Model 的能力与局限

### 参考答案

启动时启用权限模型，并只允许插件读取必要目录；不授予子进程、worker 等能力。具体 flag 应以目标 Node 版本文档为准，例如文件系统可分别配置只读、写入路径白名单。

权限模型能减少误操作和部分供应链代码的能力，但它不是恶意代码的强安全边界：同一进程共享内存与已授权能力，运行时漏洞、原生扩展、资源耗尽等仍需容器、低权限系统用户、只读文件系统、网络策略和资源限额处理。

依赖可能在启动时读取配置、证书、locale 或缓存目录，权限不足会立即失败；运行时的懒加载、动态 import、临时文件或派生 worker 也可能在特定路径才失败。因此要用最小权限在真实启动与关键请求链路上测试。

### 关键考点

- 能力白名单与纵深防御。
- 文件权限不等于网络、CPU、内存隔离。
- 第三方插件最好使用独立进程或容器，并通过窄接口通信。

### 常见追问或变体

- 如何为插件设计超时、内存上限和输出大小限制？
- 为什么 `vm` 模块不能作为安全沙箱？

## 12. NestJS 输入验证与输出序列化的安全边界

### 参考答案

更新 DTO 只声明允许修改的字段；全局或路由级 ValidationPipe 可这样配置：

```ts
new ValidationPipe({
  whitelist: true,
  forbidNonWhitelisted: true,
  transform: true,
  transformOptions: { enableImplicitConversion: false },
});
```

`whitelist` 删除没有验证装饰器的属性，`forbidNonWhitelisted` 直接拒绝未知字段，便于发现越权尝试；`transform` 把普通对象转换为 DTO 实例，但隐式类型转换可能把意外字符串转成布尔值或数字，应优先显式 `@Type` 和校验规则。

实体不能直接作为响应契约。使用响应 DTO 或序列化类明确暴露字段，密码哈希、内部角色和密钥不进入响应对象。服务层仍要按当前用户权限挑选可更新字段，DTO 验证不能代替授权。

### 关键考点

- 验证、授权、持久化字段选择、输出序列化是四层边界。
- `PartialType` 只改变可选性，不自动决定谁有权更新。
- mass assignment 常来自把 body 直接 merge 到 ORM entity。

### 常见追问或变体

- 嵌套 DTO 为什么需要 `@ValidateNested` 和 `@Type`？
- 如何统一输出验证错误又不泄漏内部细节？

## 13. 乐观锁解决库存并发更新

### 参考答案

读取库存及版本后，更新时把旧版本放入条件，并原子校验库存：

```sql
UPDATE product
SET stock = stock - :count,
    version = version + 1
WHERE id = :id
  AND version = :expected_version
  AND stock >= :count;
```

受影响行数为 1 表示成功，为 0 表示版本冲突、库存不足或记录不存在，应用需再查询区分。可以对短暂冲突做少量、带抖动的重试，但下单接口还要用幂等键避免客户端重试造成重复扣减。

若业务只需扣库存，`stock >= count` 的单条原子更新往往比“先读版本再更新”更简单。高冲突热点下，乐观锁会产生大量失败重试，悲观锁或库存队列可能更稳定，但会增加锁等待和系统复杂度。

### 关键考点

- 判断依据是受影响行数，而非先查后的内存判断。
- 乐观锁解决写冲突，不等于请求幂等。
- 数据库约束或原子条件是防超卖的最终防线。

### 常见追问或变体

- 创建订单与扣库存跨服务时如何保证最终一致性？
- Redis 预扣库存后数据库失败如何补偿？

## 14. AI 前端的生成式 UI 与可验证状态

### 参考答案

至少分三层状态：

1. 对话层保存用户消息、模型文本与引用。
2. 协议层保存 tool call id、名称、参数增量、执行状态和结果。
3. 业务组件层只接收通过 Schema 验证的确定数据，不直接执行模型生成的代码。

前端使用组件注册表把允许的工具/组件名映射到本地实现；参数流式到达时仅展示 skeleton，完整 JSON 校验成功后再渲染。tool call id 和业务幂等键用于去重，事件日志或服务端快照支持刷新恢复。

“确认付款”必须显示服务端重新计算的金额与收款对象，由用户明确确认后调用独立授权接口；模型不能携带最终授权。所有高风险行为都应有权限校验、短期确认 token、审计日志和可撤销/补偿策略。

### 关键考点

- 模型输出是不可信输入，不是可执行 UI。
- 展示状态与副作用状态分离。
- Schema、组件白名单、幂等、恢复和人工确认。

### 常见追问或变体

- 流式参数中途修改已展示内容时如何处理？
- 多个并行 tool call 的顺序与局部失败如何呈现？

## 15. MCP Elicitation 的交互与安全设计

### 参考答案

普通工具参数应在调用开始前由模型准备；Elicitation 允许 Server 在执行过程中向客户端请求结构化、非敏感的补充信息。相比模型自由文本追问，它带有明确的 message 和受限 Schema，客户端可以用原生表单展示，并返回结构化结果。

客户端必须把三种动作当成不同结果：

- `accept`：用户提交了通过 Schema 校验的数据。
- `decline`：用户拒绝本次请求，Server 应提供替代路径或结束操作。
- `cancel`：用户取消当前交互，通常中止相关流程。

Schema 只能用于受支持的基础字段，客户端还应清楚显示请求来自哪个 Server、用途是什么，并允许用户检查后提交。密码、API Key、访问令牌等敏感凭据不应通过 Elicitation 收集，应改用 OAuth、系统密钥存储或受信任的带外认证流程。Server 给出的描述和默认值同样是不可信内容，不能诱导自动授权。

### 关键考点

- Elicitation 是客户端能力协商后的结构化交互，不保证客户端一定支持。
- 用户必须掌握 accept/decline/cancel 的最终决定。
- 最小化收集、来源标识、Schema 校验、敏感信息禁区。
