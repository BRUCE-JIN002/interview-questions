## 1. `Promise.try` 的同步调用与错误归一

### 参考答案

`Promise.try(fn)` 的核心价值是把同步返回、同步抛错和异步返回统一成 Promise 链。它会同步调用 `fn`；如果 `fn` 返回普通值，返回的 Promise fulfilled；如果同步抛错，返回的 Promise rejected；如果返回 Promise 或 thenable，则跟随其最终状态。

`Promise.resolve().then(fn)` 会把 `fn` 放到微任务中异步执行，因此执行时机不同。`new Promise(resolve => resolve(fn()))` 会同步调用 `fn`，并且构造器会把 executor 内抛出的异常转成 rejected Promise，近似 `Promise.try`，但可读性和泛型构造器语义不完全一样。

近似实现：

```js
function promiseTry(fn, ...args) {
  return new Promise((resolve, reject) => {
    try {
      resolve(fn(...args));
    } catch (error) {
      reject(error);
    }
  });
}
```

它不能完全模拟规范里的子类构造器、`this` 绑定、host 兼容性细节，也不能让旧环境自动获得原生兼容性表中的行为。

### 关键考点

- 同步调用和微任务调用的差异。
- 同步异常归一为 rejected Promise。
- thenable assimilate 与 `resolve` 语义。
- Polyfill 与近似实现的边界。

### 常见追问或变体

- 为什么 `Promise.resolve(fn())` 捕获不到 `fn` 的同步异常？
- `async () => fn()` 与 `Promise.try(fn)` 的执行时机是否一致？
- 在任务队列顺序题中，把 `Promise.try` 换成 `.then` 输出会怎样变化？

## 2. TypeScript 可辨识联合与穷尽性检查

### 参考答案

可以用公共字段 `status` 作为判别字段：

```ts
type CreatedOrder = { status: "created"; id: string; createdAt: string };
type PaidOrder = { status: "paid"; id: string; paidAt: string; amount: number };
type ShippedOrder = { status: "shipped"; id: string; shippedAt: string; trackingNo: string };
type CancelledOrder = { status: "cancelled"; id: string; cancelledAt: string; reason: string };

type Order = CreatedOrder | PaidOrder | ShippedOrder | CancelledOrder;

function assertNever(value: never): never {
  throw new Error(`Unexpected order: ${JSON.stringify(value)}`);
}

function renderOrderStatus(order: Order): string {
  switch (order.status) {
    case "created":
      return `待支付：${order.createdAt}`;
    case "paid":
      return `已支付：${order.amount}`;
    case "shipped":
      return `已发货：${order.trackingNo}`;
    case "cancelled":
      return `已取消：${order.reason}`;
    default:
      return assertNever(order);
  }
}
```

如果后来新增 `refunded` 状态但没有更新 `switch`，`default` 中的 `order` 不再是 `never`，编译期会报错。

大量可选字段会让非法组合变得可表示，例如 `status: "paid"` 却没有 `paidAt`，或同时有 `paidAt` 与 `cancelledAt`。可辨识联合把状态和字段绑定起来，更适合表达业务不变量。

### 关键考点

- 判别字段、联合类型收窄、`never` 穷尽性检查。
- “让非法状态不可表示”的建模思想。
- 可选字段适合局部缺省，不适合互斥状态机。

### 常见追问或变体

- 接口数据从后端来，运行时还需要校验吗？

## 3. `Intl.Segmenter` 与前端文本截断

### 参考答案

`str.length` 统计的是 UTF-16 code unit，不是用户看到的字符。Emoji、组合音标、部分生僻字、旗帜符号可能由多个 code point 或 grapheme cluster 组成，直接 `slice` 可能把一个字符截断成乱码或缺失变体选择符。

`Intl.Segmenter` 可以按 grapheme、word、sentence 分段：

```js
function truncateGrapheme(input, max, locale = "zh") {
  if (max <= 0) return "";

  if (typeof Intl !== "undefined" && Intl.Segmenter) {
    const segmenter = new Intl.Segmenter(locale, { granularity: "grapheme" });
    const parts = [];
    for (const { segment } of segmenter.segment(input)) {
      if (parts.length >= max) break;
      parts.push(segment);
    }
    return parts.join("") + (parts.length < [...segmenter.segment(input)].length ? "..." : "");
  }

  const chars = Array.from(input);
  return chars.length > max ? chars.slice(0, max).join("") + "..." : input;
}
```

生产实现要避免为了判断长度二次遍历大字符串，可以遍历到 `max + 1` 即停止。旧环境可用 `Array.from` 做 code point 级降级，或引入专门的 grapheme splitter，但要评估包体积。

### 关键考点

- UTF-16 code unit、code point、grapheme cluster 的区别。
- 国际化文本、Emoji、组合字符的截断风险。
- API 兼容性、性能和包体积取舍。

## 4. CSS Anchor Positioning 的弹层定位边界

### 参考答案

传统方案通常在点击或滚动时用 `getBoundingClientRect` 计算触发器位置，再把菜单写入 fixed/absolute 坐标。问题是要监听滚动、resize、布局变化、缩放、虚拟列表复用和遮挡翻转，代码复杂且容易抖动。

CSS Anchor Positioning 把触发元素声明为 anchor，弹层通过 CSS 引用 anchor 的位置，浏览器参与布局计算，更适合“贴着某个元素”的 UI。它能减少 JS 测量，但不能消除所有弹层问题：焦点管理、键盘导航、点击外部关闭、层级、可访问性仍要应用处理。

渐进增强可以先保留 JS 定位作为基础方案；支持 anchor 时启用 CSS 定位。组件接口不暴露底层实现，只暴露 `placement`、`offset`、`fallbackPlacements` 等语义。上线时要覆盖滚动容器、缩放、移动端键盘、RTL 和边界碰撞。

### 关键考点

- JS 测量定位和浏览器布局定位的职责差异。
- 弹层不只是坐标问题，还包括交互和可访问性。
- 渐进增强与兼容性检测。

### 常见追问或变体

- 为什么弹层常用 portal？
- 滚动容器和 transform 会如何影响 fixed/absolute 定位？
- 如何测试弹层不会被视口边缘遮挡？

## 5. React `cacheSignal` 与服务端数据读取取消

### 参考答案

`cacheSignal()` 用于 React 缓存生命周期内的取消感知。缓存的数据读取如果不再需要，或缓存条目被废弃，内部异步工作可以收到 `AbortSignal`，从而取消 fetch、停止解析、释放资源或减少无意义日志。

它不是客户端 `AbortController` 的替代品。客户端取消通常来自用户离开页面、切换 tab、输入变化；`cacheSignal` 更贴近服务端渲染和 React 缓存生命周期。超时仍应由业务设置，例如 `AbortSignal.timeout` 或服务端请求预算；缓存失效决定数据什么时候重新读取，不等价于取消已经开始的 I/O。

设计上应把 signal 传给支持取消的下游调用；下游不支持取消时，也要在返回后检查是否仍需要结果，避免写入过期缓存或输出误导日志。日志需要区分用户取消、渲染放弃、超时、下游失败，取消一般不应按错误率告警。

### 关键考点

- React 服务端缓存生命周期。
- AbortSignal 传播、超时和取消分类。
- 取消不等于回滚外部副作用。

### 常见追问或变体

- `cache` 和请求级缓存有什么区别？
- 被取消的请求已经到达下游，服务端还能保证不执行吗？
- 取消类错误是否应该上报到错误监控？

## 6. Vue 异步组件的加载、失败与重试

### 参考答案

`defineAsyncComponent` 可以把组件加载逻辑包装起来：

```ts
import { defineAsyncComponent } from "vue";

const UserPanel = defineAsyncComponent({
  loader: () => import("./UserPanel.vue"),
  loadingComponent: LoadingPanel,
  errorComponent: LoadErrorPanel,
  delay: 200,
  timeout: 10000,
  onError(error, retry, fail, attempts) {
    if (attempts <= 2) retry();
    else fail();
  },
});
```

路由级代码分割解决“页面 chunk 何时加载”；异步组件更适合页面内部按条件加载重组件。权限判断不应只靠组件是否加载，后端接口和路由守卫仍要校验。`Suspense` 可以统一协调异步依赖的 fallback，但错误上报、重试按钮、降级 UI 仍要明确设计。

加载失败要区分网络失败、版本发布导致 chunk 404、权限不足和组件内部运行时报错。chunk 404 常需要提示刷新或触发版本更新流程；权限不足不应自动重试；运行时报错应进入错误边界或全局错误处理。

## 7. Web Worker、Transferable 与大数据渲染

### 参考答案

解析 50MB 二进制日志应把 CPU 密集解析放到 Worker，主线程只负责调度和渲染。主线程读取文件后可以把 `ArrayBuffer` 作为 Transferable 转移给 Worker，避免结构化克隆复制大块内存：

```js
worker.postMessage({ buffer }, [buffer]);
```

转移后主线程的 `buffer` 会被 detached，不能继续读取。结构化克隆更方便但会复制数据，适合小对象或不想转移所有权的场景。Worker 解析后可以分批返回可渲染片段，主线程用 `requestAnimationFrame` 合并绘制，避免一次性 setState。

`SharedArrayBuffer` 适合高频共享内存和原子同步，但要求跨源隔离，安全和部署成本更高。`OffscreenCanvas` 适合把 Canvas 绘制也移到 Worker，但要看浏览器支持、图表库兼容性和交互事件回传成本。

### 关键考点

- Worker 解决主线程 CPU 阻塞。
- structured clone 与 Transferable 的内存语义。
- SharedArrayBuffer 的跨源隔离要求。
- 分批渲染和背压。

### 常见追问或变体

- Worker 里异常如何回传并关联 traceId？
- 大文件解析进度如何计算？
- 为什么把所有数据一次性 post 回主线程仍可能卡顿？

## 8. 手写支持 `leading` / `trailing` / `cancel` / `flush` 的 `throttle`

### 参考答案

一种实现：

```js
function throttle(fn, wait, options = {}) {
  const leading = options.leading !== false;
  const trailing = options.trailing !== false;
  let timer = null;
  let lastInvoke = 0;
  let lastArgs;
  let lastThis;
  let result;

  function invoke(time) {
    lastInvoke = time;
    const args = lastArgs;
    const ctx = lastThis;
    lastArgs = lastThis = undefined;
    result = fn.apply(ctx, args);
    return result;
  }

  function remaining(time) {
    const elapsed = time - lastInvoke;
    return wait - elapsed;
  }

  function startTimer(delay) {
    timer = setTimeout(() => {
      timer = null;
      if (trailing && lastArgs) invoke(Date.now());
    }, delay);
  }

  function throttled(...args) {
    const now = Date.now();
    if (!lastInvoke && !leading) lastInvoke = now;

    lastArgs = args;
    lastThis = this;

    const waitLeft = remaining(now);
    if (waitLeft <= 0 || waitLeft > wait) {
      if (timer) {
        clearTimeout(timer);
        timer = null;
      }
      return invoke(now);
    }

    if (!timer && trailing) startTimer(waitLeft);
    return result;
  }

  throttled.cancel = () => {
    if (timer) clearTimeout(timer);
    timer = null;
    lastInvoke = 0;
    lastArgs = lastThis = undefined;
  };

  throttled.flush = () => {
    if (!timer || !lastArgs) return result;
    clearTimeout(timer);
    timer = null;
    return invoke(Date.now());
  };

  return throttled;
}
```

滚动监听通常需要 `leading: true, trailing: true`，保证开始有反馈、结束补一次。resize 可偏 trailing，减少中间计算。输入联想更常用 debounce，因为用户停下后再请求更合理；如果必须边输入边给建议，可以 throttle 加取消旧请求。

### 关键考点

- 节流与防抖的触发语义。
- this、参数、返回值、取消和立即刷新。
- 定时器边界和最后一次调用保留。

### 常见追问或变体

- 用时间戳实现和用定时器实现有什么差异？
- React 中如何避免每次渲染创建新的 throttled 函数？
- 节流函数内部返回 Promise 时如何处理并发？

## 9. 手写 LFU Cache

### 参考答案

平均 O(1) 的 LFU 通常维护三类结构：

- `keyToNode`：key 到节点，节点含 key、value、freq。
- `freqToList`：freq 到双向链表，同频内按 LRU 排列。
- `minFreq`：当前最小频次，用于淘汰。

核心实现：

```js
class Node {
  constructor(key, value) {
    this.key = key;
    this.value = value;
    this.freq = 1;
    this.prev = null;
    this.next = null;
  }
}

class List {
  constructor() {
    this.head = new Node();
    this.tail = new Node();
    this.head.next = this.tail;
    this.tail.prev = this.head;
    this.size = 0;
  }
  addFirst(node) {
    node.next = this.head.next;
    node.prev = this.head;
    this.head.next.prev = node;
    this.head.next = node;
    this.size++;
  }
  remove(node) {
    node.prev.next = node.next;
    node.next.prev = node.prev;
    node.prev = node.next = null;
    this.size--;
  }
  removeLast() {
    if (!this.size) return null;
    const node = this.tail.prev;
    this.remove(node);
    return node;
  }
}

class LFUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.minFreq = 0;
    this.keyToNode = new Map();
    this.freqToList = new Map();
  }

  get(key) {
    const node = this.keyToNode.get(key);
    if (!node) return -1;
    this.touch(node);
    return node.value;
  }

  put(key, value) {
    if (this.capacity <= 0) return;

    if (this.keyToNode.has(key)) {
      const node = this.keyToNode.get(key);
      node.value = value;
      this.touch(node);
      return;
    }

    if (this.keyToNode.size >= this.capacity) {
      const list = this.freqToList.get(this.minFreq);
      const evicted = list.removeLast();
      this.keyToNode.delete(evicted.key);
    }

    const node = new Node(key, value);
    this.keyToNode.set(key, node);
    this.getList(1).addFirst(node);
    this.minFreq = 1;
  }

  touch(node) {
    const oldFreq = node.freq;
    const oldList = this.freqToList.get(oldFreq);
    oldList.remove(node);
    if (oldFreq === this.minFreq && oldList.size === 0) this.minFreq++;

    node.freq++;
    this.getList(node.freq).addFirst(node);
  }

  getList(freq) {
    if (!this.freqToList.has(freq)) this.freqToList.set(freq, new List());
    return this.freqToList.get(freq);
  }
}
```

同频淘汰最久未使用，所以同频链表新增或访问后放到头部，淘汰尾部。更新已有 key 应提升频次。`capacity = 0` 时所有 `put` 直接忽略。

### 关键考点

- 频次桶 + 双向链表 + 哈希表。
- `minFreq` 维护。
- 同频 LRU 规则。
- O(1) 的前提是不遍历查找。

### 常见追问或变体

- LFU 和 LRU 分别适合什么业务？
- 频次无限增长怎么办？
- 如何给 LFU 加 TTL？

## 10. Next.js `after` 的后台副作用边界

### 参考答案

`after()` 适合放“响应已经可以返回，但仍希望在当前请求生命周期后执行”的轻量副作用，例如非关键日志、埋点、缓存标记、低风险通知触发。它能改善用户感知延迟，但不应承载强可靠业务。

它不能替代队列，因为进程崩溃、平台终止、超时、部署重启、第三方失败都可能让任务丢失或无法重试。发券、扣库存、退款、支付通知这类关键副作用应写入数据库或队列，由 worker 可靠执行。

在 Route Handler 中要明确哪些数据已提交、哪些只是后置操作。Server Action 中要避免把用户可见状态依赖后置任务。静态页面中使用要确认框架版本和运行环境支持，不能假设它一定有完整请求上下文。后置任务日志要带 traceId，并单独统计失败率。

### 关键考点

- 响应后副作用和可靠任务的边界。
- 用户可见状态不能依赖不可靠后置任务。
- traceId、失败监控和超时预算。

### 常见追问或变体

- `after` 与 `waitUntil` 的关系是什么？

## 11. Node.js 防缓存击穿的 singleflight 设计

### 参考答案

核心是同一个缓存 key 在同一时刻只允许一个回源请求，其余请求等待同一个 Promise：

```js
const inflight = new Map();

async function singleflight(key, loader, timeoutMs) {
  if (inflight.has(key)) return inflight.get(key);

  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), timeoutMs);

  const p = loader(controller.signal)
    .finally(() => {
      clearTimeout(timer);
      inflight.delete(key);
    });

  inflight.set(key, p);
  return p;
}
```

生产方案还要加软过期：缓存过期后短时间内仍可返回旧值，同时后台刷新，避免所有用户等待慢回源。对等待者设置独立超时，不能让一个慢 Promise 无限挂住所有请求。失败时可返回 stale 数据，并按 key、下游、失败类型打指标。

为了避免慢请求阻塞所有用户，可以限制等待队列长度、给不同用户请求独立预算、超过预算返回降级数据、按商品维度限流。singleflight 只合并同 key 请求，不能替代下游限流、熔断和缓存预热。

### 关键考点

- 请求合并、软过期、stale fallback。
- Promise 清理、防内存泄漏。
- 超时、限流、熔断与指标。

### 常见追问或变体

- singleflight 的 Promise 失败后能否缓存错误？
- 多实例部署时本地 singleflight 还有用吗？
- 如何防止热点 key 在所有实例同时回源？

## 12. NestJS 后台任务的幂等消费者

### 参考答案

消费者要假设消息至少投递一次。每条“发放优惠券”消息应有业务幂等键，例如 `couponCampaignId + userId`。数据库建去重表或在发券表上建唯一约束，消费时在同一事务内写入幂等记录和发券结果。

流程：

1. 收到消息后解析并校验 schema。
2. 开启事务，尝试插入幂等键。
3. 如果唯一键冲突，说明已处理或处理中，按当前状态返回 ack 或延后重试。
4. 执行业务写入，提交事务。
5. 失败按错误类型重试，超过阈值进入死信队列。

重试要使用指数退避和最大次数，避免下游故障时放大流量。死信队列要有告警、人工补偿和重放工具。只依赖 exactly-once 不现实，因为队列、网络、消费者提交 offset、数据库事务属于不同系统，任何一步失败都可能造成重复或不确定状态。

### 关键考点

- 至少一次投递下的幂等设计。
- 唯一约束、去重表、事务边界。
- 重试退避、死信和补偿。

## 13. 面向前端的 API 错误码与可观测性设计

### 参考答案

错误响应应稳定、机器可读、可追踪：

```json
{
  "success": false,
  "code": "ORDER.OUT_OF_STOCK",
  "message": "当前商品库存不足",
  "traceId": "01J...",
  "details": {
    "field": "skuId",
    "retryable": false
  }
}
```

错误码可按领域分层：认证类 `AUTH.*`、权限类 `PERMISSION.*`、表单类 `VALIDATION.*`、库存类 `ORDER.*`、灰度类 `ROLLOUT.*`、系统类 `SYSTEM.*`。前端根据 code 决定交互：登录失效跳登录、字段错误挂到表单项、权限错误展示无权限、可重试错误给重试按钮。

用户文案最好由前端或统一文案服务按 code 映射，后端 message 可作为兜底或调试字段。数据库错误、内部服务名、SQL、堆栈、权限策略细节不应直接展示给用户。所有错误上报都带 traceId、用户动作、页面、接口、灰度版本，但要脱敏请求参数。

### 关键考点

- 错误码稳定性和前端可决策性。
- 用户文案与后端调试信息分离。
- traceId 贯穿前端、网关、服务和日志。
- 安全脱敏。

- HTTP status 和业务 code 如何配合？
- 国际化文案放前端还是后端？
- 如何治理历史错误码膨胀？

## 14. OpenAI Agents SDK 的 handoff、guardrail 与 tracing

### 参考答案

handoff 负责把会话从一个 Agent 转交给另一个 Agent，例如通用客服识别到退款意图后转给退款 Agent，再在复杂争议时转人工客服。tool 是 Agent 可调用的外部能力，例如查订单、创建退款申请。guardrail 是输入、输出或工具调用前后的策略检查，例如敏感信息过滤、退款金额上限、越权拦截。trace 记录一次运行中的模型决策、工具调用、耗时、错误和上下文，用于调试和审计。

前端应展示当前处理者、状态和可解释进度：正在识别意图、已转退款 Agent、正在查询订单、等待用户确认、已转人工。工具执行应有 pending/success/error 状态，失败时允许重试或转人工。人工接管后要冻结或限制模型继续调用写工具，避免双写。

权限和审计校验至少发生在 handoff 到高风险 Agent、调用写工具、读取敏感订单、提交退款、人工接管和恢复自动处理时。模型不能自行决定越权，策略层必须基于用户身份、订单归属、金额、频率和业务规则做确定性校验。

### 关键考点

- handoff、tool、guardrail、trace 的职责边界。
- Agent 状态对前端可见。
- 高风险节点的权限、确认和审计。

### 常见追问或变体

- handoff 时上下文应传多少？
- guardrail 是模型判断还是规则判断？
- trace 中哪些字段必须脱敏？

## 15. Agent 长期记忆污染与策略门禁

### 参考答案

长期记忆不能把用户输入或工具结果直接提升为系统规则。写入前应区分记忆类型：用户偏好、事实摘要、任务进度、策略规则、工具观察。普通用户最多写入自己的偏好和任务事实；策略规则只能来自受信配置、管理员审批或版本化策略仓库。

每条记忆应带来源、作者、时间、作用域、置信度、过期时间和可撤销记录。读取时按作用域和优先级合并：系统策略高于组织策略，组织策略高于用户偏好，工具观察不能覆盖安全规则。对涉及权限、退款、付款、发布、数据导出等高风险动作，必须经过策略门禁重新判断，不能因为记忆里有一句“免审核”就放行。

更新和删除也要审计。可疑记忆需要隔离、降权或等待人工审批。回放审计要能回答：哪条记忆参与了本次决策、它从何而来、何时写入、是否被人工批准、最终触发了哪些工具调用。

### 关键考点

- 记忆类型分层和来源标注。
- 用户偏好不能覆盖安全策略。
- 策略门禁在工具调用前做确定性校验。
- 审计、撤销、过期和人工审批。
