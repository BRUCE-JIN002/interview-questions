# 2026-06-24 每日 15 道全栈偏前端面试题答案

## 1. JavaScript 闭包与内存释放

参考答案：

闭包会保存它可达的词法环境。只要闭包本身仍被 GC Root 间接引用，环境中可达的大对象就不能回收。闭包不是泄漏的充分条件；对象在功能需要期间被保留是正常的，页面已销毁且引用链再无业务意义才属于泄漏。

定位时可在页面进入、退出后分别做 Heap Snapshot，执行强制 GC，比较对象数量和 retained size，再沿 Retainers 查到 GC Root。常见来源：

- 事件监听器：全局对象保存回调，回调闭包引用页面状态；卸载时移除监听。
- 定时器：未清理的 interval/timeout 保存回调；卸载时清理。
- 缓存：无上限的 Map 或模块级数组持续保存对象；增加容量、TTL 或弱引用策略。
- 异步回调：请求结束前回调保留状态；使用取消信号并避免在卸载后继续更新。

关键考点：

- GC 根据可达性工作，不是离开函数就立即释放。
- Heap Snapshot 应关注 retained size 和引用链，而不只是 shallow size。

常见追问或变体：

- `WeakMap` 为什么不阻止 key 被回收？
- Detached DOM tree 常由哪些引用造成？

## 2. TypeScript：`satisfies`、类型注解与类型断言

参考答案：

- `routes1: RouteConfig` 会校验对象，但变量的可见类型被拓宽为 `RouteConfig`，具体键和部分字面量信息可能丢失。
- `as RouteConfig` 的核心是告诉编译器“按此类型看待”，可能绕过本应暴露的结构问题，不适合承担校验职责。
- `satisfies RouteConfig` 会检查表达式满足目标类型，同时尽量保留表达式自身的精确推断结果，例如保留 `home` 键和 `'/home'` 字面量。

业务配置通常优先使用 `satisfies`；如果希望变量对外只暴露抽象接口，则使用类型注解；只有开发者掌握编译器无法证明的事实时才谨慎使用断言。

关键考点：

- `satisfies` 只做兼容性检查，不把表达式结果强制改成目标类型。
- 类型断言不会生成运行时校验。

常见追问或变体：

- `as const satisfies RouteConfig` 有什么效果？
- 运行时配置为什么还需要 Zod、Valibot 等 schema 校验？

## 3. 手写带并发上限的异步任务池

参考答案：

```ts
async function asyncPool<T, R>(
  limit: number,
  items: T[],
  worker: (item: T, index: number) => Promise<R>
): Promise<PromiseSettledResult<R>[]> {
  if (!Number.isInteger(limit) || limit <= 0) {
    throw new RangeError('limit must be a positive integer');
  }

  const results = new Array<PromiseSettledResult<R>>(items.length);
  let nextIndex = 0;

  async function runWorker() {
    while (nextIndex < items.length) {
      const index = nextIndex++;
      try {
        const value = await worker(items[index], index);
        results[index] = { status: 'fulfilled', value };
      } catch (reason) {
        results[index] = { status: 'rejected', reason };
      }
    }
  }

  const workerCount = Math.min(limit, items.length);
  await Promise.all(Array.from({ length: workerCount }, runWorker));
  return results;
}
```

增加取消能力时，把 `signal` 传入任务池和 `worker`：领取新任务前检查 `signal.aborted`，并要求底层 `fetch` 等操作也接收同一个 signal。取消只能阻止或通知任务，无法强行终止不支持取消的 Promise。

关键考点：

- 固定数量的 worker 竞争领取下标，天然限制并发。
- 结果写入原下标，保证输出顺序。

常见追问或变体：

- 如何实现失败即停止领取新任务？
- 如何动态调整并发数或增加任务优先级？

## 4. XSS、CSP 与 Trusted Types

参考答案：

治理应从危险数据流入手，而不是只匹配标签：

1. 盘点 `innerHTML`、`insertAdjacentHTML`、`document.write` 等注入点。
2. 普通文本始终使用 `textContent` 或框架默认转义。
3. 必须展示富文本时使用成熟清洗库，按允许标签和属性白名单清洗。
4. 部署 CSP，优先 nonce/hash，逐步移除 `unsafe-inline`，限制脚本来源。
5. 先用 `Content-Security-Policy-Report-Only` 收集违规，再启用 `require-trusted-types-for 'script'`。
6. 建立集中 Trusted Types policy，只有审计过的清洗函数能产生可信 HTML。
7. 第三方脚本最小化、固定来源、必要时使用 SRI 或 iframe 隔离。

只过滤 `<script>` 不够，因为事件属性、`javascript:` URL、SVG、错误嵌套、编码绕过和 DOM clobbering 等都可能执行代码。

关键考点：

- 输出编码适合纯文本；富文本需要上下文相关清洗。
- CSP 和 Trusted Types 是纵深防御，不替代服务端与前端的数据处理。

常见追问或变体：

- React 的 `dangerouslySetInnerHTML` 是否自动安全？
- CSP nonce 应如何生成和下发？

## 5. React 19.2：副作用中的陈旧闭包

参考答案：

把 `theme`、`muted` 都加入依赖会避免陈旧值，但也会导致设置变化时断开并重连。`ref` 能读取最新值，但更新与读取依赖手工同步，可读性和 lint 支持较弱。`useCallback` 只缓存函数身份，依赖变化时仍会生成新函数，不能自动解决“读取最新值但不重启 Effect”。`useEffectEvent` 适合把“不应触发重连、但需读取最新提交状态”的逻辑从 Effect 中分离。

```tsx
function ChatRoom({ roomId, theme, muted }) {
  const onConnected = useEffectEvent(() => {
    if (!muted) showNotification('Connected', theme);
  });

  useEffect(() => {
    const connection = createConnection(roomId);
    connection.on('connected', onConnected);
    connection.connect();
    return () => connection.disconnect();
  }, [roomId]);
}
```

关键考点：

- 响应式依赖决定“何时重新同步”；Effect Event 负责读取最新值。
- 不能用 `useEffectEvent` 隐藏本应触发 Effect 重跑的依赖。

常见追问或变体：

- Effect Event 能否传给子组件或在点击事件中调用？
- React Strict Mode 为什么开发环境会额外执行一次 Effect 清理与初始化？

## 6. Vue 3：组合式函数的副作用作用域

参考答案：

`effectScope` 可以收集其 `run()` 内创建的计算属性和监听器，调用 `scope.stop()` 时统一停止。`getCurrentScope()` 用于判断当前是否处于某个作用域；`onScopeDispose()` 可注册与当前作用域绑定的清理逻辑。

```ts
import { effectScope, ref, watch, onScopeDispose } from 'vue';

export function createSearchSession(source) {
  const scope = effectScope();
  const result = ref([]);

  scope.run(() => {
    const stop = watch(source, async value => {
      result.value = await search(value);
    }, { immediate: true });

    onScopeDispose(() => stop());
  });

  return {
    result,
    stop: () => scope.stop()
  };
}
```

组件 `setup` 内创建的作用域通常会随组件卸载停止；脱离组件创建的 detached scope 必须由调用方显式停止。

关键考点：

- 统一管理动态创建的一组副作用，避免逐个保存 stop handle。
- `onScopeDispose` 可用于组合式函数，不必强依赖组件生命周期。

常见追问或变体：

- `effectScope(true)` 的 detached 参数有什么影响？
- 异步创建的 watcher 为什么可能无法自动绑定组件作用域？

## 7. 算法：合并区间并分析边界

参考答案：

```ts
function mergeIntervals(intervals: Array<[number, number]>) {
  if (intervals.length === 0) return [];

  const sorted = intervals
    .map(([start, end]) => start <= end ? [start, end] as [number, number] : [end, start])
    .sort((a, b) => a[0] - b[0] || a[1] - b[1]);

  const result: Array<[number, number]> = [sorted[0]];

  for (let i = 1; i < sorted.length; i++) {
    const current = sorted[i];
    const last = result[result.length - 1];

    if (current[0] <= last[1]) {
      last[1] = Math.max(last[1], current[1]);
    } else {
      result.push(current);
    }
  }

  return result;
}
```

排序时间复杂度为 `O(n log n)`，扫描为 `O(n)`；若不计排序副本，结果外额外空间可视为 `O(n)`。闭区间端点相等时应合并，因此条件是 `<=`。示例中的 `[8,10]` 和 `[10,12]` 会合并。

关键考点：

- 先排序才能一次扫描完成合并。
- 要先明确闭区间、开区间以及“相邻”是否算重叠。

常见追问或变体：

- 如果输入已经按起点排序，复杂度如何变化？
- 如何插入一个新区间并保持合并结果？

## 8. 微前端运行时隔离设计

参考答案：

- JS 隔离：禁止隐式全局变量；对历史应用使用 Proxy 沙箱或 iframe；记录并恢复子应用修改的全局副作用。
- CSS 隔离：CSS Modules、命名空间、Shadow DOM 或构建时前缀；弹层挂载点也要限定范围。
- 路由：主应用统一分配路由前缀，子应用通过协议同步，不各自抢占全局 history。
- 依赖：React/Vue 等按兼容性决定共享或隔离，避免为了减少体积强行共享不兼容单例。
- 通信：使用版本化事件协议或共享服务，避免直接读写全局 store。
- 卸载：清理监听器、定时器、订阅、DOM、路由钩子和模块缓存，并用重复挂载/卸载测试检查内存。

iframe 天然隔离 JS、CSS 和 DOM，也能提供独立安全边界；代价是路由、弹窗、尺寸、自适应、登录态、通信、无障碍和性能整合更复杂。

关键考点：

- 隔离不仅是样式，还包括运行时副作用和生命周期。
- 公共依赖共享是兼容性决策，不只是包体积优化。

常见追问或变体：

- Shadow DOM 为什么不能解决全局 JS 污染？
- Module Federation 的 shared singleton 有什么风险？

## 9. 前端线上故障复盘与可观测性

参考答案：

先止损：暂停发布、评估影响面、关闭灰度或回滚到已验证版本。定位时按 release/version 聚合白屏率，收集 URL、浏览器、网络、地区、用户操作轨迹和资源加载状态。

`Script error.` 常见于跨域脚本缺少可用的 CORS 信息。脚本响应需配置允许源，`script` 标签按场景设置 `crossorigin`。构建产物应带 release id；Source Map 上传到监控平台后从公开 CDN 删除或限制访问，由平台在服务端还原堆栈。

验证修复时先小流量灰度，观察白屏率、JS 错误率和核心流程成功率，再逐步放量。复盘应记录时间线、根因、为何监控未提前发现、修复项负责人和完成期限。

关键考点：

- Source Map、错误事件和静态资源必须关联同一发布版本。
- 回滚能力和灰度指标属于故障治理的一部分。

常见追问或变体：

- 如何检测“没有抛异常但根节点为空”的白屏？
- 如何避免监控 SDK 自身导致页面故障？

## 10. Next.js Server Actions 的安全与数据更新

参考答案：

Server Action 应被当作公开的服务端入口：

1. 使用 schema 校验 `FormData`，不要信任类型声明。
2. 从服务端 session/cookie 获取当前用户，不接收客户端声明的 userId 作为身份。
3. 查询地址记录并校验其归属当前用户。
4. 校验 Origin/Host，结合 SameSite Cookie 和框架机制防御跨站请求。
5. 提交按钮使用 pending 状态防重复点击；服务端仍需幂等或版本校验。
6. 返回结构化字段错误和通用系统错误，避免泄漏内部信息。
7. 更新成功后按页面策略调用 `revalidatePath`/`revalidateTag`，或重定向。

客户端可以伪造 Action 参数和直接构造请求，因此“代码在服务端”不等于“调用者可信”。

关键考点：

- 鉴权是“你是谁”，资源归属校验是“你能否修改这条数据”。
- 乐观 UI 最终必须与服务端权威状态收敛。

常见追问或变体：

- 两个标签页同时修改地址如何避免覆盖？
- Server Action 和 Route Handler 应如何选？

## 11. Node.js Stream 背压

参考答案：

背压是下游消费速度低于上游生产速度时的流量反馈。Writable 内部缓冲超过 `highWaterMark` 后，`write()` 返回 `false`，上游应暂停，等待 `drain` 后再继续。忽略返回值会让待写数据持续堆在内存中。

```ts
import { pipeline } from 'node:stream/promises';
import { createReadStream } from 'node:fs';

await pipeline(
  createReadStream('large.csv'),
  createCsvTransform(),
  createObjectStorageUploadStream()
);
```

`pipeline` 会连接背压传播，并在任一流失败时 reject 和销毁相关流。还应结合 `AbortSignal`、临时分片清理、上传中止和合理的 `highWaterMark`；不能把每个 chunk 无限制转成独立异步上传任务。

关键考点：

- `highWaterMark` 是缓冲阈值，不是严格内存上限。
- Stream 的价值是增量处理和背压传播。

常见追问或变体：

- objectMode 下 `highWaterMark` 的单位是什么？
- Transform 中异步处理如何避免无界并发？

## 12. NestJS 异步队列与任务可靠性

参考答案：

请求接口完成鉴权和参数校验后，先在数据库创建导出记录，再以记录 id 作为 job id 入队，立即返回任务 id。Worker 根据 id 读取权威参数，生成文件并上传对象存储，最后更新数据库状态和文件地址，再发送通知。

- 重试：只重试可恢复错误，指数退避并限制次数。
- 幂等：唯一 job id；Worker 开始前检查任务状态，最终写入使用条件更新。
- 失败：保存失败原因和最后一次错误，进入 failed/dead-letter 处理流程。
- 并发：按 CPU、数据库和对象存储能力设置 worker concurrency。
- 取消：数据库记录取消请求，Worker 在阶段边界检查；已开始的外部操作需要显式中止。
- 进度：队列可保存临时进度，数据库保存用户可查询的稳定状态。

文件正文放对象存储；业务状态、归属和文件元数据放数据库；队列保存执行调度和短期运行信息。

关键考点：

- 队列通常是至少一次投递，消费者必须幂等。
- 多实例依赖共享队列和数据库，不能把状态只放进程内存。

常见追问或变体：

- Worker 生成文件成功但更新数据库前崩溃怎么办？
- 如何避免大量导出任务拖垮业务数据库？

## 13. 接口幂等与重复提交治理

参考答案：

前端在提交后禁用按钮并显示明确状态，但这只能改善交互，无法抵御网络重试和并发请求。客户端为一次业务操作生成幂等键，通过 `Idempotency-Key` 发送；服务端把“用户/租户 + 接口 + 幂等键”作为唯一键，记录处理中、成功或失败状态及响应摘要。

订单表还应有业务唯一约束，这是最终一致性的底线。服务端只有在第一次成功插入幂等记录后才执行业务；重复请求根据状态返回已有结果、提示处理中或允许对可重试失败重新执行。

- 防抖：减少短时间点击，不能作为安全保证。
- Redis 锁：适合短期互斥，但有过期、续租和故障一致性问题。
- 唯一索引：强约束最终不重复，但只解决数据库中的唯一性。
- 幂等记录表：能返回相同结果、追踪状态，但需处理记录过期和执行中崩溃。

关键考点：

- 幂等键代表一次业务意图，重试必须复用同一个键。
- 超时不代表服务端未成功，客户端不能盲目生成新键重试。

常见追问或变体：

- 支付回调如何设计幂等？
- 创建接口与更新接口的幂等策略有何不同？

## 14. RAG 检索质量与引用可信度

参考答案：

链路可分为查询理解、召回、重排、上下文组装和生成：

- 文档切分保留标题、段落、版本、权限和稳定引用位置。
- 混合使用关键词与向量召回，对候选结果重排和去重。
- 上下文按相关性和文档结构组装，保留 chunk id 与原文位置。
- 要求答案中的每个关键结论关联证据；引用必须来自实际送入模型的片段。
- 证据不足、文档冲突或权限不足时明确拒答或说明不确定。

离线集应包含问题、标准答案、相关文档和不可回答样本。指标可分为 Recall@K/NDCG 等检索指标、答案正确性、引用准确性、引用完整性和拒答准确率。线上关注采纳率、追问率、引用点击率、人工纠错率、延迟和成本。

检索失败是相关证据未进入上下文；利用失败是证据存在但答案没正确使用；幻觉是结论缺乏证据或与证据冲突。

关键考点：

- 不能只用“答案看起来合理”评估 RAG。
- 权限过滤必须发生在检索链路，不能只靠提示词。

常见追问或变体：

- 如何构造 hard negative？
- 文档更新后如何做增量索引和版本失效？

## 15. Agent 上下文管理与链路可观测性

参考答案：

上下文应分层：

- 固定层：系统规则、权限、安全边界和任务目标，压缩时不能丢失。
- 工作层：当前计划、最近交互、待办和关键工具结果。
- 外部记忆层：原始文件、长日志、历史事实存储在可检索系统中，按需取回。
- 摘要层：在 token 阈值前生成结构化摘要，保留决策、约束、未完成事项和来源指针。

长任务在关键阶段保存检查点，包括输入摘要、状态、产物引用和幂等标识。一次用户任务对应 trace；模型调用、检索、工具执行、handoff 对应 span，记录耗时、状态、token、成本、重试和错误类型。日志默认脱敏，不保存密钥、完整个人信息或无必要的模型输入输出。

前端展示计划、当前步骤、工具名称、参数摘要、结果摘要、失败与重试，但不展示隐藏推理过程；敏感或有副作用的动作进入人工确认。

关键考点：

- 压缩目标不是简单截断，而是保持约束和任务状态。
- 可观测性要能回答“慢在哪里、贵在哪里、错在哪里、能否复现”。

常见追问或变体：

- 摘要本身出错如何检测和恢复？
- 多 Agent handoff 如何避免上下文污染和权限扩大？
