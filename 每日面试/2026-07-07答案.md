## 1. Promise `finally` 的值穿透与异常覆盖

### 参考答案

`finally(onFinally)` 不接收上一个 fulfilled value 或 rejected reason，它的默认语义是“清理后继续传递原结果”。如果 `onFinally` 正常返回普通值，原来的 value/reason 会继续向后传；如果 `onFinally` 抛错或返回 rejected Promise，新的错误会覆盖原结果。

```js
Promise.resolve(1)
  .finally(() => 2)
  .then(v => {
    console.log(v); // 1
    throw new Error('A');
  })
  .finally(() => Promise.reject('B'))
  .catch(e => console.log(e)); // B
```

`then(onFinally, onFinally)` 会把回调返回值作为下一步结果，不能自动值穿透。`finally` 更适合关闭 loading、释放锁、上报耗时，不适合修改业务返回值。

### 关键考点

- `finally` 不改变原 settled 状态，除非自身失败。
- `throw` 与 `Promise.reject` 都会让后续进入 rejected 分支。
- `finally` 回调也会排入 Promise 微任务链。

### 常见追问或变体

- `finally(() => new Promise(r => setTimeout(r, 1000)))` 会不会等待？
- `async/await` 中 `try/finally` 与 Promise `finally` 的异同是什么？

## 2. TypeScript `Awaited<T>` 与接口客户端返回类型推导

### 参考答案

`Awaited<T>` 会模拟 `await` 的类型行为：递归解包 Promise-like 类型，直到得到最终值；对于非 Promise 类型保持原样；对于 `null | undefined` 保留原类型。

```ts
type ApiResponse<T> = { code: number; data: T; message: string };
type UnwrapData<T> = Awaited<T> extends ApiResponse<infer D> ? D : Awaited<T>;

async function request<T>(input: RequestInfo): Promise<ApiResponse<T>> {
  const res = await fetch(input);
  return res.json();
}

type User = { id: string; name: string };
type UserData = UnwrapData<ReturnType<typeof request<User>>>; // User
```

在 SDK 中可以把业务函数统一声明为 `Promise<ApiResponse<T>>`，页面层用 `UnwrapData<ReturnType<typeof fn>>` 拿到数据类型，避免重复写接口返回结构。

### 关键考点

- `Awaited<Promise<Promise<T>>>` 得到 `T`。
- `infer` 可从统一响应结构中提取 `data`。
- 类型工具只约束编译期，运行时仍要校验接口结构。

### 常见追问或变体

- 如果接口可能返回分页结构 `Page<T>`，类型如何继续提取 item？
- `ReturnType` 遇到泛型函数时为什么可能得到较宽类型？

## 3. ES Decorators 与 TypeScript 装饰器迁移

### 参考答案

旧版 TypeScript 装饰器偏向实验语义，常与 `emitDecoratorMetadata`、`reflect-metadata` 搭配，用于 NestJS DI、ORM、参数装饰器等。标准 decorators 的签名不同，回调接收 value 和 context，更强调类、字段、方法的标准初始化流程，不直接等价于旧元数据方案。

迁移要检查：

- 装饰器函数签名：旧版 `(target, key, descriptor)` 不能直接套用。
- 元数据：标准 decorators 不自动提供设计类型元数据，依赖 `Reflect.getMetadata` 的逻辑要单独处理。
- 字段初始化顺序：类字段、accessor、method 的装饰时机不同。
- 框架兼容：NestJS 现有生态仍大量依赖旧装饰器语义，不能盲目切换 tsconfig。
- 构建链：Babel、SWC、tsc 对 decorators 配置必须一致。

### 关键考点

- 标准 decorators 与 legacy decorators 是两套语义。
- DI、ORM、路由声明最容易受元数据变化影响。
- 迁移应从非核心埋点类装饰器开始验证。

### 常见追问或变体

- NestJS 为什么依赖 `emitDecoratorMetadata`？
- 方法装饰器如何保留 `this` 和原始 descriptor 行为？

## 4. 手写支持循环引用的 `deepEqual`

### 参考答案

核心思路是递归比较结构，并用 `WeakMap<object, WeakMap<object, boolean>>` 记录已经比较过的对象对，避免循环引用无限递归。

```js
function deepEqual(a, b, seen = new WeakMap()) {
  if (Object.is(a, b)) return true;
  if (typeof a !== 'object' || a === null || typeof b !== 'object' || b === null) {
    return false;
  }

  let inner = seen.get(a);
  if (inner?.get(b)) return true;
  if (!inner) {
    inner = new WeakMap();
    seen.set(a, inner);
  }
  inner.set(b, true);

  if (a.constructor !== b.constructor) return false;

  if (a instanceof Date) return a.getTime() === b.getTime();
  if (a instanceof RegExp) return a.source === b.source && a.flags === b.flags;

  if (a instanceof Map) {
    if (a.size !== b.size) return false;
    for (const [key, value] of a) {
      if (!b.has(key) || !deepEqual(value, b.get(key), seen)) return false;
    }
    return true;
  }

  if (a instanceof Set) {
    if (a.size !== b.size) return false;
    for (const value of a) {
      let found = false;
      for (const other of b) {
        if (deepEqual(value, other, seen)) {
          found = true;
          break;
        }
      }
      if (!found) return false;
    }
    return true;
  }

  const keysA = Reflect.ownKeys(a);
  const keysB = Reflect.ownKeys(b);
  if (keysA.length !== keysB.length) return false;

  for (const key of keysA) {
    if (!Reflect.has(b, key) || !deepEqual(a[key], b[key], seen)) return false;
  }
  return true;
}
```

不能直接用 `JSON.stringify`，因为它会丢失 `undefined`、函数、Symbol key，不能表达 `Map`/`Set`，遇到循环引用会报错，且对象 key 顺序也可能干扰结果。

### 关键考点

- `Object.is` 能正确处理 `NaN` 和 `-0`。
- 循环引用要记录“对象对”，只记录单个对象不够。
- `Set` 深比较如果元素是对象，朴素双循环最坏会到 `O(n^2)`。

### 常见追问或变体

- 如何支持自定义类实例？
- `Map` 的 key 如果是深层对象，是否应该按引用还是按深比较？

## 5. BroadcastChannel 与 Web Locks 解决多标签页协同

### 参考答案

可以把“刷新 token 和消息轮询”设计成 leader tab 负责，其他 tab 通过 `BroadcastChannel` 接收结果。leader 选举使用 Web Locks：每个 tab 尝试申请同名锁，拿到锁的 tab 成为 leader；锁释放或页面关闭后，其他 tab 重新竞争。

关键流程：

1. 所有 tab 创建 `BroadcastChannel('app-session')`。
2. 尝试 `navigator.locks.request('session-leader', { ifAvailable: true }, ...)`。
3. leader 定时刷新 token、轮询消息，并广播 `{ type: 'token' }`、`{ type: 'messages' }`。
4. follower 监听广播更新本地状态。
5. leader 发心跳，follower 超时未收到心跳则重新竞争锁。

降级方案可以用 `localStorage` 心跳加时间戳，但要处理时钟漂移和原子性较弱的问题。`BroadcastChannel` 不适合传敏感长期密钥，只传状态变化或短期结果。

### 关键考点

- BroadcastChannel 负责跨标签通信，不负责互斥。
- Web Locks 负责同源上下文的互斥协调。
- 页面异常关闭时锁会释放，但仍需要心跳兜底。

### 常见追问或变体

- Safari 不支持 Web Locks 时怎么降级？
- token 刷新失败如何避免所有 tab 同时跳登录页？

## 6. React `useActionState` / `useFormStatus` 的表单提交流程

### 参考答案

`useActionState` 把表单 action 的结果映射成组件状态，适合服务端校验、错误回显和提交后状态更新。`useFormStatus` 从表单上下文读取 pending 状态，适合提交按钮禁用、loading 展示，不需要手动把 pending 层层传递。

```tsx
async function login(prevState: LoginState, formData: FormData) {
  const email = String(formData.get('email') || '');
  const password = String(formData.get('password') || '');
  if (!email || !password) return { error: '请输入邮箱和密码' };
  return authenticate(email, password);
}

function SubmitButton() {
  const { pending } = useFormStatus();
  return <button disabled={pending}>{pending ? '登录中' : '登录'}</button>;
}

function LoginForm() {
  const [state, formAction] = useActionState(login, { error: '' });
  return (
    <form action={formAction}>
      <input name="email" />
      <input name="password" type="password" />
      {state.error && <p>{state.error}</p>}
      <SubmitButton />
    </form>
  );
}
```

相比 `useState + onSubmit`，它更贴近表单原生提交和渐进增强；缺点是心智更依赖 action 语义，复杂客户端校验仍可能需要本地状态。

### 关键考点

- `useActionState` 管 action 结果状态。
- `useFormStatus` 管当前 form 的提交状态。
- 服务端 action 不能替代客户端即时校验和敏感字段保护。

### 常见追问或变体

- 多个提交按钮如何区分 action？
- 和 `useOptimistic` 如何组合？

## 7. Vue `shallowRef`、`markRaw` 与第三方实例管理

### 参考答案

第三方实例通常包含大量内部字段、循环引用、DOM 引用和私有状态。如果被 Vue 深层代理，会带来不必要的性能开销，也可能破坏库内部的 `this`、原型判断或只读属性。

常见做法：

```ts
const chart = shallowRef<EChartsType | null>(null);

onMounted(() => {
  chart.value = markRaw(echarts.init(container.value));
});

onBeforeUnmount(() => {
  chart.value?.dispose();
  chart.value = null;
});
```

`shallowRef` 只追踪 `.value` 的替换，不深层代理实例；`markRaw` 明确告诉 Vue 不要把对象转成响应式。配置数据仍可以是响应式的，但传给实例前要控制更新频率。

### 关键考点

- 第三方实例不是业务状态，不需要深响应式。
- 生命周期里必须释放实例，避免 DOM 和事件监听泄漏。
- 响应式配置变化应节流或按需 `setOption`。

### 常见追问或变体

- `shallowReactive` 和 `shallowRef` 怎么选？
- 如果实例需要被多个组件共享，如何封装 composable？

## 8. CORS 预检、凭证请求与 SameSite Cookie

### 参考答案

简单请求通常限制在 `GET/HEAD/POST`、简单请求头和特定 `Content-Type`。使用自定义请求头、非简单方法或 JSON 等场景会触发浏览器先发送 `OPTIONS` 预检，确认服务端允许后再发送真实请求。

凭证请求需要同时满足：

- 前端：`fetch(url, { credentials: 'include' })` 或 XHR `withCredentials = true`。
- 服务端：`Access-Control-Allow-Credentials: true`。
- 服务端：`Access-Control-Allow-Origin` 必须是明确 Origin，不能是 `*`。
- Cookie：跨站场景通常需要 `SameSite=None; Secure`，并使用 HTTPS。
- 预检响应要允许对应 method 和 headers。

`*` 不能和凭证请求搭配，是因为浏览器要避免任意站点读取带用户身份的响应。

### 关键考点

- CORS 是浏览器同源策略下的读取限制，不是服务端防火墙。
- 预检缓存可通过 `Access-Control-Max-Age` 优化。
- Cookie 的 SameSite、Secure、Domain、Path 会共同影响发送。

### 常见追问或变体

- 为什么 Postman 能请求成功但浏览器失败？
- 反向代理统一同源能解决哪些问题？

## 9. 算法：滑动窗口最大值

### 参考答案

用双端队列保存候选下标，队列对应的值保持从大到小。新元素进入时，队尾所有小于等于它的下标都不可能再成为最大值，可以弹出；窗口左边界移动时，队头过期下标弹出。

```js
function maxSlidingWindow(nums, k) {
  const deque = [];
  const result = [];

  for (let i = 0; i < nums.length; i++) {
    while (deque.length && deque[0] <= i - k) deque.shift();
    while (deque.length && nums[deque[deque.length - 1]] <= nums[i]) deque.pop();
    deque.push(i);

    if (i >= k - 1) {
      result.push(nums[deque[0]]);
    }
  }

  return result;
}
```

每个下标最多入队一次、出队一次，总时间复杂度 `O(n)`，空间复杂度 `O(k)`。

### 关键考点

- 队列存下标，才能判断是否过期。
- 单调递减保证队头始终是当前窗口最大值。
- `shift` 在 JS 数组上可能是 `O(n)`，生产代码可用头指针优化。

### 常见追问或变体

- 如何改成滑动窗口最小值？
- 如果数据是流式到达，如何持续输出当前窗口最大值？

## 10. Next.js Parallel Routes 与 Intercepting Routes 的弹窗路由

### 参考答案

典型结构是列表页保留主内容，用 parallel slot 承载 modal；从列表点击详情时用 intercepting route 在当前布局中拦截详情页并渲染弹窗，直接访问详情 URL 时走正常详情页面。

示例结构：

```text
app/
  products/
    page.tsx
    [id]/page.tsx
    @modal/
      default.tsx
      (.)[id]/page.tsx
```

列表内 `<Link href="/products/123">` 会在客户端导航时显示 `@modal/(.)[id]`；刷新或直接打开 `/products/123` 时没有列表上下文，渲染 `products/[id]/page.tsx`。弹窗关闭可 `router.back()` 或跳回列表。

SEO 上完整详情页要有可索引内容和元信息；弹窗要处理焦点陷阱、Esc、滚动锁定和屏幕阅读器语义。不要只做纯客户端弹窗，否则 URL、刷新和分享体验会差。

### 关键考点

- Parallel Routes 让一个 layout 同时渲染多个 slot。
- Intercepting Routes 改变客户端导航时的呈现位置。
- 刷新行为和直接访问行为必须设计清楚。

## 11. Node.js BFF 中的级联取消与超时预算

### 参考答案

核心是把客户端连接、接口总预算和下游子请求预算串成同一套取消信号。Node.js 内置 `fetch` 支持 `AbortSignal`，可以用 `AbortController` 或 `AbortSignal.timeout` 控制超时。

```js
async function handler(req, res) {
  const clientAbort = new AbortController();
  req.on('close', () => clientAbort.abort(new Error('client closed')));

  const timeout = AbortSignal.timeout(800);
  const signal = AbortSignal.any([clientAbort.signal, timeout]);

  const calls = services.map(service =>
    fetch(service.url, { signal })
      .then(r => r.json())
      .catch(error => ({ degraded: true, service: service.name, error: error.name }))
  );

  const data = await Promise.all(calls);
  res.json({ data });
}
```

实际项目要把总预算拆给关键下游，例如总 800ms，核心库存 500ms，推荐服务 200ms；非核心失败返回降级结果，核心失败返回明确错误。日志要记录 traceId、下游耗时、取消原因和降级字段。

### 关键考点

- 客户端断开后继续请求下游会浪费资源。
- 超时要有总预算和子预算，不能每个下游各自无限等待。
- 取消不是错误吞掉，仍要区分 timeout、abort、业务失败。

### 常见追问或变体

- `Promise.all` 和 `Promise.allSettled` 在聚合接口中怎么选？
- 下游不支持取消时如何做连接池和并发保护？

## 12. NestJS Guard、Pipe、Interceptor、Filter 的执行顺序

### 参考答案

一次请求的大致链路是：Middleware 先处理原始请求；Guard 决定是否允许进入路由；Interceptor 在路由前后包裹执行；Pipe 在参数绑定阶段做转换和校验；Controller/Provider 执行业务；异常由 Exception Filter 捕获翻译。

职责放置：

- 登录鉴权：Guard，判断 token、权限、角色。
- 参数校验：Pipe，例如 `ValidationPipe`。
- 响应包装和耗时统计：Interceptor。
- 异常翻译：Exception Filter。
- 原始日志、body parser、跨域：Middleware。

Interceptor 可以在 controller 前后执行，但不应该承担权限判断的主职责；Filter 只处理异常，不应该包装正常响应。

### 关键考点

- Guard 早于 Pipe，未授权请求不应进入参数校验和业务。
- Pipe 面向参数级转换与校验。
- Interceptor 适合横切增强，比如缓存、日志、响应映射。

### 常见追问或变体

- 全局、Controller、Route 级别的执行顺序如何叠加？
- GraphQL 或 WebSocket 场景下生命周期有什么差异？

## 13. Redis 缓存穿透、击穿、雪崩与数据库一致性

### 参考答案

- 缓存穿透：请求不存在的数据，缓存和数据库都查不到。可用参数校验、布隆过滤器、空值短 TTL 缓存。
- 缓存击穿：热点 key 过期瞬间大量请求打到数据库。可用互斥锁重建、逻辑过期、热点 key 永不过期加异步刷新。
- 缓存雪崩：大量 key 同时失效或 Redis 故障。可用 TTL 随机抖动、多级缓存、限流降级、集群高可用。

商品详情设计：

1. key 带业务版本：`product:detail:v1:{id}`。
2. 正常 TTL 加随机抖动，热点商品逻辑过期。
3. 不存在商品写空值缓存，TTL 短。
4. 重建时用 Redis 锁或 singleflight，只有一个请求查 DB。
5. 商品更新时先写 DB，再删除缓存；必要时延迟双删或订阅 binlog 异步失效。

一致性上要接受缓存和数据库的短暂不一致，并用消息补偿、版本号或更新时间戳降低脏读窗口。

### 关键考点

- 三类缓存问题的触发条件不同。
- 热点 key 重建要防止并发打爆 DB。
- 更新缓存比删除缓存更容易写入旧值。

### 常见追问或变体

- Redis 故障时接口如何降级？
- 本地缓存和 Redis 缓存如何避免双层不一致？

## 14. AI Tool Calling 的前端进度态与人工确认

### 参考答案

前端不要只展示一段模型文本，而要把一次 AI 执行拆成可见状态：生成中、准备调用工具、等待确认、执行工具、工具成功、工具失败、继续生成、完成。

高风险工具必须二次确认，例如退款、发券、修改地址、删除数据、发送消息。确认页要展示工具名、关键参数、影响范围、可撤销性和操作者身份。低风险只读工具可以自动执行，但也应展示来源和耗时。

状态机示例：

```text
idle -> streaming
streaming -> tool_pending
tool_pending -> needs_confirmation | tool_running
needs_confirmation -> tool_running | cancelled
tool_running -> tool_succeeded | tool_failed
tool_succeeded -> streaming
tool_failed -> retrying | failed
streaming -> completed
```

工具参数要以结构化面板展示，避免藏在自然语言里。失败重试要区分模型重试、工具重试和用户修改参数后重试。

### 关键考点

- Tool Calling 是有副作用的系统交互，不只是聊天。
- 读操作和写操作的权限、确认、审计等级不同。
- 流式 UI 要能处理中途插入工具事件。

### 常见追问或变体

- 如何防止模型把用户未确认的参数传给工具？
- 工具执行一半失败，前端如何展示补偿动作？

## 15. Agent 运行追踪、回放与安全边界

### 参考答案

可观测方案要按一次 run 建立 trace，记录关键 span：用户输入、系统提示版本、检索 query、检索片段 ID、上下文压缩结果、模型请求摘要、工具选择、工具参数、权限判定、工具结果、最终输出和人工接管。每个 span 都要有 runId、traceId、耗时、状态和错误原因。

回放不是简单保存完整 prompt。推荐保存：

- 可复现元数据：模型名、参数、工具版本、提示词版本、检索索引版本。
- 输入输出摘要：敏感字段脱敏，必要时保存加密原文并限制访问。
- 工具调用：参数、返回摘要、错误码、权限快照。
- 上下文变化：哪些记忆被读取、压缩、丢弃。
- 安全事件：提示注入命中、越权拦截、PII 脱敏。

隐私边界上，日志默认脱敏，敏感字段按 allowlist 采集；回放环境必须禁用真实副作用工具或改用 sandbox/mock；审计后台要有访问控制和操作留痕。

### 关键考点

- Agent 调试需要 trace，而不是只看最终回答。
- 回放要固定工具、索引、提示词和权限版本。
- 观测数据本身可能包含隐私和商业机密。
