# 2026-07-21 全栈偏前端每日面试参考答案

## 1. JavaScript：`Array.fromAsync` 与异步迭代消费

**参考答案：**`Array.fromAsync` 消费 async iterable 或 sync iterable，并等待每个元素（及可选 mapFn）的结果；`Array.from` 只同步取值；`Promise.all` 等待一组已存在的 Promise。它会收集全部结果，不适合无限流或超大流。应使用 `for await...of` 边消费边处理，并把 `AbortSignal` 传给生产者；提前退出时调用迭代器的 `return()`（`for await` 的 `break` 会触发）。

**关键考点：**惰性消费不等于 `Array.fromAsync` 的整体结果惰性；Promise 数组在传入前可能已启动。

**常见追问或变体：**若需最多保留 N 条，使用固定窗口/批处理，不要先聚合为数组。

## 2. TypeScript：断言函数与边界数据收窄

**参考答案：**类型守卫返回布尔值，调用方需要在分支中使用；断言函数在正常返回后直接收窄，失败时必须抛错。

```ts
type User = { id: string; name: string };

function assertUser(value: unknown): asserts value is User {
  if (!value || typeof value !== 'object') throw new TypeError('invalid user');
  const v = value as Record<string, unknown>;
  if (typeof v.id !== 'string' || typeof v.name !== 'string') {
    throw new TypeError('invalid user');
  }
}
```

类型只在编译期存在；接口、localStorage、JSON 等运行时边界必须做实际校验，生产项目宜用 schema 库统一错误格式。

**关键考点：**`unknown` 先校验再使用；断言函数的实现错误会造成类型不安全。

**常见追问或变体：**区分缺字段、字段类型错与业务规则不满足，映射为稳定错误码。

## 3. ES6+：`Symbol` 作为协议键的设计边界

**参考答案：**局部 `Symbol()` 唯一且不会与普通属性或别的库协议名碰撞，可把约定暴露为常量：`obj[serialize]()`。`Symbol.for()` 使用全局注册表，同 key 可跨模块取得，隔离更弱。私有字段只允许类声明内部访问；WeakMap 可保存对象关联私有状态但需要持有 map 才能读。Symbol 不是安全边界：拿到 symbol 或反射到 symbol 属性仍可能访问。

**关键考点：**避免命名冲突、可发现性与封装不是一回事。

**常见追问或变体：**对跨包公共协议导出 symbol 常量；不要以 `Symbol.for` 承载秘密。

## 4. Promise：异步任务的“已开始”与“受控并发”

**参考答案：**Promise executor 创建时立即执行，所以并发池拿到 `Promise[]` 时请求通常已全部发出。传入 thunk 后，调度器仅在有空位时调用 thunk；调用本身用 `Promise.resolve().then(task)` 包住，才能把同步 throw 归一成 rejection。明确 fail-fast 或收集全部错误；取消要把 signal 传给任务，不能只停止调度。

**关键考点：**并发控制的是“启动时机”；取消需要底层操作配合。

**常见追问或变体：**重试应仅针对可重试错误，并计入总超时预算。

## 5. 浏览器：`ResizeObserver` 循环与布局抖动

**参考答案：**回调读取尺寸后同步写入影响自身/祖先布局的样式，浏览器再次通知，形成同一帧循环；多组件互相改尺寸会放大问题。先用 DevTools Performance、断点和日志定位写样式的调用链，比较写前后尺寸。修复为：只在尺寸实际变化时更新、把写入批到 `requestAnimationFrame`、用 CSS 约束替代 JS 调整，并设置合理最小/最大尺寸。不要只吞掉全局 error；那会隐藏视觉错误。

**关键考点：**观测回调不是任意同步布局写入点。

**常见追问或变体：**列表中复用 observer，卸载时 `unobserve`，避免回调持有已销毁节点。

## 6. 网络：`fetch` 请求体流、重定向与重试

**参考答案：**请求 body 是一次性可读流，读取/转发后通常 `bodyUsed`，不能原样再次发送。对可重试上传，应使用可重新创建的文件分片、每片 checksum 和 upload/session id；服务端以分片序号和幂等键去重。重试只针对网络错误或明确可重试状态，并受总截止时间约束。对跨域重定向白名单目的地；不要让客户端盲跟携带敏感凭证，必要时由 BFF 终止重定向后重新签名请求。

**关键考点：**流不可复用、重试的幂等前提、重定向信任边界。

**常见追问或变体：**小请求可在发送前 `clone()`，但仍要注意内存与浏览器限制。

## 7. React：`useSyncExternalStore` 的 hydration 契约

**参考答案：**`subscribe` 订阅变化并返回清理函数；`getSnapshot` 对未变化状态必须返回 `Object.is` 相同的缓存引用，否则会反复渲染。SSR 时 `getServerSnapshot` 的值必须与首次客户端渲染一致；浏览器专属信息可先返回服务器默认值，hydration 后通过订阅更新。不要在 render 中直接读会变化的 localStorage。

**关键考点：**服务端快照一致性、快照引用稳定、订阅清理。

**常见追问或变体：**若必须首屏展示真实偏好，可将安全的初始值序列化进 HTML，并防范 XSS。

## 8. Vue 3：自定义指令的资源生命周期

**参考答案：**在 `mounted` 创建/登记 observer，在 `updated` 比较 binding 后按需重绑，在 `unmounted` 取消观察并删除关联状态。可用一个 observer 按配置分组复用，用 `WeakMap<Element, Callback>` 保存回调。`KeepAlive` 的组件可能不会卸载；若元素持续留在 DOM，观察仍有效；若业务要求停用，应由组件 `onDeactivated` 控制开关或改为组件封装。

**关键考点：**指令只面向 DOM，资源清理由 `unmounted` 负责。

**常见追问或变体：**上报去重可记录首次可见状态；不要在每次 `updated` 无条件新建 observer。

## 9. 算法：分割等和子集

**参考答案：**总和为奇数直接 false；目标为 `sum / 2`。`dp[j]` 表示是否能凑出 j，初始 `dp[0] = true`。每个数从 target 倒序更新，防止同一元素在本轮被重复选取。

```ts
function canPartition(nums: number[]): boolean {
  const sum = nums.reduce((a, b) => a + b, 0);
  if (sum % 2) return false;
  const target = sum / 2;
  const dp = Array<boolean>(target + 1).fill(false);
  dp[0] = true;
  for (const n of nums) {
    for (let j = target; j >= n; j--) dp[j] ||= dp[j - n];
  }
  return dp[target];
}
```

时间复杂度 O(n × sum)，空间 O(sum)。

**关键考点：**0/1 背包、倒序更新。

**常见追问或变体：**若允许无限使用元素，改为正序更新（完全背包）。

## 10. Next.js：`generateStaticParams`、ISR 与新增路径

**参考答案：**`generateStaticParams` 只决定构建时预生成哪些参数，不会在 ISR revalidation 时自动再次运行以发现新 slug。`dynamicParams` 为 `true` 时，未预生成的有效 slug 可按需生成；为 `false` 时会 404。热门集合只取有限条，详情页以缓存/`revalidate` 控制更新；发布内容后通过按路径或标签失效，或等待/触发按需生成。数据源失效和路由是否允许按需生成必须一致。

**关键考点：**构建期枚举、按需路径、缓存失效是不同问题。

**常见追问或变体：**slug 不存在应由数据查询后 `notFound()`，不要把任意未知路径都视为文章。

## 11. Node.js：`AsyncLocalStorage` 上下文传播与泄漏

**参考答案：**在请求入口用 `asyncLocalStorage.run({ traceId }, handler)` 建立独立 store，在任意后续异步链用 `getStore()` 读取；不要把 store 对象放到全局变量。应用关闭或实例废弃时调用 `disable()`。原生 Promise、timer 通常会传播，但自行保存后由不受控线程/库在另一上下文调用的 callback、跨进程消息和部分任务队列需要在创建点用 `bind()`/`snapshot()` 包装，或显式携带 traceId。

**关键考点：**入口建上下文、跨边界显式传播、避免全局可变引用。

**常见追问或变体：**日志框架应在写日志时读取 store，而不是缓存上一个请求的字段。

## 12. NestJS：多 Transport 的 DTO 与错误契约

**参考答案：**将 DTO、schema、领域命令和错误码放在 transport-neutral 包；HTTP controller 和 message handler 都先执行同一输入校验，再调用 application service。领域层返回/抛出业务错误（如 `ORDER_ALREADY_EXISTS`），各 adapter 映射为 HTTP status 或消息 ack/nack/重试策略。幂等键与消费者去重记录应在应用服务/持久层实现，而非只靠 HTTP header。

**关键考点：**领域错误不等于 `HttpException`；消息失败分类决定是否重试。

**常见追问或变体：**schema 版本化，允许消费者在演进期兼容旧字段。

## 13. 数据库：唯一约束与“先查后写”竞态

**参考答案：**先查后写不是原子操作，唯一索引才是最终裁决。给规范化后的邮箱建立唯一约束；插入时捕获唯一冲突，并映射为稳定的“邮箱已注册”业务错误，而不是 500。预查询可以改善提示但不能代替约束；注册相关副作用应在插入成功后再发，或通过 outbox 投递。

**关键考点：**数据库约束、异常映射、大小写/空白规范化。

**常见追问或变体：**软删除若允许复用邮箱，要让唯一索引策略与业务规则一致。

## 14. AI 前端：流式输出中的敏感内容拦截

**参考答案：**把 token 放入本地暂存缓冲，审核通过后再按小段提交到可见消息；高风险场景可完全审核后展示。收到违规事件或审核失败时立即 abort 流、丢弃未提交缓冲，把已提交内容替换为统一安全提示，并记录模型版本、规则版本、request id 和时间线。重试创建新消息/版本，不能在原违规流上继续拼接。服务端也必须执行最终审核，前端不能作为唯一屏障。

**关键考点：**可见提交点、取消、不可篡改审计、双端治理。

**常见追问或变体：**缓冲会增加首 token 延迟，应按风险等级选择逐段审核或全量审核。

## 15. Agent/MCP：工具注解与用户授权界面

**参考答案：**不能。annotations 是服务器提供的提示，不是安全保证，客户端必须按自身策略和实际调用上下文授权。确认页展示工具名、关键参数的可读摘要、目标资源/租户、预计副作用、是否可撤销和授权有效期；高风险写操作要求显式确认。服务端仍做身份、scope、资源归属和参数校验，并记录用户、会话、工具版本、参数摘要、批准决定、结果和关联 traceId。不要把工具返回文本当作授权指令。

**关键考点：**提示性元数据不替代强制鉴权；授权最小化、可解释、可审计。

**常见追问或变体：**将只读、可逆写入、不可逆/外部写入分级，分别配置静默、一次确认和二次确认策略。
