## 1. `Promise.withResolvers()` 的适用场景与风险

### 参考答案

它返回 `{ promise, resolve, reject }`。语义上等价于在 `new Promise((res, rej) => ...)` 中保存两个函数，但写法更直接，适合 Promise 的完成时机由外部事件决定的场景。

```ts
function waitNextMessage(socket: WebSocket, signal?: AbortSignal) {
  const { promise, resolve, reject } = Promise.withResolvers<MessageEvent>();

  const cleanup = () => {
    socket.removeEventListener("message", onMessage);
    socket.removeEventListener("error", onError);
    signal?.removeEventListener("abort", onAbort);
  };
  const onMessage = (event: MessageEvent) => {
    cleanup();
    resolve(event);
  };
  const onError = () => {
    cleanup();
    reject(new Error("WebSocket error"));
  };
  const onAbort = () => {
    cleanup();
    reject(signal?.reason ?? new DOMException("Aborted", "AbortError"));
  };

  socket.addEventListener("message", onMessage, { once: true });
  socket.addEventListener("error", onError, { once: true });
  signal?.addEventListener("abort", onAbort, { once: true });
  return promise;
}
```

风险是把 `resolve`、`reject` 暴露到更大作用域后，所有权不清晰，容易重复完成、忘记清理监听器，或长期持有闭包引用。

### 关键考点

- 外部事件驱动的 Promise。
- Promise 只能首次完成，但副作用和监听器仍需主动清理。
- 取消不是 Promise 原生能力，通常要组合 `AbortSignal`。

## 2. TypeScript `NoInfer<T>` 与泛型推断控制

### 参考答案

普通写法会同时从 `options` 和 `defaultValue` 推断 `T`，非法默认值可能扩大联合类型。应阻止第二个参数参与推断：

```ts
declare function createSelect<T extends string>(
  options: readonly T[],
  defaultValue: NoInfer<T>,
): void;

createSelect(["red", "green"] as const, "red");  // 正确
createSelect(["red", "green"] as const, "blue"); // 类型错误
```

`NoInfer<T>` 仍然进行类型检查，只是不把该位置作为推断候选。类型断言则由开发者覆盖编译器判断，可能掩盖真实错误。

### 关键考点

- 泛型参数可能从多个参数位置共同推断。
- `NoInfer` 控制推断来源，不改变运行时代码。
- `as const` 用于保留字面量联合，而不是扩大为 `string[]`。

## 3. 手写支持取消、超时和重试的请求函数

### 参考答案

```ts
type RetryOptions = RequestInit & {
  timeoutMs: number;
  retries: number;
  retryStatus?: number[];
  signal?: AbortSignal;
};

async function requestWithRetry(url: string, options: RetryOptions) {
  const {
    timeoutMs,
    retries,
    retryStatus = [500, 502, 503, 504],
    signal,
    ...init
  } = options;

  const timeoutController = new AbortController();
  const timer = setTimeout(
    () => timeoutController.abort(new DOMException("Timeout", "TimeoutError")),
    timeoutMs,
  );
  const combinedSignal = AbortSignal.any([
    timeoutController.signal,
    ...(signal ? [signal] : []),
  ]);

  try {
    for (let attempt = 0; ; attempt++) {
      combinedSignal.throwIfAborted();
      try {
        const response = await fetch(url, { ...init, signal: combinedSignal });
        if (!retryStatus.includes(response.status) || attempt >= retries) {
          return response;
        }
      } catch (error) {
        if (combinedSignal.aborted || attempt >= retries) throw error;
      }

      const delay = Math.min(1000 * 2 ** attempt, 10_000);
      await new Promise<void>((resolve, reject) => {
        const id = setTimeout(resolve, delay);
        combinedSignal.addEventListener("abort", () => {
          clearTimeout(id);
          reject(combinedSignal.reason);
        }, { once: true });
      });
    }
  } finally {
    clearTimeout(timer);
  }
}
```

生产实现还应加入随机抖动，避免大量客户端同时重试。非幂等请求默认不应自动重试，除非服务端支持幂等键。

### 关键考点

- 总超时与单次超时的语义要明确。
- 用户取消由外部 Signal 触发；总超时由内部 Controller 触发。
- 仅重试可恢复错误，处理幂等性、退避和抖动。
- `finally` 清理总超时定时器。

### 常见追问或变体

- 如何兼容不支持 `AbortSignal.any()` 的环境？
- 如何为每次请求设置独立超时？
- `429` 应如何结合 `Retry-After` 处理？

## 4. BFCache 导致页面状态过期如何处理

### 参考答案

BFCache 保存的是完整页面快照，包括 JavaScript 堆和 DOM；返回时页面不是重新加载，而是恢复执行。HTTP 缓存缓存的是网络响应，重新导航时仍会经历页面创建流程。

可监听 `pageshow`，通过 `event.persisted` 判断本次是否从 BFCache 恢复：

```ts
window.addEventListener("pageshow", (event) => {
  if (event.persisted) {
    revalidateOrderList();
  }
});
```

更稳妥的策略是记录数据更新时间。恢复时先展示快照，若超过新鲜度阈值或关键业务状态可能变化，则后台重新验证；必要时显示“数据已更新”。避免使用 `unload` 监听，因为它会影响 BFCache 资格；清理逻辑优先放到 `pagehide`。

### 关键考点

- BFCache 是页面级内存快照，不是 HTTP 响应缓存。
- `pageshow.persisted`、`pagehide`。
- stale-while-revalidate 式刷新兼顾返回速度与新鲜度。

### 常见追问或变体

- 如何在 Chrome DevTools 中检查页面未进入 BFCache 的原因？
- WebSocket、IndexedDB 连接对 BFCache 有什么影响？
- React Query 等数据层如何在恢复时触发重新验证？

## 5. 多标签页登录态与数据同步

### 参考答案

同源现代浏览器可优先使用 `BroadcastChannel` 发布结构化事件，如 `TOKEN_UPDATED`、`LOGOUT`、`CART_CHANGED`。兼容方案可用 `localStorage` 写入带时间戳和随机 ID 的事件，由其他标签页监听 `storage`。

Token 刷新必须单飞：

1. 标签页发现 Token 即将过期后尝试获取短租约锁，可用 Web Locks API 或带过期时间的本地锁。
2. 获得锁的标签页执行刷新并广播结果。
3. 其他标签页等待结果，而不是同时请求。
4. 服务端仍要支持 Refresh Token 轮换、重放检测和失败后全局退出。

购物车等业务数据不应只广播最终本地快照，可广播“版本已变化”，其他页面从服务端重新拉取。冲突可通过服务端版本号、更新时间或 ETag 解决。

### 关键考点

- `BroadcastChannel` 简单直接，但只限同源且有兼容性边界。
- `storage` 事件不会在发起写入的当前页面触发。
- SharedWorker 可集中连接和状态，但部署、兼容成本更高。
- 敏感 Token 不应通过 URL、日志或不可信消息通道传递。

### 常见追问或变体

- 无 Web Locks API 时如何实现带租约的互斥？
- 标签页崩溃后如何释放锁？
- 如何处理用户在两个标签页同时编辑同一资源？

## 6. React 外部状态订阅与并发渲染一致性

### 参考答案

`useEffect` 在提交后才订阅，渲染读取与订阅建立之间存在窗口；外部 Store 若在此期间变化，组件可能漏掉更新。并发渲染还可能让不同组件读取到不同版本的状态。

```ts
let value = 0;
const listeners = new Set<() => void>();

const store = {
  getSnapshot: () => value,
  subscribe(listener: () => void) {
    listeners.add(listener);
    return () => listeners.delete(listener);
  },
  set(next: number) {
    value = next;
    listeners.forEach((listener) => listener());
  },
};

function Counter() {
  const count = useSyncExternalStore(
    store.subscribe,
    store.getSnapshot,
    () => 0,
  );
  return <button onClick={() => store.set(count + 1)}>{count}</button>;
}
```

`getSnapshot` 在 Store 未变化时必须返回 `Object.is` 相等的值，否则会无限重渲染。SSR 时 `getServerSnapshot` 的初始结果应与客户端水合时一致。

### 关键考点

- 读取快照和订阅之间的一致性协议。
- `subscribe` 返回取消订阅函数，引用应稳定。
- 可变 Store 应缓存不可变快照。
- SSR 初始快照必须可复现。

### 常见追问或变体

- 浏览器在线状态如何接入该 Hook？
- 为什么不能在 `getSnapshot` 中每次返回新对象？
- Zustand、Redux 等库如何处理外部 Store 订阅？

## 7. Vue 3 大型不可变数据的响应式优化

### 参考答案

`reactive` 和普通 `ref(object)` 会把对象深层转换为响应式代理，适合需要深层修改追踪的数据。`shallowReactive` 只代理根属性；`shallowRef` 只追踪 `.value` 的替换；`markRaw` 则让对象不被转换。

对于大型只读列表，可使用：

```ts
const rows = shallowRef<ReadonlyArray<Row>>([]);

async function reload() {
  rows.value = await fetchRows();
}
```

使用 `shallowRef` 后，`rows.value.push(item)` 不会触发更新，必须采用不可变更新：

```ts
rows.value = [...rows.value, item];
```

`markRaw` 更适合第三方类实例或明确不应响应式化的对象；滥用会导致 UI 不更新且难以排查。

### 关键考点

- 深层代理成本与数据规模有关。
- 浅响应要求根级替换。
- 响应式优化不能替代虚拟列表，十万 DOM 节点仍不可接受。

### 常见追问或变体

- `triggerRef` 适用于什么情况？
- `toRaw` 能否作为长期持有对象的手段？
- 如何结合虚拟滚动、分页和计算缓存？

## 8. 算法：数据流中的第 K 大元素

### 参考答案

维护一个大小最多为 `k` 的最小堆。堆中保留当前最大的 `k` 个元素，堆顶就是第 K 大。

```ts
class KthLargest {
  private heap: number[] = [];

  constructor(private k: number, nums: number[]) {
    nums.forEach((num) => this.add(num));
  }

  add(value: number): number {
    this.heap.push(value);
    this.up(this.heap.length - 1);
    if (this.heap.length > this.k) this.pop();
    return this.heap[0];
  }

  private up(index: number) {
    while (index > 0) {
      const parent = (index - 1) >> 1;
      if (this.heap[parent] <= this.heap[index]) break;
      [this.heap[parent], this.heap[index]] =
        [this.heap[index], this.heap[parent]];
      index = parent;
    }
  }

  private pop() {
    const last = this.heap.pop()!;
    if (!this.heap.length) return;
    this.heap[0] = last;
    let index = 0;
    while (true) {
      let smallest = index;
      const left = index * 2 + 1;
      const right = left + 1;
      if (left < this.heap.length && this.heap[left] < this.heap[smallest]) {
        smallest = left;
      }
      if (right < this.heap.length && this.heap[right] < this.heap[smallest]) {
        smallest = right;
      }
      if (smallest === index) break;
      [this.heap[index], this.heap[smallest]] =
        [this.heap[smallest], this.heap[index]];
      index = smallest;
    }
  }
}
```

每次 `add` 为 `O(log k)`，空间为 `O(k)`。因此无需保存完整数据流，适合总数据量远大于内存的场景。

### 关键考点

- 第 K 大使用大小为 K 的最小堆。
- 不要排序全部历史数据。
- 重复值也应参与排名。

### 常见追问或变体

- 如何求第 K 小？
- 如何一次性从数组中求第 K 大，Quickselect 的复杂度如何？
- `k` 动态变化时如何设计？

## 9. 前端灰度发布与快速回滚设计

### 参考答案

构建产物使用内容哈希并保持不可变，HTML 或版本清单只引用同一发布版本的资源。灰度规则在服务端或边缘层根据稳定用户 ID、地区和百分比计算，保证同一用户稳定落到同一版本。

发布流程：

1. 上传新静态资源，但暂不切流。
2. 灰度少量用户，观察 JS 错误率、白屏率、核心接口失败率、Web Vitals 和业务转化。
3. 指标超过阈值时停止扩量并切回旧 HTML/版本清单。
4. 旧版本资源保留足够时间，避免用户持有旧 HTML 时资源已被删除。
5. Service Worker 按版本管理 Cache Storage，先安装新缓存，激活后再清理旧缓存，避免半更新。

回滚应切换入口版本，而不是覆盖同名资源。关键功能还可用服务端 Feature Flag 做运行时止损。

### 关键考点

- 内容哈希、资源不可变、入口可切换。
- 灰度分桶必须稳定且可审计。
- 技术指标与业务指标共同判断。
- CDN、HTML、Service Worker 三层缓存一致性。

### 常见追问或变体

- 未登录用户如何稳定分桶？
- 前后端接口不兼容时如何灰度？
- Source Map 如何安全上传并关联发布版本？

## 10. Next.js Node.js Runtime 与 Edge Runtime 的边界

### 参考答案

Edge Runtime 启动和地域分布有优势，但 Web API 集合受限，不能假设所有 Node.js 原生模块、原生扩展和长连接数据库驱动可用。Node.js Runtime 兼容完整 Node 生态，更适合数据库连接池、文件系统和依赖原生模块的业务。

可将 Edge 中间件职责限制为：

- 读取 Cookie、校验轻量签名或进行快速重定向。
- 生成短生命周期、带签名的内部身份声明。
- 不执行复杂数据库权限查询。

Node Route Handler 再验证该声明的签名、有效期、受众和请求绑定信息，并执行最终授权。不能只信任客户端可伪造的请求头；边缘层应覆盖或移除外部同名头。

### 关键考点

- Runtime 选择取决于依赖能力、延迟、连接模型和部署平台。
- 认证解决“是谁”，授权仍应在业务边界最终校验。
- 内部身份传递需要签名、防重放和可信网络边界。

### 常见追问或变体

- Edge 环境如何访问数据库？
- Middleware 是否适合完成全部 RBAC 查询？
- 如何避免区域间身份密钥不一致？

## 11. Node.js 异步上下文与请求链路追踪

### 参考答案

```ts
import { AsyncLocalStorage } from "node:async_hooks";
import { randomUUID } from "node:crypto";

type Context = { requestId: string; userId?: string };
const requestContext = new AsyncLocalStorage<Context>();

function middleware(req, res, next) {
  requestContext.run(
    { requestId: req.headers["x-request-id"] ?? randomUUID() },
    next,
  );
}

function log(message: string) {
  console.log({ ...requestContext.getStore(), message });
}
```

它避免把上下文参数穿透每一层函数，也便于日志、Tracing 和审计统一读取。应为每个请求使用 `run` 建立独立上下文，避免用 `enterWith` 后忘记恢复。

上下文可能因自行封装的回调 API、错误复用 EventEmitter、在上下文外创建并共享异步资源等情况丢失。可用 `AsyncResource` 绑定自定义异步边界。Store 中不要放大对象或长期引用请求对象，以免增加内存压力。

### 关键考点

- 异步资源链上的上下文传播。
- 每请求隔离，不能把上下文放普通全局变量。
- 与结构化日志、OpenTelemetry Trace ID 对接。

### 常见追问或变体

- `run` 与 `enterWith` 有什么区别？
- Worker Thread 之间是否自动传播？
- 如何验证高并发下没有串请求？

## 12. NestJS 中统一日志、响应格式与异常处理

### 参考答案

合理分工如下：

- Middleware：请求进入框架早期的通用处理，如 request ID、基础访问日志。
- Guard：认证和授权，决定请求能否进入 Handler。
- Pipe：参数转换与校验。
- Interceptor：围绕 Handler 执行，统计耗时、包装成功响应、缓存或埋点。
- Exception Filter：捕获异常并映射为稳定的 HTTP 错误格式，同时记录错误上下文。

执行链可设计为：Middleware 建立链路上下文 → Guard 鉴权 → Pipe 校验参数 → Interceptor 计时并调用 Handler → 成功时包装响应；异常由 Filter 统一转换。领域层应抛领域异常，不应直接依赖 HTTP 异常。

### 关键考点

- 不要在多个层重复包装响应或重复打印同一异常。
- Filter 处理失败路径，Interceptor 适合成功路径和环绕逻辑。
- 日志需包含 request ID、路由、耗时、错误码，但要脱敏。

### 常见追问或变体

- Interceptor 能否捕获异常？
- ValidationPipe 应放全局还是局部？
- GraphQL 或微服务传输层下如何复用领域异常？

## 13. 数据库事务与消息投递一致性

### 参考答案

在创建订单的同一个本地数据库事务中，同时写入订单表和 Outbox 表。事务提交后，由独立 Relay 持续扫描未发送记录并投递消息。这样不会出现订单已提交但待投递事件完全丢失。

投递通常是“至少一次”，因此：

- Outbox 事件有全局唯一 `event_id`。
- 消费者使用幂等表、唯一键或业务状态机去重。
- Relay 成功投递后标记状态；失败指数退避，超过阈值进入死信和告警。
- 同一聚合的事件带版本号，按聚合 ID 分区，消费者拒绝旧版本或等待缺失版本。
- 监控 Outbox 积压量、最老事件年龄、失败次数和消费延迟。

不能简单使用“先写库再发消息”或“先发消息再写库”，两者都存在进程崩溃窗口。

### 关键考点

- 本地事务保证业务数据与事件记录原子提交。
- 端到端通常接受至少一次投递，并靠幂等收敛。
- Outbox 解决不丢，不天然保证全局严格顺序。

### 常见追问或变体

- Relay 如何避免多实例重复扫描？
- CDC 与轮询 Outbox 各有什么取舍？
- 消费成功但确认消息前进程崩溃怎么办？

## 14. AI 操作的人工审批与可恢复交互

### 参考答案

后端应保存持久化状态机，而不是仅靠前端内存：

`generating → awaiting_approval → approved/rejected/expired → executing → succeeded/failed`

模型只能生成“工具调用提案”，包含不可变 `proposalId`、工具名、参数、风险摘要和过期时间。前端流式展示推理结果后渲染审批卡片，允许用户在受控范围内修改参数；修改后应生成新版本并重新进行服务端校验。

批准接口使用幂等键和状态条件更新，只有 `awaiting_approval` 能进入 `approved`，防止重复点击。执行 Worker 再检查操作者权限、金额限制和提案版本。刷新页面后通过会话或任务 ID 拉取状态，并通过 SSE/WebSocket 恢复进度。拒绝、过期和执行失败均保留审计记录。

### 关键考点

- 生成建议与执行副作用必须分离。
- 服务端状态是事实来源，UI 只是投影。
- 审批不能等同于“点击按钮”，还需重新鉴权、校验和幂等。
- 展示最终生效参数，不能只展示模型自然语言。

## 15. Agent 工具返回内容中的提示注入防护

### 参考答案

网页、文档、邮件和 RAG 片段都属于不可信数据。它们可以作为事实依据，但不能自动升级为系统指令。防线应分层：

1. 提示层明确标记数据边界，要求模型忽略数据中的指令，但不能只依赖提示词。
2. Agent 使用最小权限工具；读取工具与写入、转账、导出工具分离。
3. 敏感工具参数由结构化 Schema 校验，并在服务端执行资源范围、租户、权限和数量限制。
4. 高风险操作要求人工确认，审批 UI 展示真实参数与影响范围。
5. RAG 按租户和 ACL 检索，避免模型看到无权访问的数据。
6. 对外输出执行 DLP、敏感信息过滤和目标域名限制。
7. 记录模型版本、输入来源、工具调用、审批人和结果，建立异常调用告警。

提示注入无法只靠“更强的系统提示”彻底解决，核心是即使模型被诱导，系统权限和执行层也限制其造成损害。

### 关键考点

- 指令与数据分离。
- 最小权限、默认拒绝、服务端强校验。
- 间接提示注入来自工具读取的外部内容。
- 防泄露与防越权需要确定性控制。
