# 2026-07-20 全栈偏前端每日面试参考答案

## 1. JavaScript：ES Module 的 live binding、循环依赖与初始化时序

- 参考答案：命名导入是只读的 live binding；导出方重新赋值后，导入方下次读取会得到新值。循环依赖会先建立绑定、再按依赖图求值；若模块顶层在对方初始化完成前读取 `let`/`const`/`class` 绑定，会落入 TDZ 并报错。将常量、类型和无副作用函数抽到第三模块；把相互依赖的顶层执行改为显式初始化函数或延后调用。
- 关键考点：模块链接与求值；live binding；TDZ；循环依赖。
- 常见追问或变体：CommonJS 的 `require` 暴露的是执行中模块的部分 `exports` 对象，语义不同。

## 2. TypeScript：`exactOptionalPropertyTypes` 与 PATCH 接口语义

- 参考答案：建议用显式指令表达 PATCH，例如 `{ displayName?: string | null }`：缺失表示不更新，`null` 表示清空；不以 `undefined` 承载业务语义。开启 `exactOptionalPropertyTypes` 后，`x?: T` 不再可随意赋 `undefined`，能发现模糊调用；`x: T | undefined` 则要求键存在。服务端仍需 schema/DTO 校验，区分字段是否存在并拒绝未知字段。
- 关键考点：可选属性；`undefined`；`null`；编译期与运行时边界。
- 常见追问或变体：JSON 会省略 `undefined`，因此客户端序列化前要先归一化。

## 3. Promise：thenable 吸收与重复 settle 防护

- 参考答案：Promise 必须“吸收”任意 thenable 的最终状态，而非把它当普通值；读取 `then` 也可能抛错。用一次性开关防止恶意 thenable 双调用，并拒绝以自身解析，避免循环。

```js
function resolvePromise(promise, x, resolve, reject) {
  if (promise === x) return reject(new TypeError('cycle'));
  if (x && (typeof x === 'object' || typeof x === 'function')) {
    let called = false;
    try {
      const then = x.then;
      if (typeof then !== 'function') return resolve(x);
      then.call(x,
        y => { if (!called) { called = true; resolvePromise(promise, y, resolve, reject); } },
        e => { if (!called) { called = true; reject(e); } },
      );
    } catch (e) { if (!called) reject(e); }
  } else resolve(x);
}
```

- 关键考点：thenable；访问器异常；once guard；自解析循环。
- 常见追问或变体：真实 Promise 实现还要保证回调异步排入微任务。

## 4. 浏览器：页面冻结、恢复与后台任务可靠性

- 参考答案：`visibilitychange` 用于降低轮询/动画频率；`pagehide` 的 `persisted` 可识别 BFCache，`pageshow` 后应恢复但重新校验数据版本。支持时在 `freeze` 释放可重建资源，在 `resume` 重新建连；WebSocket 以连接状态和服务端版本/序号恢复，不假设后台仍在运行。重要保存应在编辑过程中防抖持久化，不能依赖 `unload`，它不可靠且会破坏 BFCache。
- 关键考点：Page Lifecycle；BFCache；幂等恢复；版本校验。
- 常见追问或变体：`sendBeacon` 适合小型尽力而为的遥测，不适合关键业务写入。

## 5. 网络：`Vary`、CDN 缓存键与内容错配

- 参考答案：共享缓存以 URL 加上 `Vary` 指定请求头的值区分响应。语言可按规范化后的 `Accept-Language` 或明确 locale 路径纳入键；设备通常优先用响应式页面，确需差异时只用有限 device class；实验桶应是受控、低基数键。鉴权内容应 `private` 或绕过共享缓存。完整 Cookie 高基数且携带会话，会造成命中率崩溃和用户数据混淆。
- 关键考点：共享缓存；缓存键；变体爆炸；认证边界。
- 常见追问或变体：CDN 自定义缓存键时仍要让 origin 的 `Vary` 策略和边缘策略一致。

## 6. React：Suspense 瀑布、预加载与边界拆分

- 参考答案：在路由匹配或用户意图出现时启动数据/模块预加载，让子组件读取同一请求缓存，避免“渲染后才请求”的串行链。用一个能承载主内容的边界先稳定骨架，再把次要区块拆为独立边界；导航更新用 `startTransition` 保留旧界面直到新界面可用。边界过细会导致 loading 碎片化、协调成本和视觉跳动，应按用户可感知的加载单元划分。
- 关键考点：render-as-you-fetch；请求去重；Suspense；Transition。
- 常见追问或变体：错误与等待分开处理，Suspense 不是错误边界。

## 7. Vue 3：`computed`、`watch` 与副作用调度

- 参考答案：`computed` 应是无副作用的派生值；请求放入 `watch` 或显式 action。把筛选状态收敛为一个 query source，`watch` 中用 `onCleanup`/`AbortController` 取消旧请求，给请求加序号或 key，仅接收最新结果；相同参数去重。读取更新后 DOM 用 `flush: 'post'` 或 `nextTick`，不要让多个 watcher 交叉回写同一状态。
- 关键考点：派生状态；副作用；清理回调；竞态控制；flush。
- 常见追问或变体：深度监听大对象代价高，优先监听最小且稳定的序列化查询条件。

## 8. 手写：支持并发、超时与取消的 `mapLimit`

- 参考答案：维护下一个索引、运行数与结果数组；每个 worker 领取一个索引，结束后再领取。统一的 controller 把外部 signal 和每个任务的超时合并，取消后停止派发。默认 fail-fast：任一失败 abort 其余任务并 reject；也可显式提供 all-settled 策略。

```js
async function mapLimit(items, limit, mapper, { signal, timeout } = {}) {
  const controller = new AbortController();
  const cancel = () => controller.abort(signal?.reason || new Error('aborted'));
  signal?.addEventListener('abort', cancel, { once: true });
  const out = Array(items.length); let next = 0;
  const worker = async () => {
    while (!controller.signal.aborted) {
      const i = next++; if (i >= items.length) return;
      let timer;
      try {
        const task = mapper(items[i], i, controller.signal);
        out[i] = await (timeout ? Promise.race([
          task,
          new Promise((_, reject) => { timer = setTimeout(() => reject(new Error('timeout')), timeout); }),
        ]) : task);
      } catch (e) { controller.abort(e); throw e; }
      finally { clearTimeout(timer); }
    }
  };
  try {
    await Promise.all(Array.from({ length: Math.min(limit, items.length) }, worker));
    if (controller.signal.aborted) throw controller.signal.reason;
    return out;
  } finally { signal?.removeEventListener('abort', cancel); }
}
```

- 关键考点：并发上限；顺序写回；取消传播；失败策略。
- 常见追问或变体：`mapper` 必须协作式响应 signal，取消无法强杀已经执行的 JavaScript。

## 9. 算法：网格中的最短路径与状态扩展

- 参考答案：BFS 的状态为 `(row, col, remain)`，其中 `remain` 是剩余可消除障碍数；同一坐标但 remain 更多的状态更优，所以不能只以坐标判重。可用 `best[row][col]` 记录到达该格时最大的 remain，只在新的 remain 更大时入队。首次到终点即为最短步数，时间复杂度 `O(m*n*k)`，空间同量级。
- 关键考点：BFS 最短路；状态维度；支配关系剪枝。
- 常见追问或变体：若边权不全为 1，改用 Dijkstra 或 0-1 BFS。

## 10. Next.js：用户相关数据与缓存隔离

- 参考答案：公共商品数据可在不读取用户请求信息的服务端边界缓存；用户价格/权益在读取 session 后计算，使用私有/请求级缓存或把 identity 纳入明确缓存键，绝不写入共享公共缓存。认证读取下沉到最小需要它的组件或 Route Handler，避免静态布局被意外动态化。登录、登出、权益变更后，失效用户相关 tag/key 并在测试中以两个账号交叉读取验证不会串值。
- 关键考点：公共与私有数据；缓存键；动态 API；缓存失效测试。
- 常见追问或变体：缓存不是授权；每次敏感操作仍要在服务端重新授权。

## 11. Node.js：JWT 密钥轮换与 JWKS 缓存

- 参考答案：先解析受限大小的 header，以 `kid` 查本地 JWKS 缓存；未命中时 singleflight 刷新一次并重试，避免并发请求击穿 IdP。缓存按 HTTP 缓存头/合理 TTL 更新，保留短暂旧 key 以覆盖传播窗口；刷新失败只能在未过期的可信缓存内短暂降级，不能盲目跳过验签。校验签名算法白名单、`iss`、`aud`、`exp`、`nbf`、`iat`（按策略）及需要的 nonce/azp。
- 关键考点：`kid`；密钥轮换；并发合并；claim 验证；观测指标。
- 常见追问或变体：不要仅按 JWT header 声称的 `alg` 选择验签算法。

## 12. NestJS：多协议应用的认证与异常契约

- 参考答案：把身份到 principal、权限策略和领域错误定义放在 transport 无关的应用层；HTTP Guard 从 request 提取身份，WS Guard 从 handshake/connection 提取，消息消费者从 headers/envelope 提取。各 adapter 把同一 `ForbiddenError` 映射为 HTTP 状态、WS error event 或可重试/不可重试的消息结果。消息重试只针对瞬态错误，达到上限转死信；不要把 HTTP Exception Filter 原样复用到所有 transport。
- 关键考点：hexagonal boundary；ExecutionContext；领域错误；重试/死信。
- 常见追问或变体：鉴权失败通常不可重试，且日志不得记录 token 原文。

## 13. 数据库：避免偏移分页在高频列表中的漏读与重复

- 参考答案：使用 keyset/cursor pagination：固定 `ORDER BY created_at DESC, id DESC`，cursor 编码最后一条的 `(created_at, id)`；下一页条件为 `created_at < ts OR (created_at = ts AND id < id)`。建立与筛选条件匹配的复合索引，例如 `(tenant_id, status, created_at DESC, id DESC)`。cursor 应包含版本和筛选条件摘要；筛选变化就从第一页重新开始，不能复用旧 cursor。
- 关键考点：keyset pagination；稳定排序；tie-breaker；复合索引；cursor 契约。
- 常见追问或变体：更新排序字段仍会造成跨请求移动，需在产品上声明快照语义或使用快照版本。

## 14. AI 前端：流式生成的状态一致性与断线恢复

- 参考答案：每个生成任务有 `runId`、单调 `sequence`、事件类型和可选 resumable cursor；状态机至少包含 queued、streaming、awaiting-confirmation、completed、cancelled、failed。客户端按 runId/sequence 幂等合并，缺号时重连补拉；取消请求也带 runId，服务端确认后再落为 cancelled。最终消息由明确 completed 事件或服务端快照确认，工具卡片和结构化块独立版本化，刷新后从持久快照恢复。
- 关键考点：事件幂等；序号；状态机；断线续传；最终态。
- 常见追问或变体：不要用“流结束”一项信号判断完成，网络中断和服务端完成无法仅靠它区分。

## 15. Agent/MCP：工具输出 Schema 校验与不可信内容隔离

- 参考答案：为工具声明严格的 input/output JSON Schema，客户端对 `structuredContent` 做校验，失败时降级为不可执行的原始证据而非继续驱动流程。文本、网页和知识库内容全部标为不可信数据，与系统策略、用户意图、工具调用参数分区；模型只能从 schema 允许的字段构造后续动作。UI 只展示最小必要摘要和可溯源引用；发送、写库等高风险操作需基于具体参数的用户确认与短时授权。
- 关键考点：schema validation；数据/指令分离；provenance；human-in-the-loop；最小授权。
- 常见追问或变体：工具 annotations 也不能默认可信，服务端身份和能力声明需要独立验证。
