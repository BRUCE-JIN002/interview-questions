# 2026-07-14 全栈偏前端每日面试参考答案

## 1. JavaScript：`this` 绑定规则与回调丢失上下文

**参考答案：** 调用点决定普通函数的 `this`：`new` 绑定优先级最高；`call`、`apply`、`bind` 是显式绑定；`obj.fn()` 是隐式绑定；其余普通调用为默认绑定（严格模式为 `undefined`）。传递 `obj.method` 时只传了函数值，计时器调用它已不是对象方法调用。箭头函数捕获定义处词法 `this`，适合回调；`bind` 适合固定方法接收者；包装回调 `() => obj.method()` 可在调用时保留对象且便于读取最新方法。

**关键考点：** 调用点、箭头函数无自身 `this`、`bind` 不可重新绑定、严格模式。

**追问：** 为什么不应把需要动态调用者的原型方法全部写成箭头函数？

## 2. TypeScript：用泛型约束实现类型安全的事件表

**参考答案：** 将事件表作为泛型参数，事件名约束为 `keyof Events`，payload 由索引访问类型取得。

```ts
type Events = {
  login: { userId: string };
  error: { code: number; message: string };
  ready: void;
};

class Emitter<E extends Record<PropertyKey, unknown>> {
  private listeners: { [K in keyof E]?: Array<(payload: E[K]) => void> } = {};
  on<K extends keyof E>(type: K, fn: (payload: E[K]) => void) {
    (this.listeners[type] ??= []).push(fn);
  }
  emit<K extends keyof E>(type: K, payload: E[K]) {
    this.listeners[type]?.forEach(fn => fn(payload));
  }
}
```

无 payload 事件可用 `void`，调用时传 `undefined`；若要做到 `emit('ready')` 无参数，需要用条件元组类型重载。内部存储处的断言应局限在实现边界，公开 API 不可退化为 `any`。

**关键考点：** `keyof`、索引访问类型、泛型关联、条件元组。

## 3. ES2024+：正则 `v` 标志与 Unicode 字符集合

**参考答案：** `u` 使正则按 Unicode code point 处理；`v` 在此基础上提供更强的 Unicode Sets 语义，例如集合运算和可匹配字符串的 Unicode 属性。`[A-z]` 会误包含 `[\\]^_\``，且只覆盖 ASCII。可用 Unicode 属性匹配字母和数字，并单独规定长度、规范化和禁用字符策略；例如 `/^[\\p{L}\\p{N}]{1,24}$/v`（需按浏览器支持度降级）。前端只负责即时提示，服务端必须以同一业务规则校验、规范化并审计。

**关键考点：** code point、Unicode property escapes、字符归一化、前后端一致性。

**追问：** 为什么仅限制正则仍不能完全阻止视觉混淆字符？

## 4. 浏览器：`content-visibility` 的渲染优化与可访问性边界

**参考答案：** 对折叠外、可独立跳过渲染的卡片使用 `content-visibility: auto`，浏览器可跳过其内部 style/layout/paint；同时用 `contain-intrinsic-size` 预留近似高度，减少滚动时跳动。先用 Performance、LCP/INP 和真实低端设备验证，不要对首屏、展开动画中或依赖精确尺寸的节点盲加。浏览器为页面查找和锚点导航时可能要求渲染目标；程序化测量隐藏内容前也可能触发工作。键盘焦点、读屏顺序和跳转后的可见性必须做实际验收。

**关键考点：** 渲染跳过、固有尺寸占位、CLS、可访问性回归。

## 5. 网络：TLS 1.3 0-RTT 的性能收益与重放风险

**参考答案：** TLS 1.3 会话恢复可减少握手往返；0-RTT 允许客户端在恢复连接时随 early data 发送请求，从而降低回访延迟。但 early data 可能被攻击者重放，服务端不能假设它只执行一次，因此不能用于非幂等写操作。若启用，应仅允许安全/幂等的读取接口，明确拒绝带副作用的路由，并在服务端仍做鉴权、限流、重放检测和幂等保护；不要把它当作业务幂等的替代品。

**关键考点：** session ticket、early data、replay、幂等语义。

## 6. React：`useEffectEvent` 与副作用中的最新状态

**参考答案：** Effect 依赖应描述“何时重连”，这里是 `roomId`；消息到达后的非响应式逻辑可抽为 Effect Event，使它读取最新 `theme`、策略等而不成为重连依赖。Effect Event 只应从 Effect 或其清理函数触发，不能拿来绕过真实依赖或当作任意事件处理器。`useRef` 是较低层替代方案，需要每次渲染手动同步 `.current`；故意漏依赖会留下陈旧闭包且不受 lint 保护。

**关键考点：** reactive dependency、陈旧闭包、Effect Event 边界、连接生命周期。

**追问：** 哪些值仍必须在依赖数组中，即使回调能读到最新值？

## 7. Vue 3：`v-memo` 的跳过更新语义与误用

**参考答案：** `v-memo="[row.id, row.version]"` 表示依赖未变时复用该 VNode 子树，跳过其更新；列表中应同时使用稳定 `:key="row.id"`，因为 `key` 解决身份对齐，`v-memo` 解决更新跳过。依赖数组必须覆盖影响该行输出的所有响应式值，例如选中态或全局语言。若依赖遗漏，UI 会陈旧；带复杂 slot、transition、依赖隐式外部状态或本地状态需要频繁同步的区域，应先用 profiling 定位再决定，不能把它当通用优化开关。

**关键考点：** VNode 复用、稳定 key、依赖完整性、性能测量。

## 8. 手写：Koa 风格洋葱模型中间件组合

**参考答案：** 用递归 `dispatch(i)` 驱动下一个中间件；以递增索引拦截重复 `next()`，并用 `Promise.resolve` 同时接住同步抛错和异步 reject。

```ts
type Middleware<C> = (ctx: C, next: () => Promise<void>) => void | Promise<void>;

function compose<C>(middlewares: Middleware<C>[]) {
  if (middlewares.some(fn => typeof fn !== 'function')) throw new TypeError('middleware must be a function');
  return (ctx: C) => {
    let index = -1;
    const dispatch = (i: number): Promise<void> => {
      if (i <= index) return Promise.reject(new Error('next() called multiple times'));
      index = i;
      const fn = middlewares[i];
      if (!fn) return Promise.resolve();
      try { return Promise.resolve(fn(ctx, () => dispatch(i + 1))); }
      catch (error) { return Promise.reject(error); }
    };
    return dispatch(0);
  };
}
```

空数组返回的函数会直接 resolve。调用者应 `await` compose 的结果，让错误交给统一错误中间件或框架边界处理。

**关键考点：** 洋葱执行顺序、递归 dispatch、重复 next、防同步异常。

## 9. 算法：接雨水

**参考答案：** 双指针维护左右已见最高柱 `leftMax`、`rightMax`。较低一侧的水位已被另一侧更高边界保证，因此可立刻结算：若 `leftMax <= rightMax`，累加 `leftMax - height[left]` 并右移 left；否则对 right 对称处理。每个指针只移动一次，时间 `O(n)`、额外空间 `O(1)`。单调递减栈在遇到更高右边界时弹出凹槽底部，按左右边界计算一段水量，适合解释每个凹槽的构成。

**关键考点：** 循环不变量、较低边界可结算性、双指针、单调栈。

## 10. Next.js：多区域部署下的缓存失效与一致性

**参考答案：** 发布先将内容写入权威数据源并记录版本，再以带事件 ID 的 webhook 触发页面/tag 失效；消费者对事件 ID 去重并可安全重试。文章详情和列表可接受短暂最终一致，响应应带版本或更新时间；支付、权限等请求必须查询权威源或使用严格 `no-store` 策略。CDN、Next 数据缓存和应用内缓存要分别定义 TTL/失效责任，发布队列应限速、合并相同 tag 的请求，监控各区域命中率、失效延迟、回源量和旧版本比例。

**关键考点：** cache layers、最终一致、幂等 webhook、缓存雪崩、版本可观测性。

## 11. Node.js：优雅重启中的连接排空与请求截止时间

**参考答案：** 收到 `SIGTERM` 后立即将 readiness 置为失败并停止接新连接/新请求，同时停止领取后台任务；给现有请求设统一 deadline。上传、流式响应和可取消下游调用应关联 `AbortSignal`，到期后返回可识别错误并强制关闭残留 socket；最后关闭 server、DB、队列和日志 exporter。`terminationGracePeriodSeconds` 必须大于业务 deadline 并留出清理余量，负载均衡摘流与 preStop/readiness 更新共同避免新流量进入。

**关键考点：** SIGTERM、drain、deadline、AbortSignal、Kubernetes 生命周期。

## 12. NestJS：模块化配置与密钥轮换

**参考答案：** 以配置域拆分模块（数据库、支付、OAuth），通过受控 provider 注入已解析的配置对象，而不是在业务代码随处读取 `process.env`。启动期用 schema 校验必填项、格式和环境差异，失败即拒绝启动。密钥应来自 Secret Manager/KMS，日志只记录 key id、版本和失败类别；验证端同时接受当前与上一版本一段窗口，签发端只用当前版本，轮换后撤销旧版本并审计。租户密钥按租户 ID 在服务端解析和短时缓存，缓存失效需可控。

**关键考点：** config validation、依赖注入、secret manager、key id、双版本轮换。

## 13. GraphQL：Persisted Query、查询成本与 BFF 防护

**参考答案：** Persisted Query 由客户端发送已登记 query hash，服务端只执行已知文档或在开发阶段登记它，能减少传输、提升 CDN 命中并缩小任意查询面。还要在解析前限制 body 大小和查询深度，按字段权重计算 complexity，限制 list first/last，设置 resolver 超时/并发预算，并在 resolver 做字段级授权和数据过滤。关闭 introspection 只减少 schema 枚举，不会阻止攻击者构造深层查询、猜测字段或滥用已知查询，不能替代成本与权限控制。

**关键考点：** allowlist、query complexity、pagination cap、resolver authorization、resource budget。

## 14. AI 前端：结构化输出的增量校验与降级展示

**参考答案：** 传输层先按事件/字节分片重组；JSON 未闭合前只作为内部 buffer 或纯文本预览，不能当对象使用。对象完整后执行 JSON parse 与 schema 校验，只有通过的字段才能进入可交互计划卡片；可为已闭合的独立 item 做逐项校验以渐进展示。UI 至少区分 generating、validating、partial-valid、complete、invalid、interrupted；校验失败时保留原始文本/重新生成入口，隐藏预订、支付等操作。服务端仍是执行与权限校验的最终边界。

**关键考点：** framing、schema validation、渐进渲染、fail closed、执行边界。

## 15. Agent：多 Agent 并行执行的状态归并与副作用控制

**参考答案：** 编排器为每个子任务定义 schema 化输入输出、trace/run/task ID、只读或有副作用属性、依赖和 deadline；只对无依赖任务并行，聚合器按来源、置信度、版本和完成状态合并结果，最终回答由一个明确的责任主体生成。取消和超时应从父 run 向子任务与工具传播，并记录部分完成结果。副作用任务先创建带幂等键的持久化任务记录，再执行；重试复用同一键和结果，必要时要求人工确认或串行化。全链路记录授权决定、工具参数摘要、耗时、失败和归并依据。

**关键考点：** DAG、structured contract、cancellation propagation、idempotency key、traceability。

**追问：** 为什么“让模型判断是否重试”不能替代服务端幂等约束？
