## 1. ES Modules 与 CommonJS 的加载差异

### 参考答案

CommonJS 是运行时加载，`require` 执行后得到导出对象的当前值，常见于 Node.js 旧项目。ES Modules 是静态模块结构，导入导出在编译阶段可分析，导出是 live binding，更利于 Tree Shaking。

循环依赖中，CommonJS 更容易拿到未初始化完成的导出对象；ESM 会建立绑定关系，但访问未初始化的绑定可能触发暂时性死区错误。迁移到 ESM 时要检查 `__dirname`、`__filename`、`require.resolve`、JSON/非 JS 资源导入、默认导出互操作、包的 `type` 字段、测试工具和构建配置。

### 关键考点

- CJS 运行时加载，ESM 静态结构。
- ESM 导出是绑定，不是简单值拷贝。
- ESM 更利于静态分析和 Tree Shaking。
- 迁移时重点关注互操作和工具链。

### 常见追问或变体

- `import()` 和静态 `import` 有什么区别？
- 为什么混用默认导出和命名导出容易出兼容问题？
- 前端打包器如何利用 ESM 做 Tree Shaking？

## 2. DOM 事件传播、委托与默认行为

### 参考答案

事件传播包含捕获、目标、冒泡三个阶段。`stopPropagation` 阻止事件继续传播，`stopImmediatePropagation` 还会阻止同一元素上后续监听器执行，`preventDefault` 阻止默认行为，例如链接跳转或表单提交。

列表事件委托可以把点击监听放在父容器，通过 `event.target.closest('[data-row-id]')` 找到行，再判断是否命中按钮、链接、输入框等交互元素。对按钮执行自己的逻辑时阻止冒泡；对链接通常保留默认跳转，除非业务明确接管导航。

### 关键考点

- 传播控制和默认行为控制是两类问题。
- 事件委托依赖冒泡和目标查找。
- 嵌套交互元素要明确点击优先级。

## 3. TypeScript 类型收窄与自定义类型守卫

### 参考答案

常见收窄方式包括 `typeof`、`instanceof`、`in`、判空、字面量判别字段、控制流分析和自定义类型守卫。自定义守卫的返回类型使用 `value is User`，让调用处获得明确类型。

```ts
type User = { type: 'user'; id: string; name: string; email?: string };
type Org = { type: 'org'; id: string; title: string };

function isUser(value: User | Org | null): value is User {
  return Boolean(value && value.type === 'user' && 'name' in value);
}

function render(value: User | Org | null) {
  if (isUser(value)) {
    return value.name;
  }
  return value?.id ?? 'empty';
}
```

滥用 `as` 会绕过编译器检查，导致运行时读取不存在字段、分支逻辑错误、后续重构失去类型保护。

### 关键考点

- 类型守卫是运行时判断和编译期收窄的结合。
- 判别联合适合复杂业务状态。
- `as` 应用于边界收口，不应用于掩盖建模问题。

### 常见追问或变体

- 如何给后端枚举变更设计类型保护？
- `satisfies` 是否能替代类型守卫？

## 4. 手写 `Object.create` 与原型链判断

### 参考答案

`Object.create(proto)` 创建一个新对象，并把新对象的内部原型指向 `proto`。简化实现可以借助空构造函数。

```js
function objectCreate(proto) {
  if (proto !== null && (typeof proto !== 'object' && typeof proto !== 'function')) {
    throw new TypeError('Object prototype may only be an Object or null');
  }
  function F() {}
  F.prototype = proto;
  return new F();
}

function isPrototypeInChain(proto, obj) {
  let current = Object.getPrototypeOf(obj);
  while (current) {
    if (current === proto) return true;
    current = Object.getPrototypeOf(current);
  }
  return false;
}
```

`new` 会创建对象、绑定原型、执行构造函数，并在构造函数没有返回对象时返回新对象；`Object.create` 只负责设置原型，不执行构造逻辑。

### 关键考点

- 原型链本质是对象之间的内部原型引用。
- `Object.create(null)` 可创建无原型对象。
- `new` 比 `Object.create` 多构造函数执行步骤。

## 5. Promise 错误冒泡与 `unhandledrejection`

### 参考答案

Promise 回调中 `throw` 会让返回的 Promise 变为 rejected；`return Promise.reject(error)` 效果类似。`catch` 如果没有继续抛错或返回 rejected Promise，后续 `then` 会进入 fulfilled 分支。

`unhandledrejection` 用于兜底收集未被捕获的 Promise 错误，适合做监控上报、日志补充和发现遗漏分支。它不适合作为主要业务错误处理，因为触发时机偏兜底，缺少业务上下文，也无法可靠恢复当前交互状态。

### 关键考点

- Promise 链会把同步异常转换成 rejected。
- `catch` 本身也会返回新的 Promise。
- 全局兜底只负责观测，不负责业务恢复。

### 常见追问或变体

- `finally` 会不会改变链路状态？
- `try/catch` 为什么捕不到未 `await` 的 Promise？
- 前端错误监控如何区分资源错误和 Promise 错误？

## 6. React Compiler 与手动 memo 优化取舍

### 参考答案

手动使用 `memo`、`useMemo`、`useCallback` 可以减少部分重复渲染，但会增加依赖数组维护成本，可能引入陈旧闭包、错误缓存和可读性下降。很多场景下，缓存本身的比较成本也可能高于重新计算。

React Compiler 的目标是通过编译期分析自动优化组件和表达式。引入前应先确认项目是否满足规则，例如组件纯度、Hook 规则、不可变数据更新和受支持语法。迁移策略应包括：保留必要的语义型 memo，删除明显无效的手动优化要谨慎分批；补充性能基线、交互耗时、渲染次数和线上错误监控；让 Lint 与编译器配置保持一致。

### 关键考点

- 性能优化要有测量基线。
- 手动 memo 不是越多越好。
- 编译期优化依赖组件纯度和规则约束。

### 常见追问或变体

- 哪些场景仍然需要手动 `useMemo`？

## 7. Vue 3 `Suspense`、异步组件与加载态设计

### 参考答案

异步组件适合按需加载体积较大的组件，`Suspense` 适合统一管理组件树中的异步依赖和 fallback。复杂页面不应把所有异步内容包在一个顶层加载态里，否则任何一个慢接口都可能造成全页不可用。

更稳妥的设计是按业务区块拆分异步边界：核心内容优先渲染，非核心模块局部骨架屏；失败时展示局部错误和重试入口；对推荐、广告、扩展信息等弱依赖模块允许降级隐藏；路由切换时保留上一屏关键状态，避免白屏。

### 关键考点

- 异步边界应贴近业务模块。
- 加载、错误、空态和重试需要一起设计。
- 非核心模块应支持局部降级。

### 常见追问或变体

## 8. 算法：最长无重复字符子串

### 参考答案

使用滑动窗口和哈希表记录字符最近出现位置。右指针逐个扫描字符，如果当前字符在窗口内出现过，就把左指针移动到上次出现位置的下一位。

```ts
function lengthOfLongestSubstring(s: string): number {
  const lastIndex = new Map<string, number>();
  let left = 0;
  let ans = 0;

  for (let right = 0; right < s.length; right++) {
    const ch = s[right];
    const prev = lastIndex.get(ch);
    if (prev !== undefined && prev >= left) {
      left = prev + 1;
    }
    lastIndex.set(ch, right);
    ans = Math.max(ans, right - left + 1);
  }

  return ans;
}
```

时间复杂度 `O(n)`，空间复杂度取决于字符集大小。

### 关键考点

- 左指针只能向右移动，不能回退。
- 只有重复字符出现在当前窗口内才移动左边界。
- 哈希表存最近出现位置。

### 常见追问或变体

## 9. 前端大文件上传的切片、断点续传与秒传

### 参考答案

前端可先计算文件 hash，把文件按固定大小切片，为每片生成序号、范围、hash 和状态。上传前向服务端查询已上传切片：如果完整文件 hash 已存在，直接秒传；否则只上传缺失切片。上传过程维护任务状态、切片状态、进度、失败次数和暂停标记。

服务端应校验切片 hash、大小和序号，所有切片完成后再合并，并校验最终文件 hash。前端限制并发数，对单片做有限重试，暂停时中止未完成请求并保留状态。为避免损坏，合并应由服务端原子执行，并防止同一文件多客户端重复触发合并。

### 关键考点

- 文件级 hash 用于秒传和最终校验。
- 切片级状态用于断点续传。
- 并发、重试、暂停和合并都需要协议配合。

### 常见追问或变体

- 浏览器端计算大文件 hash 如何避免卡顿？
- 上传凭证过期如何处理？
- 多文件队列如何做全局并发控制？

## 10. Next.js 服务端组件中的数据获取边界

### 参考答案

Server Components 可以在服务端读取数据库、调用内部服务、使用密钥并返回可序列化的渲染结果。Client Components 运行在浏览器，适合处理状态、事件、动画和浏览器 API。

不应把服务端密钥、数据库连接、大型原始对象或敏感字段传入客户端组件，因为这会扩大泄露面、增加 bundle 或序列化成本。交互型筛选页可以让服务端组件负责读取初始数据和安全边界，客户端组件负责筛选控件、局部状态和触发 URL/search params 更新，再由服务端重新获取数据。

### 关键考点

- 服务端组件适合数据和安全边界。
- 客户端组件适合交互和浏览器状态。
- 跨边界传递的数据必须可序列化且最小化。

## 11. Node.js BFF 聚合接口的超时与降级

### 参考答案

BFF 聚合接口应为每个下游设置独立超时和整体超时。核心服务失败时返回明确错误；非核心服务失败时可以返回部分数据和降级标记。并发请求可使用有限并发或 `Promise.allSettled`，避免一个失败导致所有结果丢失。

重试只适合短暂网络错误、幂等接口和有限次数，并加退避。日志需要记录 trace id、下游耗时、超时、重试次数、降级原因和返回数据完整性。避免拖垮进程的关键是限制并发、设置超时、控制响应体大小、避免 CPU 密集任务阻塞事件循环，并通过熔断或限流保护下游。

### 关键考点

- 区分核心依赖和非核心依赖。
- 超时、重试和降级必须有上限。
- BFF 的可观测性要覆盖下游维度。

## 12. NestJS WebSocket 鉴权与连接状态管理

### 参考答案

WebSocket 可在握手阶段从 header、cookie 或 query 中读取 token，通过 Guard 或 Gateway 钩子完成校验，并把用户信息挂到 socket 上。建立连接后，用内存 Map 或 Redis 维护 userId 到 socketId 的映射，多实例部署时需要 Redis Adapter 或消息总线同步房间和广播。

心跳用于发现断开连接，断线重连需要客户端重新鉴权并恢复订阅房间。权限变更时应主动刷新服务端连接状态，必要时踢出连接或撤销房间。Token 过期不能只依赖首次握手，长连接中应设计续期、服务端关闭并要求重连，或对关键事件再次校验权限。

### 关键考点

- WebSocket 首次鉴权不等于永久授权。
- 多实例连接状态需要外部共享或适配器。
- 房间、心跳、重连和权限变更要成套设计。

### 常见追问或变体

- Gateway 中如何拿到当前用户？
- 如何避免同一账号多端连接互相覆盖？
- WebSocket 消息如何做幂等和 ACK？

## 13. 接口版本演进与向后兼容

### 参考答案

## 14. AI 前端中的流式 Markdown 渲染安全

### 参考答案

流式 Markdown 渲染的风险包括：模型输出恶意 HTML、链接钓鱼、代码块未闭合导致布局异常、增量解析抖动、XSS、以及把工具返回的非可信内容直接渲染。方案上应把 Markdown 解析、HTML 白名单净化和链接策略分层处理。

前端可以先按文本流追加到缓冲区，使用支持增量或容错的 Markdown 渲染方式；代码块未闭合时使用临时展示状态，闭合后再高亮；禁止直接渲染原始 HTML，或只允许严格白名单；链接默认加 `rel="noopener noreferrer"`，外链做安全提示；工具输出、检索片段和模型回答都按不可信输入处理。

### 关键考点

- AI 输出不是可信 HTML。
- 流式渲染要处理半截 Markdown。
- 安全净化、链接策略和代码高亮要分层。

### 常见追问或变体

- 如何处理模型输出的 `<script>` 或事件属性？
- 为什么服务端净化和客户端净化可能都需要？

## 15. Agent 长对话记忆压缩与上下文预算

### 参考答案

短期上下文应保留当前目标、最新用户约束、正在修改的文件、未解决错误、当前计划和最近工具结果。长期记忆适合保存稳定偏好、项目约定、已验证命令、环境限制、历史决策和可复用排错结论。

压缩策略可以分层：每轮保留原始最近消息；阶段结束时生成结构化摘要，包括已完成、待办、阻塞、文件、命令和验证结果；重要约束使用明确字段保存，不只放自然语言总结。更新记忆前应去重并标记时间，避免旧结论覆盖新事实。恢复上下文时优先加载目标、约束、文件状态和失败证据。

### 关键考点

- 上下文压缩要保留可执行状态，而不只是聊天摘要。
- 稳定偏好和当前任务状态应分开存储。
- 记忆可能过期，需要时间戳和验证机制。

### 常见追问或变体

- 如何防止用户隐私或密钥进入长期记忆？
- Agent 多工具调用时如何记录可审计轨迹？
- 压缩摘要错误会导致什么连锁问题？
