## 1. JavaScript `Proxy` 不变量与可撤销代理

### 参考答案

Proxy trap 仍必须满足目标对象的语言不变量。例如不可配置且不可写的数据属性不能被 `get` 伪造成另一个值，目标不可扩展时 `ownKeys` 不能增删键。违反约束会抛出 `TypeError`。`Reflect` 提供与内部操作相近的默认语义，便于转发，但对 receiver 的处理仍要谨慎。

```js
const { proxy, revoke } = Proxy.revocable(model, {
  get(target, key, receiver) {
    return Reflect.get(target, key, receiver);
  },
});

useModel(proxy);
revoke(); // 此后任何代理操作都会抛 TypeError
```

撤销代理切断的是访问入口；它不会冻结原对象，持有原对象的代码仍可修改。`Object.freeze` 限制目标自身的扩展和属性写入，也不等于深冻结。

### 关键考点

- trap 不能破坏 ECMAScript 对象不变量。
- `Reflect` 用于语义一致的默认转发。
- 撤销、冻结和访问器是不同维度的机制。

### 常见追问或变体

## 2. TypeScript 条件类型的分发与关闭分发

### 参考答案

当检查端是裸类型参数时，联合类型会逐成员代入并重新合并：

```ts
type ToArray<T> = T extends unknown ? T[] : never;
type ToArrayNonDist<T> = [T] extends [unknown] ? T[] : never;

type A = ToArray<string | number>;        // string[] | number[]
type B = ToArrayNonDist<string | number>; // (string | number)[]
```

元组包装让检查端不再是裸类型参数，从而关闭分发。`never` 是空联合，分发后仍为 `never`；检测它常用 `[T] extends [never]`。`any` 可能同时走真假分支并传播不安全性，`unknown` 更适合作为安全顶层类型。

### 关键考点

- 分发发生的语法条件，而非所有条件类型都会分发。
- 联合数组与元素为联合类型的数组不同。
- `never`、`any` 的特殊行为需要显式测试。

### 常见追问或变体

- 如何实现 `IsUnion<T>`？
- `infer` 在协变和逆变位置会得到什么结果？

## 3. 手写带 `leading`、`trailing`、`cancel`、`flush` 的 debounce

### 参考答案

```js
function debounce(fn, wait, { leading = false, trailing = true } = {}) {
  let timer = null;
  let lastArgs;
  let lastThis;
  let result;

  const invoke = () => {
    const args = lastArgs;
    const context = lastThis;
    lastArgs = lastThis = undefined;
    result = fn.apply(context, args);
    return result;
  };

  function debounced(...args) {
    lastArgs = args;
    lastThis = this;
    const callNow = leading && timer === null;
    clearTimeout(timer);
    timer = setTimeout(() => {
      timer = null;
      if (trailing && lastArgs && (!leading || !callNow)) invoke();
      else lastArgs = lastThis = undefined;
    }, wait);
    if (callNow) return invoke();
    return result;
  }

  debounced.cancel = () => {
    clearTimeout(timer);
    timer = null;
    lastArgs = lastThis = undefined;
  };
  debounced.flush = () => {
    if (timer === null || !lastArgs) return result;
    clearTimeout(timer);
    timer = null;
    return invoke();
  };
  return debounced;
}
```

契约应明确：leading 与 trailing 同开时，单次调用通常只执行 leading；等待窗口内再次调用才补 trailing。异步 `fn` 的调用结果是 Promise，但未真正执行的调用无法得到“自己的”新 Promise，生产代码应明确定义返回语义。

### 关键考点

- 保留最后一次参数和 `this`。
- `cancel` 必须清理全部待执行状态。
- 边界语义比定时器代码本身更重要。

## 4. Speculation Rules 预渲染的性能与安全边界

### 参考答案

prefetch 主要预取文档供后续导航使用；prerender 会在隐藏上下文中提前加载并执行页面，导航更快但 CPU、内存和网络成本更高。候选 URL 应同源、低风险且基于明确意图选择，避免全站枚举。

GET 页面必须无副作用；埋点应等 `document.prerendering` 结束或 `prerenderingchange` 后发送。库存扣减、登出等必须由显式用户动作和受保护的非 GET 请求触发。登录态页面要防止共享缓存泄露，并通过真实命中率、激活延迟、误预测流量和内存评估收益。不支持时正常导航即可。

### 关键考点

- prerender 会执行页面代码，不只是下载资源。
- 安全 HTTP 方法和激活时机决定副作用边界。
- 预测准确率决定性能收益是否值得。

### 常见追问或变体

- 如何用 eagerness 控制推测强度？

## 5. HTTP `103 Early Hints` 与关键资源加载

### 参考答案

源站可在最终响应前先返回：

```http
HTTP/1.1 103 Early Hints
Link: </app.css>; rel=preload; as=style
Link: </font.woff2>; rel=preload; as=font; type="font/woff2"; crossorigin
```

浏览器据此提早发现资源，但最终 `200` 仍需发送完整响应。HTML preload 要等首批 HTML 到达；Server Push 由服务端主动传资源，缓存与带宽浪费更难控制，103 只是提示客户端自行请求。

只应提示最终响应也允许加载的可信资源，避免经重定向泄露敏感 URL。上线要确认 CDN/代理是否透传，并比较 LCP、资源请求提前量、重复下载率及无 103 对照组。

### 关键考点

- 103 是临时响应，不替代最终状态码。
- `as`、CORS 属性必须与真实请求匹配。
- 效果依赖中间设施和服务端思考时间。

### 常见追问或变体

- 103 能否改善所有页面的 LCP？
- 为什么错误的 preload 可能争抢关键带宽？

## 6. React 19 中 `ref` 作为 prop 与命令式句柄

### 参考答案

```tsx
type InputHandle = { focus(): void; clear(): void };

function SmartInput({ ref }: { ref: React.Ref<InputHandle> }) {
  const inputRef = useRef<HTMLInputElement>(null);
  useImperativeHandle(ref, () => ({
    focus: () => inputRef.current?.focus(),
    clear: () => {
      if (inputRef.current) inputRef.current.value = '';
    },
  }), []);
  return <input ref={inputRef} />;
}
```

只暴露必要能力能保留封装，避免父组件修改内部 DOM、样式或监听器。句柄引用在依赖不变时保持稳定；若方法闭包读取 props/state，应加入依赖或让方法读取稳定 ref。React 18 兼容构建仍需 `forwardRef`，组件库可通过主版本、不同入口或最低 peer dependency 明确边界。

### 关键考点

- React 19 的 ref prop 减少了 `forwardRef` 样板。
- 命令式 API 应最小化。
- 依赖遗漏会产生陈旧闭包。

## 7. Vue `customRef` 实现防抖输入的响应式契约

### 参考答案

```ts
import { customRef, onScopeDispose } from 'vue';

function useDebouncedRef<T>(initial: T, delay = 200) {
  let value = initial;
  let timer: ReturnType<typeof setTimeout> | undefined;
  onScopeDispose(() => clearTimeout(timer));

  return customRef<T>((track, trigger) => ({
    get() {
      track();
      return value;
    },
    set(next) {
      clearTimeout(timer);
      timer = setTimeout(() => {
        value = next;
        trigger();
      }, delay);
    },
  }));
}
```

getter 被消费时调用 `track`，真正提交新值后调用 `trigger`。getter 每次创建新对象会让子组件看到引用变化，即使 customRef 没有触发父组件更新，也可能造成依赖判断不一致。若防抖只属于一次搜索请求，更清晰的做法通常是在事件或请求层防抖并保留输入即时状态。

### 关键考点

- 自定义 ref 自行控制依赖收集和触发时机。
- 返回值的引用稳定性属于响应式契约。
- effect scope 销毁时要释放计时器。

### 常见追问或变体

- `shallowRef` 与 `customRef` 的用途有何不同？

## 8. Service Worker 更新中的多版本一致性

### 参考答案

新 Worker 下载后进入 install，成功后通常 waiting，直到旧 Worker 不再控制客户端才 activate。activate 中删除旧缓存，随后可 `clients.claim()` 控制未受控客户端。缓存名应带构建版本，HTML/API 契约要向后兼容，页面与 Worker 可交换版本号并提示用户刷新。

`skipWaiting()` 让新 Worker 跳过等待，但已打开页面的 JS 仍可能是旧版本；新 Worker 立即接管其请求会形成混合版本。因此只有协议兼容、发布策略明确时才强制更新。更稳妥的是提示用户，在确认后让新 Worker 激活并统一 reload 所有标签页。

### 关键考点

- Worker、页面资源和缓存可能同时存在多个版本。
- cache-first 不能忽略版本与失效。
- 强制激活不是无成本的“立即更新”。

### 常见追问或变体

- 如何避免离线页面被清缓存后不可用？

## 9. 算法：并查集判断动态连通性

### 参考答案

```ts
class DSU<T> {
  private parent = new Map<T, T>();
  private size = new Map<T, number>();

  find(x: T): T {
    if (!this.parent.has(x)) {
      this.parent.set(x, x);
      this.size.set(x, 1);
    }
    const p = this.parent.get(x)!;
    if (p !== x) this.parent.set(x, this.find(p));
    return this.parent.get(x)!;
  }

  union(a: T, b: T): boolean {
    let ra = this.find(a), rb = this.find(b);
    if (ra === rb) return false;
    if (this.size.get(ra)! < this.size.get(rb)!) [ra, rb] = [rb, ra];
    this.parent.set(rb, ra);
    this.size.set(ra, this.size.get(ra)! + this.size.get(rb)!);
    return true;
  }

  connected(a: T, b: T) { return this.find(a) === this.find(b); }
}
```

逐边调用 `union`，第一次返回 `false` 的边就是形成环的边。路径压缩配合按大小合并后，单次操作均摊为 `O(α(n))`，实际近似常数。Map 版本直接支持字符串或稀疏节点。

### 关键考点

- `union` 前根相同意味着边两端已连通。
- 两种优化必须说明各自作用。
- Map 避免依赖连续整数编号。

### 常见追问或变体

## 10. Next.js 16 Cache Components 与 `use cache`

### 参考答案

启用 Cache Components 后，静态 JSX 和可在预渲染阶段完成的缓存内容可进入静态壳；未缓存的数据访问在请求时执行，并用 Suspense 提供壳中的 fallback。公共商品查询可放在 `'use cache'` 函数内，按商品 ID 建键，设置 `cacheLife` 和商品 tag；写操作后用 `updateTag` 或 `revalidateTag` 失效。

cookie 等请求数据应在缓存作用域外读取。个人价格通常保持动态；若把 session ID 传给缓存函数，它会成为键的一部分，但会产生高基数、隐私、生命周期和跨用户错误复用风险。除非有明确私有缓存策略，不应把用户数据放入共享缓存。

### 关键考点

- 静态、缓存、动态三类内容可以交错组合。
- 闭包值和可序列化参数会参与缓存键。
- Suspense 是动态边界，cache tag 是业务失效边界。

### 常见追问或变体

## 11. Node.js `diagnostics_channel` 的低侵入观测

import { channel } from 'node:diagnostics_channel';
const started = channel('acme.http.request.start');

## 12. NestJS 请求作用域 Provider 的成本与替代方案

### 参考答案

singleton 在应用生命周期内复用；request scope 每个请求创建一次；transient 每个注入消费者获得新实例。singleton 依赖 request-scoped provider 时会被作用域冒泡，整条依赖链按请求实例化，增加对象创建、GC 和依赖解析成本。

无状态服务保持 singleton。用户或租户 ID 可作为方法参数传递，或在明确治理下由 AsyncLocalStorage 保存请求上下文；不要把可变请求数据写入 singleton 字段。验证应包含并发请求隔离测试、DI 图审查、吞吐/延迟和堆分配对比。

### 关键考点

- scope 会沿依赖图传播。
- 请求上下文便利性不能牺牲隔离和吞吐。
- transient 不等于 request scope。

### 常见追问或变体

## 13. 复合索引、最左前缀与回表成本

### 参考答案

候选索引通常从等值过滤开始，再接排序/范围列，例如 `(tenant_id, status, created_at DESC)`。它能缩小租户和状态范围，并在相应数据库支持下按索引顺序扫描。范围列之后的索引列通常难以继续用于定位；把返回列纳入覆盖索引可减少回表，但会扩大索引。

索引会增加存储、写放大、缓存压力和优化器选择成本，不能按查询字段机械堆叠。应使用 `EXPLAIN ANALYZE` 检查估算与实际行数、扫描方式、排序、回表数量，并用生产近似的数据分布和分页深度测量。

### 关键考点

- 等值、范围、排序顺序共同决定索引设计。
- 覆盖收益需要与索引宽度权衡。
- 基数和执行计划比口号更可靠。

### 常见追问或变体

- 状态字段低基数时为何仍可能值得进入复合索引？

## 14. AI 多模态输入的前端上传与降级设计

### 参考答案

每个附件应有独立状态：local-preview、uploading、uploaded、processing、ready、failed/cancelled；消息整体只有在所需附件 ready 后才能提交。上传使用稳定附件 ID、幂等键和 AbortController，重试复用业务 ID，不能重复创建模型任务。

前端先限制数量、大小和声明类型，服务端仍需校验魔数、恶意内容和权限。原文件使用短期私有存储及过期清理；OCR/解析失败应定位到具体附件。模型不支持 PDF 时，可经用户知情后抽取文本，或明确阻止提交，不能静默丢弃内容。

### 关键考点

- 附件生命周期与消息生命周期分离。
- 客户端校验不能替代服务端安全检查。
- 降级必须可见、可取消、可解释。

### 常见追问或变体

## 15. Agent 追踪数据的隐私、脱敏与可回放性

### 参考答案

默认 trace 只记录模型、参数、耗时、token、状态码、工具名、schema 版本和内容长度，不记录 prompt/response 正文。需要正文的调试会话应显式授权、短期保存、强访问控制，并在采集入口按字段与模式脱敏；日志系统不能接收原始密钥后再补救。

采用按错误和风险分层的采样、租户隔离、最短保留期、访问审计及可验证删除。无正文回放可保存模型与 prompt 模板版本、工具 schema、随机性参数、输入规范化后的不可逆哈希、事件顺序和脱敏结构摘要。它能复现控制流和比较评测，但不能承诺逐 token 完全重现。

### 关键考点

- 可观测性数据本身属于高风险数据资产。
- 脱敏应发生在持久化之前。
- 可回放性必须声明能力边界。

### 常见追问或变体

- 哈希为什么不能直接保护低熵个人信息？
- 如何处理用户提出的数据删除请求？
