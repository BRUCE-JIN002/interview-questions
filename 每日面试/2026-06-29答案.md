## 1. 浏览器任务调度：微任务、渲染与 `scheduler.postTask`

### 参考答案

浏览器通常执行一个任务，清空微任务队列，再获得渲染机会；具体是否渲染由浏览器决定。大量微任务会持续占用主线程，推迟绘制，因此不应使用递归 `queueMicrotask` 拆分长计算。

- `queueMicrotask`：当前任务后的短小一致性工作。
- `requestAnimationFrame`：下一次绘制前更新布局或动画状态。
- `requestIdleCallback`：低优先级、可延迟且需自行检查剩余时间的工作。
- `scheduler.postTask`：有明确优先级的可调度任务；仍应做能力检测和降级。

搜索计算应切成有时间预算的小片，输入事件提升优先级，旧查询通过 `AbortController` 取消。每片结束后让出主线程，并避免低优先级任务无限延期。

### 关键考点

- 微任务过多同样会阻塞渲染。
- 调度 API 的选择取决于截止时间和优先级。
- 长任务必须可拆分、可取消、可降级。

- 如何用 Long Tasks API 验证优化？
- INP 与任务切片有什么关系？

## 2. TypeScript 模板字面量类型解析路由参数

### 参考答案

可以逐段匹配 `/:参数名`，将参数名映射为属性，再与剩余部分递归结果合并。

```ts
type RouteParams<Path extends string> =
  Path extends `${string}:${infer Param}/${infer Rest}`
    ? { [K in Param | keyof RouteParams<Rest>]: string }
    : Path extends `${string}:${infer Param}`
      ? { [K in Param]: string }
      : {};

type P = RouteParams<'/users/:id/posts/:postId'>;
// { id: string; postId: string }
```

终止条件是路径中不再包含参数，此时返回 `{}`。类型系统适合为固定路由表提供开发期约束，但不应承担完整 URL 语法解析；可选参数、通配符和编码规则复杂时，运行时路由库应是事实来源。

### 关键考点

- 模板字面量类型可以分解字符串。
- 递归类型必须有明确终止分支。
- 编译期类型不能替代运行时校验。

### 常见追问或变体

- 如何支持 `:id?` 可选参数？
- 重复参数名如何处理？
- `{}` 类型为什么不表示“没有任何属性”？

## 3. `structuredClone`、JSON 深拷贝与可转移对象

### 参考答案

JSON 方案会丢失 `undefined`、`BigInt`、`Map`、`Set`、循环引用和对象原型，并改变部分特殊值。`structuredClone` 支持循环引用、`Map`、`Set`、日期、类型化数组等结构化克隆类型，但不能克隆函数、DOM 节点、WeakMap 等，且不会保留普通对象的自定义原型语义。

手写递归实现容易遗漏属性描述符、Symbol、循环引用和内建类型，业务中优先选择明确的数据模型或平台 API。

```js
const buffer = new ArrayBuffer(1024);
const copy = structuredClone(buffer, { transfer: [buffer] });
console.log(buffer.byteLength); // 0，原缓冲区已 detached
console.log(copy.byteLength);   // 1024
```

transfer 转移底层资源所有权，避免大型数据复制，适合主线程与 Worker 间传输。

### 关键考点

- “深拷贝”必须先定义需要支持的数据类型。
- transfer 是所有权转移，不是复制。
- 平台 API 也有不可克隆类型。

### 常见追问或变体

- 结构化克隆是否保留 getter？
- 什么时候浅拷贝反而是正确选择？

## 4. 手写支持优先级与取消的任务调度器

### 参考答案

等待队列按优先级存放任务。启动任务前增加运行计数，在 `finally` 中减少计数并继续调度，确保成功、失败都释放槽位。取消只处理尚未启动的任务；已运行任务是否能取消取决于任务是否消费同一个 signal。

```ts
type Job<T> = {
  run: () => Promise<T>;
  resolve: (value: T) => void;
  reject: (reason?: unknown) => void;
  signal?: AbortSignal;
};

class Scheduler {
  private running = 0;
  private high: Job<unknown>[] = [];
  private low: Job<unknown>[] = [];

  constructor(private readonly limit = 2) {}

  add<T>(
    run: () => Promise<T>,
    priority: 'high' | 'low' = 'low',
    signal?: AbortSignal,
  ): Promise<T> {
    return new Promise<T>((resolve, reject) => {
      if (signal?.aborted) return reject(signal.reason);
      const job: Job<T> = { run, resolve, reject, signal };
      (priority === 'high' ? this.high : this.low).push(job as Job<unknown>);
      this.drain();
    });
  }

  private drain(): void {
    while (this.running < this.limit) {
      const job = this.high.shift() ?? this.low.shift();
      if (!job) return;
      if (job.signal?.aborted) {
        job.reject(job.signal.reason);
        continue;
      }
      this.running++;
      Promise.resolve()
        .then(job.run)
        .then(job.resolve, job.reject)
        .finally(() => {
          this.running--;
          this.drain();
        });
    }
  }
}
```

该实现优先处理高优任务，可能使低优任务饥饿。只有需求明确时才增加 aging 或高低队列配额，不应默认承诺严格公平。

### 关键考点

- 并发槽位必须在所有完成路径释放。
- 取消等待任务与中止运行任务是两件事。
- 优先级、公平性和吞吐量存在取舍。

### 常见追问或变体

- 如何避免低优先级任务饥饿？

## 5. EventSource 流式响应的重连与一致性

### 参考答案

服务端为每个任务维护单调递增事件 ID，并发送 `id:` 字段。客户端保存最后成功处理的 ID；重连时原生 EventSource 会发送 `Last-Event-ID`，服务端从其后一条重放。客户端仍需按 `(taskId, eventId)` 去重，因为网络边界无法保证恰好一次。

服务端设置有限事件保留窗口。若客户端位置已过期，返回显式的 `reset` 事件，让客户端查询任务快照后继续订阅。心跳用于穿透代理超时；完成态必须可重复查询，重连后也能确认任务已经结束。

原生 EventSource 重连方便，但自定义请求头、POST body 和细粒度错误处理受限；`fetch` 流更灵活，需自行实现解析、重连和断点恢复。

### 关键考点

- SSE 通常实现“至少一次”，客户端必须幂等。
- 增量事件之外还要有快照恢复接口。
- 鉴权方式会影响 EventSource 与 fetch 的选择。

### 常见追问或变体

- SSE 与 WebSocket 如何选择？
- 代理缓冲导致消息延迟如何排查？
- token 过期后如何安全续订？

## 6. React Activity 与“隐藏但保留状态”的界面设计

### 参考答案

卸载会销毁组件状态和副作用，重新进入需要重建。CSS 隐藏仍保留组件、DOM 和正在运行的副作用，高频订阅或定时器可能继续消耗资源。Activity 边界适合在隐藏界面时保留状态，同时让 React 对隐藏内容降低工作优先级并处理副作用生命周期。

实现时仍要让网络请求、媒体播放和外部订阅具备明确的暂停语义，不能假设边界能自动理解所有业务资源。上线前比较标签切换耗时、隐藏页面 CPU、内存、订阅数量、表单恢复正确率和重新显示后的数据新鲜度。

### 关键考点

- 状态保留不等于副作用持续运行。
- CSS 可见性不是组件生命周期。
- 新能力必须通过性能与正确性指标验证。

## 7. Vue SSR Hydration 不匹配的定位与修复

### 参考答案

Hydration 要求客户端首次渲染结果与服务端 HTML 一致。当前时间、随机数和浏览器专属数据在两端取值不同，会破坏这一前提。

- 时间由服务端生成并作为初始数据传入，或挂载后再显示客户端时间。
- 随机 ID 使用稳定 ID、服务端下发 seed，或仅在客户端挂载后生成。
- 浏览器宽度不要直接决定首屏 DOM，可用 CSS 响应式；确需 JS 时提供一致默认值后再更新。
- localStorage 数据在 `onMounted` 中读取，首屏使用服务端可确定的占位状态。

应先缩小到具体组件并比对服务端 HTML 与客户端首帧。压制警告会掩盖事件绑定、DOM 复用和闪烁问题，不能恢复一致性。

### 关键考点

- SSR 首次输出必须具有确定性。
- 浏览器 API 应延迟到客户端生命周期。
- 修复目标是消除差异，而不是隐藏警告。

### 常见追问或变体

- 时区差异如何处理？
- 第三方组件不支持 SSR 怎么办？
- hydration 与纯客户端渲染有何取舍？

## 8. 虚拟列表处理动态高度与滚动锚定

### 参考答案

为每项维护已测高度，未知项使用估算值；用前缀和结构记录累计偏移，可用 Fenwick Tree 支持高度更新和按偏移二分查找。元素渲染后由 `ResizeObserver` 更新真实高度，若变化发生在锚点上方，同时修正 `scrollTop`，保持锚点消息在视口中的相对位置。

向上加载前记录第一条可见消息的 ID 和元素顶部位置，插入历史数据并完成测量后，将滚动位置补偿为插入前的位置。图片应预留尺寸，overscan 根据滚动速度设置。缓存键使用稳定消息 ID，内容或宽度变化时失效。

普通数组前缀和重算最坏为 `O(n)`；Fenwick Tree 单次更新和前缀查询为 `O(log n)`，可见项查找也可做到 `O(log n)`。

### 关键考点

- 动态高度需要测量、估算和失效机制。
- 滚动锚定依赖稳定 ID 和位置补偿。
- 数据结构决定高度更新成本。

### 常见追问或变体

- 用户停留底部时新消息如何自动跟随？
- ResizeObserver 回调如何避免抖动？
- 无障碍阅读顺序是否受虚拟化影响？

## 9. 算法：课程依赖与拓扑排序

### 参考答案

使用 Kahn 算法。建立从前置课程指向后续课程的邻接表和入度数组，将所有入度为 0 的节点入队；每取出一个节点，就降低其后继节点入度。最终访问数量小于 `n` 说明剩余节点处于环中。

```ts
function findOrder(n: number, prerequisites: number[][]): number[] {
  const graph = Array.from({ length: n }, () => [] as number[]);
  const indegree = Array(n).fill(0);

  for (const [course, before] of prerequisites) {
    graph[before].push(course);
    indegree[course]++;
  }

  const queue: number[] = [];
  for (let i = 0; i < n; i++) {
    if (indegree[i] === 0) queue.push(i);
  }

  const order: number[] = [];
  for (let head = 0; head < queue.length; head++) {
    const current = queue[head];
    order.push(current);
    for (const next of graph[current]) {
      if (--indegree[next] === 0) queue.push(next);
    }
  }

  return order.length === n ? order : [];
}
```

时间复杂度 `O(V + E)`，空间复杂度 `O(V + E)`。

### 关键考点

- 入度为 0 表示当前没有未满足依赖。
- 无法处理完全部节点即存在环。
- 数组配合 head 指针避免 `shift()` 的移动成本。

## 10. Next.js `instrumentation` 与服务端可观测性

### 参考答案

在 instrumentation 注册阶段初始化 OpenTelemetry SDK、导出器和资源属性，并保证初始化只执行一次。入口 span 从传入的 W3C `traceparent` 恢复上下文；调用下游 HTTP 服务时注入上下文，使 Route Handler、服务端渲染和下游请求串联。

只记录路由模板、状态码、耗时、缓存命中等低基数字段，不记录 token、Cookie、完整 URL 参数或用户正文。错误请求可提高采样率，正常流量采用概率或尾部采样。

Node.js Runtime 能使用较完整的 Node SDK 和原生依赖；Edge Runtime 的 API、包兼容性和执行生命周期受限，需要选择兼容的轻量实现，不能假设 Node instrumentation 可直接复用。

### 关键考点

- trace context 必须跨进程传播。
- 遥测属性要控制敏感信息和基数。
- Runtime 差异影响 SDK 与导出方式。

### 常见追问或变体

## 11. Node.js `worker_threads` 的任务池设计

### 参考答案

Promise 只改变结果组织方式，不会把 CPU 计算移出主线程。同步图片哈希或大型 JSON 转换仍会长期占用事件循环。

任务池预先创建接近 CPU 核数的 Worker，每个 Worker 同时执行一个任务；其余任务进入有界队列。任务包含 ID、截止时间和最小输入。超时后终止或废弃结果，并补充 Worker；Worker 异常退出时拒绝当前任务并按策略重建。大型二进制数据使用 transfer，避免复制。

队列达到上限时应拒绝、降级或返回稍后重试，不能无限堆积。还要监控队列长度、排队时间、执行时间、Worker 重启次数和进程内存。

### 关键考点

- Promise 不提供 CPU 并行。
- Worker 池避免频繁创建线程。
- 有界队列是过载保护的核心。

### 常见追问或变体

- Worker 数量为什么不应越多越好？
- 子进程和 worker_threads 如何选择？
- 如何处理任务取消？

## 12. NestJS CQRS 与事务一致性边界

### 参考答案

Command 表达修改意图，由 Command Handler 执行业务规则并写入模型；Query 只读取并返回视图；Event 表示已经发生的事实，供读模型或外部消费者响应。

不能在数据库提交前直接发布事件，也不能在提交后无保护地发布，否则进程崩溃会丢事件。常用 Transactional Outbox：在同一数据库事务中写订单和 outbox 记录，后台发布器可靠投递，消费者按事件 ID 幂等处理，读模型接受最终一致。

简单 CRUD、单体小团队、读写模型没有明显差异时，CQRS 会增加类型、处理器、追踪和一致性成本，不值得引入。

### 关键考点

- CQRS 分离的是职责和模型，不一定分离数据库。
- Outbox 解决事务提交与事件发布的原子性缝隙。
- 消费端仍必须幂等。

### 常见追问或变体

- Event Sourcing 与 CQRS 有什么区别？
- 如何处理读模型延迟？
- NestJS EventBus 是否天然保证可靠投递？

## 13. OAuth 2.1 / OIDC 的前后端登录安全

### 参考答案

浏览器生成 `code_verifier`，授权请求发送其摘要 `code_challenge`；回调换 token 时提交 verifier，防止授权码被截获后滥用。`state` 绑定当前登录发起状态并防 CSRF；OIDC 的 `nonce` 绑定 ID Token，防止重放和错配。

BFF 在服务端完成换 token并保存 token，浏览器只持有不透明会话 Cookie。Cookie 应设置 `HttpOnly`、`Secure` 和合适的 `SameSite`；状态修改请求再配合 CSRF token 或严格 Origin 校验。刷新 token 要轮换并检测重用，会话撤销后清除服务端凭据。

前端日志、URL、localStorage 和可读 Cookie 中都不应出现 access token。

### 关键考点

- OAuth 负责授权，OIDC 增加身份层。
- PKCE、state、nonce 防护目标不同。
- BFF 模式缩小 token 暴露面，但仍需 CSRF 防护。

### 常见追问或变体

- SameSite=Lax 是否足够？
- 多实例 BFF 如何保存会话？
- ID Token 能否当作 API access token？

## 14. MCP Sampling 的授权与安全边界

### 参考答案

Sampling 是 Server 向 Client 请求模型生成，不代表 Server 获得宿主模型和数据的隐式授权。宿主应展示 Server 身份、请求目的、完整模型输入摘要、将暴露的数据、期望输出用途及模型费用。涉及敏感数据、外部副作用、权限提升或新增工具调用时必须逐次确认。

宿主重新应用自己的系统策略、数据访问控制、工具 allowlist 和输出过滤；Server 提供的提示只作为不可信内容，不能覆盖宿主指令。Sampling 请求默认不应继承用户在其他 Server 或会话中的上下文。记录请求方、授权决定、输入数据类别、模型和结果去向，敏感正文按策略脱敏。

### 关键考点

- Sampling 是能力请求，不是权限委托。
- Server 内容必须按不可信输入处理。
- 数据范围、工具范围和结果去向都要授权。

### 常见追问或变体

- 如何设计“本次允许”和“始终允许”？

## 15. Agent 端到端评测与回归门禁

### 参考答案

离线集应覆盖正常问答、歧义澄清、无权限请求、检索无结果、工具失败和高风险操作，并保存期望结果及允许的工具轨迹范围。结果级指标包括事实正确性、任务完成率、引用准确率和安全违规率；轨迹级指标包括工具选择、参数正确率、无效调用数、重试次数、延迟和成本。

模型评审器需要用人工标注样本校准，持续检查一致率，关键安全项不能只依赖单一 LLM judge。上线采用固定回归集硬门禁，加小流量灰度和线上抽样；高风险错误零容忍，其余指标与基线比较并设置置信区间。

通过分层回放定位问题：固定检索结果测试模型与工具决策；固定模型轨迹测试接口；检查召回集定位检索；验证 JSON Schema、描述和示例定位工具定义。

### 关键考点

- Agent 评测必须同时看最终结果和执行轨迹。
- 自动评审器需要人工校准。
- 可复现的分层回放才能定位责任层。

### 常见追问或变体

- 非确定性输出如何做稳定回归？
- 如何评估“应该先澄清”？
- 线上用户反馈如何回流到离线集？
