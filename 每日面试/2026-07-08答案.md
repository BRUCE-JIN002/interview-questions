# 2026-07-08 每日面试题参考答案

## 1. JavaScript Temporal 与日期时间可靠性

### 参考答案

`Date` 的主要问题是同时承担时间点、日历日期和本地展示三类语义，解析字符串时还容易受运行时和时区影响；对象可变，月份从 0 开始，夏令时切换日可能出现不存在或重复的本地时间。跨时区会议应把“绝对发生时间”和“用户输入的日历语义”分开：服务端存 UTC instant、会议所在时区、原始本地日期时间和必要的日历规则；前端展示时用用户当前时区或会议时区格式化。

Temporal 把 `Instant`、`PlainDateTime`、`ZonedDateTime` 等语义拆开，能减少隐式转换。落地时要关注运行环境支持度、polyfill 体积、和后端时间格式约定。

### 关键考点

- 时间点、时区、本地日期时间是不同概念。
- 服务端持久化应避免只存本地时间字符串。
- 夏令时和跨时区展示必须显式携带 IANA timezone。

### 常见追问或变体

- 为什么只存时间戳不能完整表达“每周一 10:00”的日程？
- 前端如何处理用户修改系统时区后的展示变化？

## 2. TypeScript Instantiation Expressions 与泛型复用

### 参考答案

Instantiation expression 允许对泛型函数或构造器直接固定类型参数，例如 `const makeStringBox = makeBox<string>`，得到一个更具体的函数类型，而不用额外写包装函数。它保留原实现，只在类型层面裁剪不匹配的签名。

相比显式类型注解，它更接近“从已有泛型函数派生一个专用版本”；相比包装器，它没有运行时额外调用层。API client 中可以把通用 `request<T>` 派生成 `request<UserDTO>` 风格的专用函数，也可以为事件总线固定事件 map。

### 关键考点

- 类型参数固定发生在类型层面，不改变运行时实现。
- 适合复用泛型工厂、构造器、请求函数。
- 不是所有重载签名都会保留，不匹配类型参数数量的签名会被过滤。

### 常见追问或变体

- 它和 `const fn: (x: string) => Box<string> = makeBox` 有什么不同？
- 泛型类构造器能否使用这种写法？

## 3. 手写带权重的 Promise 并发限制器

### 参考答案

```ts
type Task<T> = () => Promise<T> | T;

function createWeightedLimiter(maxWeight: number) {
  let used = 0;
  const queue: Array<{
    weight: number;
    task: Task<unknown>;
    resolve: (value: unknown) => void;
    reject: (reason?: unknown) => void;
  }> = [];

  function schedule() {
    for (let i = 0; i < queue.length; i++) {
      const item = queue[i];
      if (used + item.weight > maxWeight) continue;
      queue.splice(i, 1);
      i--;
      used += item.weight;

      Promise.resolve()
        .then(item.task)
        .then(item.resolve, item.reject)
        .finally(() => {
          used -= item.weight;
          schedule();
        });
    }
  }

  return function run<T>(weight: number, task: Task<T>): Promise<T> {
    if (weight <= 0 || weight > maxWeight) {
      return Promise.reject(new Error("invalid weight"));
    }
    return new Promise<T>((resolve, reject) => {
      queue.push({ weight, task, resolve: resolve as (v: unknown) => void, reject });
      schedule();
    });
  };
}
```

失败任务要在 `finally` 释放权重。上面实现会扫描后续可运行的小任务，吞吐较好，但大权重任务可能被小任务持续插队；如果强调公平，可以只看队首或加入等待时长优先级。取消可通过 `AbortSignal` 标记队列项，尚未启动的任务直接拒绝，已启动的任务把 signal 传给 task。

### 关键考点

- 并发控制的单位不只是任务数，也可以是资源权重。
- `finally` 释放资源，避免失败导致死锁。
- 公平性和吞吐量通常存在取舍。

### 常见追问或变体

- 如何支持优先级？
- 如何避免一个超大任务长期饥饿？

## 4. Navigation API 与 SPA 路由拦截

### 参考答案

传统 History API 只能手动维护 push、replace、popstate，无法统一拦截所有导航，也难以表达一次导航的生命周期。Navigation API 提供当前 entry、导航事件、拦截和 transition 等更直接的模型。

SPA 可把拦截链路拆成：权限检查、离开确认、数据预取、加载态、真正切换、错误回滚、埋点上报。离开确认需要区分用户手势和程序跳转；加载态应绑定本次 navigation，过期请求不能污染新页面；错误时保留旧页面或进入错误边界。兼容性不足时降级为 History API + 路由库导航守卫。

### 关键考点

- 路由切换是一个有生命周期的异步事务。
- 需要处理竞态、回滚、取消和埋点一致性。
- 新 API 落地必须考虑浏览器兼容和框架集成。

### 常见追问或变体

- 页面刷新和 SPA 内部跳转的确认逻辑有什么不同？
- 埋点应该记录发起、成功还是失败？

## 5. WebSocket 背压与消息堆积治理

### 参考答案

WebSocket 没有像 Stream 那样完整的内建背压模型。前端处理过慢时，浏览器和应用层队列可能持续增长，导致内存升高、UI 卡顿、消息过期。治理思路是先分类消息：状态快照类可以合并或只保留最新值；事件流水类必须按序消费；低优先级消息可以丢弃。

客户端应监控队列长度、处理耗时、`bufferedAmount` 和渲染帧耗时。服务端可支持订阅粒度、降频、快照补偿和 ack 游标。重连后用 last sequence 拉取缺失消息，不能只依赖“断了重连”。

### 关键考点

- 背压需要客户端和服务端协商。
- 消息语义决定能否丢弃、合并或重排。
- 实时系统要有序列号、快照和补偿机制。

### 常见追问或变体

- 行情数据和聊天消息的丢弃策略有什么不同？
- 如何定位消息堆积是网络、JS 计算还是渲染导致？

## 6. React Server Actions 的渐进增强与失败恢复

### 参考答案

表单 action 能让无 JS 或 JS 未加载完成时仍通过原生表单提交到服务端，这是渐进增强的核心。客户端增强后可以结合 pending 状态、乐观更新和错误展示。

设计状态流转时，提交前生成幂等 key；提交中禁用关键按钮但不阻塞全页；服务端校验失败返回字段级错误；成功后以服务端结果为准刷新相关数据；乐观更新失败时回滚或标记冲突。重复提交靠幂等 key 和服务端状态机兜底，不能只靠前端按钮禁用。

### 关键考点

- Server Action 不等于跳过 API 安全校验。
- 渐进增强要求原生表单路径也可用。
- 乐观 UI 必须能被服务端真相纠正。

### 常见追问或变体

- 文件上传适合放在 Server Action 吗？
- 如何防止用户篡改隐藏字段？

## 7. Vue Vapor Mode 思路与编译优化边界

### 参考答案

Vapor 方向的核心是让编译器更精确地把模板映射为细粒度 DOM 更新，减少虚拟 DOM diff 的运行时成本。它适合依赖关系清晰、模板结构相对静态、更新热点明确的场景。

边界在于动态性越强，编译期越难证明更新范围；复杂动态插槽、手写 render function、运行时组件组合和高度动态 schema 可能需要回退到通用路径。对业务开发来说，不应为了追逐编译优化牺牲组件边界和可维护性。

### 关键考点

- 编译期优化依赖静态可分析性。
- 虚拟 DOM 的价值是统一抽象和动态表达能力。
- 性能优化要看真实瓶颈，不是框架模式越新越好。

### 常见追问或变体

- React Compiler 和 Vue Vapor 的优化方向有什么相似和不同？
- 为什么动态插槽会增加优化难度？

## 8. 前端安全下载与文件预览

### 参考答案

下载链路不能只看文件后缀。服务端应校验权限、记录审计、设置准确 `Content-Type` 和 `Content-Disposition`，对用户上传内容优先走附件下载或隔离域预览，避免脚本内容在主站上下文执行。浏览器端创建对象 URL 后要在不再使用时 `URL.revokeObjectURL`。

在线预览要按类型隔离：PDF 可用沙箱 iframe 或专门预览域；Office 文件可转码为安全格式；图片要限制大小和解码风险；压缩包通常不直接预览。跨域 Cookie 和下载鉴权要避免把长期 token 暴露在 URL 中。

### 关键考点

- MIME、后缀、内容嗅探不能互相信任。
- 用户上传内容应尽量隔离域名或沙箱执行。
- 下载和预览同样需要权限、审计和过期控制。

### 常见追问或变体

- 为什么 `X-Content-Type-Options: nosniff` 有价值？
- 对象 URL 泄漏会带来什么问题？

## 9. 算法：反转 K 个一组链表

### 参考答案

```ts
class ListNode {
  val: number;
  next: ListNode | null;
  constructor(val = 0, next: ListNode | null = null) {
    this.val = val;
    this.next = next;
  }
}

function reverseKGroup(head: ListNode | null, k: number): ListNode | null {
  const dummy = new ListNode(0, head);
  let groupPrev: ListNode = dummy;

  while (true) {
    let kth: ListNode | null = groupPrev;
    for (let i = 0; i < k && kth; i++) kth = kth.next;
    if (!kth) break;

    const groupNext = kth.next;
    let prev: ListNode | null = groupNext;
    let cur: ListNode | null = groupPrev.next;

    while (cur !== groupNext) {
      const next = cur!.next;
      cur!.next = prev;
      prev = cur;
      cur = next;
    }

    const oldHead = groupPrev.next!;
    groupPrev.next = kth;
    groupPrev = oldHead;
  }

  return dummy.next;
}
```

每组先找到第 k 个节点，不足 k 个直接结束。反转时把 `prev` 初始化为 `groupNext`，这样组尾能自然接回后续链表。时间复杂度 `O(n)`，额外空间 `O(1)`。

### 关键考点

- dummy 节点简化头节点被反转的情况。
- 先确认足够 k 个节点，再反转。
- 保存 `groupNext` 和旧组头，避免断链。

### 常见追问或变体

- 如果要求递归实现，空间复杂度是多少？
- k 等于 1 或链表为空时如何处理？

## 10. Next.js Draft Mode 与预览发布链路

### 参考答案

Draft Mode 通常通过服务端设置预览 Cookie，让后续请求进入草稿渲染路径，绕开或改变静态缓存读取方式，从 CMS 拉取未发布内容。它适合内容编辑预览，但本质上是一个带权限的临时访问能力。

安全链路应包括：预览入口校验编辑权限和一次性 token；服务端设置短期预览 Cookie；页面渲染时只在 draft 状态下读取草稿；预览页显示明显状态；退出预览时清 Cookie；发布后触发缓存失效。预览链接泄露时应能过期或撤销，不能把 CMS 长期密钥暴露给浏览器。

### 关键考点

- Draft Mode 是服务端状态，不是简单 query 参数。
- 草稿预览要和缓存失效、权限校验结合。
- 预览 token 应短期、可撤销、服务端校验。

### 常见追问或变体

- 为什么预览接口不能只校验 slug？
- 发布后如何避免用户仍看到旧缓存？

## 11. Node.js 子进程隔离与任务执行安全

### 参考答案

CPU 密集或不可信任务不能在主事件循环同步执行，否则会阻塞所有请求。`worker_threads` 适合同进程内 CPU 计算，共享内存能力更强但隔离较弱；`child_process` 适合调用外部工具，进程隔离更清晰；任务队列适合削峰、重试和异步结果；独立服务适合资源要求高或安全边界强的能力。

安全设计包括：白名单命令和参数，不拼接 shell 字符串；设置超时和最大输出；限制输入文件大小；任务 ID 关联日志；失败结果结构化；必要时用容器或沙箱限制 CPU、内存和文件系统访问。

### 关键考点

- 主线程不能承载长时间 CPU 或阻塞 IO。
- `exec`、`spawn`、worker、队列的边界不同。
- 用户输入进入命令执行前必须白名单化。

### 常见追问或变体

- 为什么 `exec("convert " + filename)` 危险？
- 如何处理子进程产生大量 stdout 导致内存上涨？

## 12. NestJS 文件上传接口的可靠性设计

### 参考答案

上传接口应先由 `Guard` 做身份和业务权限判断，再由上传中间件或 `Interceptor` 接入 multipart 处理，由 `Pipe` 或专门服务做文件元数据、大小、类型、checksum 校验。大文件更适合直传对象存储或分片上传，应用服务负责签名、合并、状态机和回调校验。

数据库中维护 `uploading`、`uploaded`、`verified`、`failed` 等状态。对象存储回调和数据库更新要做到幂等；分片合并失败可重试；最终业务使用文件前必须确认状态已完成，不能只相信前端返回的 URL。

### 关键考点

- 鉴权、解析、校验、状态更新应分层。
- 大文件上传不宜全部进入应用内存。
- 对象存储回调和数据库状态需要幂等。

### 常见追问或变体

- 文件类型校验为什么不能只看扩展名？
- 分片上传如何清理长期未完成的临时对象？

## 13. OpenAPI 契约治理与前后端协作

### 参考答案

要避免文档和实现不一致，OpenAPI 必须进入研发流程，而不是发布后补文档。推荐流程是：接口设计先评审 schema；前端基于 schema 生成类型和 mock；后端实现时用 schema 校验请求响应；CI 做 breaking change 检查；契约测试覆盖关键接口；发布时同步版本和 changelog。

兼容性规则要明确：新增可选字段通常兼容；删除字段、修改字段类型、把可选改必填属于破坏性变更。前端生成代码应锁定版本，避免后端未发布 schema 直接影响生产构建。

### 关键考点

- 契约要前置到设计和 CI。
- schema、mock、类型生成、测试应来自同一源。
- 兼容性检查比口头约定可靠。

### 常见追问或变体

- OpenAPI 和运行时 zod schema 如何取舍？
- 如何处理灰度期间前后端版本不一致？

## 14. AI 前端可中断流式响应

### 参考答案

点击停止后，前端应立即把会话状态从 streaming 转为 cancelling，调用 `AbortController.abort()` 或取消请求，并保留已接收内容。BFF 要把取消信号传递给模型请求和工具执行层，停止继续写 SSE；模型服务若支持取消则释放推理资源，不支持时也要丢弃后续 token。

状态机可包含 `idle -> streaming -> cancelling -> cancelled | completed | failed`。已经渲染的 Markdown 要保证增量解析安全，取消后可标记“已停止”。工具调用如果有副作用，必须记录是否已执行和执行结果；账单按服务端实际完成 token 或调用记录结算，不能由前端估算。

### 关键考点

- 前端 abort 不等于服务端和模型一定停止。
- 流式 UI 需要明确状态机和半成品展示。
- 副作用工具调用要有幂等和审计。

### 常见追问或变体

- 停止生成后用户继续追问，历史上下文应包含半截回答吗？
- SSE 和 fetch stream 的取消差异是什么？

## 15. Agent 多工具规划与上下文压缩

### 参考答案

多工具 Agent 应把工具定义成明确 schema：能力范围、参数类型、权限级别、返回结构和错误码。执行时先规划最小步骤，再逐步调用工具，每次观察结果要摘要成和目标相关的事实，避免把完整日志无限塞回上下文。

上下文压缩可以分层：用户目标和约束常驻；关键决策和文件路径保留；大段输出变成摘要和引用；过期尝试移出主上下文。高风险操作如写文件、提交、发消息、删数据、外部支付、修改权限必须人工确认或进入沙箱。失败重试要区分参数错误、权限错误、网络错误和业务拒绝。

### 关键考点

- 工具 schema 是 Agent 安全和可观测的基础。
- 上下文压缩要保留决策依据，不是简单截断。
- 副作用工具必须有权限分级、确认和审计。

### 常见追问或变体

- 如何防止工具返回内容中的提示注入？
- Agent 什么时候应该停止自动重试并请求人工介入？
