## 1. JavaScript `WeakRef` 与 `FinalizationRegistry` 的适用边界

### 参考答案

`WeakRef` 不阻止目标被垃圾回收，但 GC 是否执行、何时执行由引擎决定；`deref()` 随时可能返回 `undefined`。`FinalizationRegistry` 的回调同样不保证及时执行，进程退出时甚至可能完全不执行。因此它适合做可丢失缓存或辅助性清理，不能承担关闭文件、释放锁、提交事务等正确性要求。

显式资源应通过 `dispose`、`try/finally`、组件卸载钩子等确定性释放。若闭包、普通 `Map`、事件监听器仍引用对象，对象仍是强可达的，弱引用不会帮助回收。

### 关键考点

- 弱引用不提供生命周期时序保证。
- `WeakMap` 适合将附加数据绑定到对象生命周期。
- 不应在 finalizer 中实现关键业务逻辑。

### 常见追问或变体

- `WeakMap` 为什么不能遍历？
- 如何定位事件监听器造成的内存泄漏？

## 2. TypeScript 品牌类型约束业务 ID

### 参考答案

品牌类型在结构类型上增加不可伪造的类型标记；构造边界负责运行时校验：

```ts
declare const userIdBrand: unique symbol;
declare const orderIdBrand: unique symbol;

type UserId = string & { readonly [userIdBrand]: 'UserId' };
type OrderId = string & { readonly [orderIdBrand]: 'OrderId' };

function parseUserId(value: unknown): UserId {
  if (typeof value !== 'string' || !/^usr_[a-z0-9]+$/.test(value)) {
    throw new Error('Invalid UserId');
  }
  return value as UserId;
}
```

品牌只在编译期存在，JSON 序列化后仍是字符串；从网络、存储读取后必须重新校验和构造。不要把 `as UserId` 散落在业务代码中。跨模块应从单一契约包导出同一个 `unique symbol` 品牌，否则可能产生互不兼容的类型。

### 关键考点

- 品牌类型弥补结构化类型对语义相同基础类型的区分。
- 类型安全依赖受控构造入口。
- 它不能替代运行时校验。

### 常见追问或变体

- 品牌类型与包装 class 各有什么取舍？
- 如何用 Schema 库同时生成校验器和类型？

## 3. 手写可提前终止的异步迭代器分页器

### 参考答案

```ts
type Page<T> = { items: T[]; nextCursor: string | null };

async function* paginate<T>(
  fetchPage: (cursor: string | null, signal: AbortSignal) => Promise<Page<T>>,
  signal: AbortSignal,
): AsyncGenerator<T> {
  let cursor: string | null = null;

  while (true) {
    signal.throwIfAborted();
    const page = await fetchPage(cursor, signal);
    for (const item of page.items) {
      signal.throwIfAborted();
      yield item;
    }
    if (page.nextCursor === null) return;
    cursor = page.nextCursor;
  }
}
```

异步生成器按需拉取页面，只保留当前页。`for await...of` 提前 `break` 会调用生成器的 `return()` 并进入其 `finally`；若生成器自行创建控制器，可在 `finally` 中取消在途工作。本题由外部传入 signal，调用方负责在不再需要结果时 `abort()`。请求错误会从迭代位置直接抛给消费者。

### 关键考点

- 异步迭代提供背压式按需消费。
- `break` 与网络取消不是同一件事。
- `finally` 用于释放生成器自己拥有的资源。

### 常见追问或变体

- 如何预取下一页但仍限制并发为 1？
- 如何避免服务端返回重复 cursor 导致死循环？

## 4. HTTP/2 多路复用仍然可能出现的队头阻塞

### 参考答案

HTTP/2 在一个 TCP 连接内交错传输多个 stream，消除了 HTTP/1.1 应用层按请求排队的问题。但 TCP 必须向上层交付连续有序字节；一个数据包丢失时，后续已到达的数据也要等待重传，因此同一连接上的所有 HTTP/2 stream 都可能停顿。

QUIC 在 UDP 上为各 stream 单独维护有序交付，某个 stream 丢包通常不阻塞其他 stream；但同一 stream 内仍需有序，服务端计算、连接拥塞、应用锁竞争等阻塞也不会消失。选型还需观察 UDP 可达性、握手与恢复耗时、丢包率、回退比例、CDN/代理支持和 CPU 成本。

### 关键考点

- 区分 HTTP 层与传输层队头阻塞。
- HTTP/3 不是“没有阻塞”。
- 协议收益必须通过真实网络数据验证。

### 常见追问或变体

- HTTP/2 为什么不再需要域名分片？
- QUIC 的连接迁移解决什么问题？

## 5. React Transition 与延迟值的取舍

### 参考答案

`startTransition` 把其同步回调内触发的 state 更新标记为非紧急；`useTransition` 额外返回 `isPending`。当无法控制产生更新的位置，或希望让某个派生值滞后时，可使用 `useDeferredValue(value)`。

输入框的受控 value 必须同步跟随键盘输入，否则会出现光标和回显异常。应同步更新 query，再把列表更新标记为 Transition，或让列表消费 deferredQuery：

```tsx
const [query, setQuery] = useState('');
const deferredQuery = useDeferredValue(query);
const results = useMemo(() => search(items, deferredQuery), [items, deferredQuery]);
const stale = query !== deferredQuery;
```

可用 `isPending` 或 stale 状态降低旧结果透明度。Transition 只改变渲染优先级，不减少请求次数；网络防抖仍需计时器、取消与竞态治理。

### 关键考点

- 紧急输入更新与可中断渲染更新分层。
- Transition 不等于延时执行或防抖。
- 优化前应确认瓶颈确实在渲染。

### 常见追问或变体

- Transition 中的异步更新有哪些注意事项？
- `useMemo` 为什么不能保证交互一定流畅？

## 6. Vue `provide` / `inject` 的响应式与封装边界

### 参考答案

普通值不会自动变成响应式；提供 `ref` 后，注入方拿到同一个 ref，Vue 不会自动解包它。推荐由提供方维护写操作，只向后代暴露只读状态和明确命令：

```ts
import type { InjectionKey, Ref } from 'vue';

type FormContext = {
  state: Readonly<Ref<FormState>>;
  updateField(name: string, value: unknown): void;
};

export const formKey: InjectionKey<FormContext> = Symbol('form');

provide(formKey, {
  state: readonly(state),
  updateField,
});
```

`readonly` 是运行时写保护提示，但深层边界仍取决于使用的响应式 API。SSR 中不能把用户状态放在模块级单例或全局 app provide 里复用；应为每个请求创建应用和状态容器。

### 关键考点

- 响应式状态与修改函数应共同定义契约。
- Symbol `InjectionKey` 避免键冲突并提供类型。
- SSR 必须隔离请求级状态。

### 常见追问或变体

## 7. 用 INP 定位交互延迟

### 参考答案

INP 观察一次交互从输入到下一次绘制的延迟，包含输入延迟、事件回调执行时间和呈现延迟。先用 RUM 按页面、设备和交互目标定位长尾，再在相近设备上用 Performance 面板录制：检查事件前是否有 Long Task，展开事件回调调用栈，最后检查样式计算、布局和绘制。

接口 80ms 并不能解释 500ms 的全部交互延迟；主线程可能正在执行大包初始化、同步过滤、第三方脚本或大范围布局。优化包括拆分长任务并让出主线程、把纯计算移入 Worker、减少同步 DOM 读写和布局抖动、虚拟化大列表、延迟非关键第三方脚本。

### 关键考点

- INP 测量交互响应，不等同于接口耗时。
- 实验室数据用于复现，RUM 用于确认真实影响。
- 应定位具体交互和具体长任务，而非只看平均值。

### 常见追问或变体

- `requestIdleCallback` 是否适合所有后台任务？

## 8. Form-associated Custom Elements 的表单契约

### 参考答案

```js
class DateRange extends HTMLElement {
  static formAssociated = true;
  internals = this.attachInternals();

  set value({ start, end }) {
    const data = new FormData();
    data.append(`${this.getAttribute('name')}.start`, start);
    data.append(`${this.getAttribute('name')}.end`, end);
    this.internals.setFormValue(data);

    const valid = Boolean(start && end && start <= end);
    this.internals.setValidity(
      valid ? {} : { customError: true },
      valid ? '' : '请选择有效日期范围',
    );
  }
}
```

`ElementInternals` 让自定义元素作为一个表单控件参与提交、约束校验、禁用和重置等生命周期，而不是依赖 Shadow DOM 内部字段被外部表单发现。组件还应关联可访问名称、实现 `focus()`、正确代理 label 点击，并设置合适的 role、状态与错误描述。

### 关键考点

- Shadow DOM 会形成封装边界。
- 表单值可为字符串、文件或 `FormData`。
- 原生提交能力不自动保证完整无障碍。

## 9. 算法：实现支持前缀搜索的 Trie

### 参考答案

```ts
class TrieNode {
  children = new Map<string, TrieNode>();
  end = false;
}

class Trie {
  private root = new TrieNode();

  insert(word: string) {
    let node = this.root;
    for (const char of word) {
      let next = node.children.get(char);
      if (!next) {
        next = new TrieNode();
        node.children.set(char, next);
      }
      node = next;
    }
    node.end = true;
  }

  private find(value: string) {
    let node: TrieNode | undefined = this.root;
    for (const char of value) node = node.children.get(char);
    return node;
  }

  search(word: string) {
    return this.find(word)?.end === true;
  }

  startsWith(prefix: string) {
    return this.find(prefix) !== undefined;
  }
}
```

每次操作的时间复杂度为 `O(L)`，空间为全部不同前缀节点数。空字符串插入会标记根节点，重复插入不增加节点。热词补全可在每个节点维护有界 top-K，插入或热度变化时沿路径更新。

### 关键考点

- `end` 区分完整单词与前缀。
- Map 适合字符集较大的情况。
- top-K 用空间换查询时间。

### 常见追问或变体

- 如何实现删除且回收无用节点？
- 中文联想应按字符还是分词存储？

## 10. Next.js Route Handler 的流式上传边界

### 参考答案

`request.formData()` 通常需要解析完整 multipart body，大文件会增加内存和临时存储压力。优先让服务端鉴权后签发短时、限制对象键和大小的上传凭证，由浏览器直传对象存储；需要服务端检查或隐藏存储端时才流式转发。超大文件和弱网使用分片上传，服务端记录 upload id 并支持重试。

Route Handler 应选择支持所需流和 SDK 的运行时，验证会话、Content-Length、实际读取字节数、MIME 魔数和对象键；失败或超时要终止流并清理未完成分片。上传进度来自浏览器上传通道，不应依赖服务端响应流。文件完成后再异步杀毒和转码，并在通过前保持不可公开。

### 关键考点

- 应避免把应用服务器变成不必要的数据中转站。
- 签名 URL 仍需最小权限和短有效期。
- 元数据声明不能替代内容检查。

### 常见追问或变体

- 如何防止用户覆盖其他用户的对象？
- Serverless 平台的请求体和执行时长限制如何影响方案？

## 11. Node.js 服务的优雅停机

### 参考答案

```ts
let shuttingDown = false;

process.once('SIGTERM', async () => {
  if (shuttingDown) return;
  shuttingDown = true;

  markNotReady(); // 先让负载均衡停止送入新流量
  server.close(); // 停止接受新连接

  const force = setTimeout(() => {
    server.closeAllConnections();
    process.exitCode = 1;
  }, 30_000);
  force.unref();

  try {
    await Promise.all([
      waitForInFlightRequests(),
      queue.stop(),
      db.end(),
      logger.flush(),
    ]);
    clearTimeout(force);
    process.exitCode = 0;
  } catch (error) {
    process.exitCode = 1;
  }
});
```

实践中需先切 readiness 并预留负载均衡传播时间；`server.close()` 等待现有连接，必要时配合关闭空闲连接和最终强制截止。直接 `process.exit()` 会立即终止事件循环，尚未刷出的日志、响应和异步清理可能丢失。

### 关键考点

- 停止接流量、排空请求、释放资源有明确顺序。
- 停机必须有截止时间。
- 后台消费者需停止拉取并完成或归还任务。

### 常见追问或变体

- WebSocket 长连接如何处理？
- Kubernetes 的 readiness 与 terminationGracePeriod 如何配合？

## 12. NestJS 模块边界与循环依赖治理

### 参考答案

大量 `forwardRef()` 表明模块职责和依赖方向不清。支付完成不必同步回调整个 `OrderService`：可由应用层协调“支付并确认订单”，或让支付模块发布领域事件，订单模块订阅并做幂等状态迁移。双方真正共享的能力可抽为更小的独立模块，但不要建立装杂物的 SharedModule。

`forwardRef()` 适合短期兼容无法立即拆分的双向关联；它只推迟依赖解析，没有消除运行时耦合和不确定的实例化顺序。测试除单个 provider 外，还应编译真实模块图，验证导入导出、启动钩子和事件处理的幂等性。

### 关键考点

- 先梳理业务依赖方向，再选择技术手段。
- 事件解耦会引入最终一致性和失败重试问题。
- 模块装配测试能发现单元测试遗漏的 DI 错误。

### 常见追问或变体

- `ModuleRef` 动态获取依赖有什么代价？
- 领域事件放在事务内还是事务后发布？

## 13. 数据库事务隔离级别与重试边界

### 参考答案

读已提交避免脏读，但同一事务重复查询可能看到新结果；可重复读保证已读行稳定，不同数据库对幻读的实现不同；串行化最接近串行执行，但冲突、等待和回滚成本最高。

优惠券领取的最终不变量应由数据库唯一约束如 `(user_id, coupon_id)` 保证，库存用带条件的原子更新。接口幂等键避免同一业务请求重复执行。死锁或串行化失败只对明确的可重试错误做有限次数、带抖动退避的整事务重试，事务内外部副作用需使用 outbox 等隔离。

提高隔离级别只能控制并发可见性，不能表达“每人每券一次”等业务唯一性，也不能修复缺少幂等的外部调用。

### 关键考点

- 隔离级别、唯一约束和幂等解决不同问题。
- 重试必须重放完整事务并设置上限。
- 缩短事务、固定锁顺序可减少死锁。

### 常见追问或变体

- 数据库 upsert 能否完全替代幂等键？
- 如何保留原始成功响应供重复请求返回？

## 14. 流式 Structured Outputs 的前端状态机

### 参考答案

前端至少区分 `receiving`、`complete-valid`、`complete-invalid`、`refused`、`failed`。接收期间保存原始事件序号和文本缓冲；可以用支持增量 JSON 的解析器生成草稿，但草稿字段必须标记为未确认，不能触发付款、预订等副作用。只有收到完成事件并对完整对象执行 Schema 校验后，才提升为可信业务状态。

断线恢复应携带响应 ID 或最后事件序号，并按事件 ID 去重；不支持续传时重新生成新版本，不能把两次输出拼接。Schema 失败保留原文和诊断并允许重试；拒绝是正常终态，不应当成解析错误。UI 应使用稳定 item id，并允许后续 patch 覆盖草稿，避免把中途合法片段当最终事实。

### 关键考点

- 传输完整、语法合法、Schema 合法是三层状态。
- 增量 UI 只能展示草稿，副作用必须等待最终确认。
- 重连需要明确的去重和版本协议。

### 常见追问或变体

- JSON Patch 与整对象快照各有什么取舍？
- 如何对超长数组做增量渲染和内存控制？

## 15. Agent 工具权限的最小授权与逐次提升

### 参考答案

任务开始时只挂载所需工具：代码审查只读仓库，不需要生产发布。命令执行限制工作目录、可执行程序、参数和资源；网络使用域名 allowlist；凭据使用短期、任务绑定、最小 scope 的令牌；执行环境采用隔离容器并默认无生产网络。

删除、支付、发送外部消息、写主分支、发布生产等不可逆或高影响操作应在展示具体目标和参数后逐次确认，策略引擎还要独立校验，而不是相信模型自报意图。审计记录包括用户授权、工具名、规范化参数、策略结果、输出摘要、主体、时间和关联 ID，并对敏感值脱敏。

策略必须评估组合能力，例如“写脚本 + CI 凭据 + 触发流水线”可能等价于直接发布；应在真正的权限边界和出口层阻断，而非只隐藏高风险工具名称。

### 关键考点

- 最小功能、最小权限、最小自治三个维度。
- 确认应绑定具体动作，不能一次授权永久放行。
- 审计日志不能包含明文密钥。

### 常见追问或变体

- 如何防止工具输出中的提示注入诱导提权？
- 人工审批超时或拒绝后 Agent 应如何恢复？
