## 1. Promise 与事件循环输出分析

参考答案：

输出顺序是：`A G I D H E B C F`。

同步阶段先执行 `A`，注册第一个 `setTimeout`，注册 Promise 微任务链，调用 `run()` 输出 `G`，`await null` 后面的 `H` 进入微任务，最后同步输出 `I`。第一轮微任务按入队顺序执行：先输出 `D`，`return Promise.resolve('E')` 会让后续 `.then` 等待该 Promise 解析；再执行 `await` 后续输出 `H`；然后执行第二个 `.then` 输出 `E` 并注册 `F` 的定时器。宏任务阶段先执行第一个定时器输出 `B`，其内部 Promise 微任务输出 `C`；下一轮宏任务输出 `F`。

关键考点：

- `async/await` 本质上会把 `await` 后续逻辑放入微任务。
- 每轮宏任务结束后会清空当前微任务队列。
- Promise 链中返回 Promise 会影响后续 `.then` 的入队时机。

常见追问或变体：

- 如果把 `await null` 改成 `await Promise.resolve()`，结果是否变化？
- Node.js 中 `process.nextTick` 和 Promise 微任务谁优先？

## 2. 手写 Promise.allSettled

参考答案：

```js
function allSettled(iterable) {
  return new Promise((resolve) => {
    const list = Array.from(iterable);
    const results = new Array(list.length);
    let count = 0;

    if (list.length === 0) {
      resolve([]);
      return;
    }

    list.forEach((item, index) => {
      Promise.resolve(item).then(
        (value) => {
          results[index] = { status: 'fulfilled', value };
        },
        (reason) => {
          results[index] = { status: 'rejected', reason };
        }
      ).finally(() => {
        count += 1;
        if (count === list.length) {
          resolve(results);
        }
      });
    });
  });
}
```

关键考点：

- 用 `Promise.resolve` 统一处理普通值、thenable 和 Promise。
- 每个任务单独捕获结果，不能因为一个 reject 导致整体 reject。
- 用下标写入结果保证顺序。

常见追问或变体：

- 如何实现 `Promise.all`、`Promise.any`、`Promise.race`？
- `finally` 回调抛错会怎样影响链路？

## 3. TypeScript 类型体操：DeepReadonly

参考答案：

```ts
type Primitive = string | number | boolean | bigint | symbol | null | undefined;

type DeepReadonly<T> =
  T extends Primitive ? T :
  T extends (...args: any[]) => any ? T :
  T extends readonly any[] ? { readonly [K in keyof T]: DeepReadonly<T[K]> } :
  T extends object ? { readonly [K in keyof T]: DeepReadonly<T[K]> } :
  T;
```

关键考点：

- 条件类型按分支处理基础类型、函数、数组/元组和对象。
- `readonly [K in keyof T]` 可以保留元组索引结构。
- 函数通常不递归处理，否则会破坏调用签名。

常见追问或变体：

- 如何实现 `DeepPartial`、`DeepRequired`？
- 如何避免递归类型过深导致编译性能问题？

## 4. 浏览器渲染与性能定位

参考答案：

定位先分层：Network 看 HTML、JS、CSS、图片是否阻塞首屏；Performance 看长任务、主线程火焰图、Layout/Paint；Lighthouse 或 Web Vitals 看 LCP、INP、CLS；Memory 看是否有泄漏或频繁 GC。

优化可以分三类：

- 加载性能：图片压缩和现代格式、首屏图预加载、懒加载非首屏图、JS 分包、减少首屏同步请求、CDN 缓存。
- 交互性能：拆分长任务、事件回调节流、防止重复提交、把重计算放到 Web Worker 或空闲时段。
- 渲染性能：避免频繁读写布局、减少大面积重排、虚拟列表、合理使用 `transform/opacity` 动画、给图片和广告位预留尺寸避免 CLS。

关键考点：

- 不要只说“压缩资源”，要能用指标定位。
- LCP 偏加载，INP 偏交互响应，CLS 偏布局稳定性。

常见追问或变体：

- 首屏图片是 LCP 元素时如何优化？
- React 大列表滚动卡顿怎么定位到组件层？

## 5. HTTP 缓存与协商缓存设计

参考答案：

`Cache-Control` 控制强缓存，例如 `max-age`、`immutable`、`no-cache`。强缓存命中时浏览器不请求服务器。`ETag` 和 `Last-Modified` 用于协商缓存，浏览器带 `If-None-Match` 或 `If-Modified-Since` 请求服务器，未变化则返回 304。通常 `ETag` 精度更高，优先级高于 `Last-Modified`。

静态资源发布策略：

- JS/CSS/图片文件名带内容 hash，设置 `Cache-Control: public, max-age=31536000, immutable`。
- HTML 不长期缓存，设置 `no-cache` 或较短缓存，让入口文件能及时拿到新资源地址。
- 发布保留历史 hash 资源，避免用户 HTML 和资源版本不一致。
- 回滚时切回旧 HTML 或发布旧资源引用，不覆盖同名 hash 文件。

关键考点：

- 长缓存依赖内容 hash，而不是覆盖原文件。
- HTML 和静态资源缓存策略应区分。

常见追问或变体：

- `no-cache` 和 `no-store` 区别是什么？
- CDN 缓存和浏览器缓存不一致怎么排查？

## 6. React 19 乐观更新与失败回滚

参考答案：

可以把评论提交建模为状态机：`idle -> optimistic -> success/failed`。提交时先插入一条带临时 id 和 `pending` 状态的评论；成功后用服务端返回的真实 id 替换；失败后删除临时评论或标记失败，并提示用户重试。

方案取舍：

- `useState` 手动维护：兼容性好，控制细，但要自己处理临时 id、并发提交和失败回滚。
- `useOptimistic`：适合 Action 期间的临时 UI，能让乐观状态和真实状态收敛，代码更贴近 React 19 模型。
- 服务端重新拉取：一致性最好，但即时反馈差，适合作为成功后的最终校准。

关键考点：

- 乐观更新不是跳过错误处理，必须处理失败、重试和重复提交。
- 并发提交时需要临时 id 或队列保证替换准确。

常见追问或变体：

- 如何避免用户连续点击产生重复评论？
- Server Actions 场景下如何结合 `useActionState`？

## 7. Vue 3 响应式边界与性能

参考答案：

Vue 3 使用 Proxy 拦截读取和写入。渲染或 `computed/watch` 执行时读取响应式属性会触发依赖收集；属性变化时触发对应 effect 调度，从而更新视图或重新计算。

大对象优化思路：

- 不需要深层响应式的数据用 `shallowRef`、`shallowReactive` 或 `markRaw`。
- 大列表展示使用虚拟列表，只渲染可视区域。
- 编辑态用局部副本，只让当前编辑行响应式。
- 列表项组件拆分并稳定 `key`，避免父级变更导致全量重渲染。
- 派生数据用 `computed`，避免模板里反复执行重计算。

关键考点：

- 响应式越深不一定越好，要控制响应式边界。
- 大列表性能通常同时涉及响应式、渲染数量和组件更新范围。

常见追问或变体：

- `ref` 和 `reactive` 如何选择？
- `watch` 和 `watchEffect` 的区别是什么？

## 8. 手写 LRU Cache

参考答案：

现代 JavaScript 可用 `Map` 的插入顺序实现简洁版 LRU：每次访问先删除再重新插入，把最近使用的 key 移到末尾。

```js
class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.map = new Map();
  }

  get(key) {
    if (!this.map.has(key)) return -1;
    const value = this.map.get(key);
    this.map.delete(key);
    this.map.set(key, value);
    return value;
  }

  put(key, value) {
    if (this.map.has(key)) {
      this.map.delete(key);
    }
    this.map.set(key, value);

    if (this.map.size > this.capacity) {
      const oldestKey = this.map.keys().next().value;
      this.map.delete(oldestKey);
    }
  }
}
```

关键考点：

- LRU 的核心是“访问后更新新旧顺序”。
- `Map` 保留插入顺序，删除再插入可以表达最近访问。
- 经典实现是哈希表 + 双向链表，适合语言没有有序 Map 的情况。

常见追问或变体：

## 9. 前端权限与路由守卫设计

参考答案：

权限模型可分为路由权限、菜单权限、按钮权限和数据权限。登录后获取用户信息、角色、权限点和菜单树；前端根据权限点生成可访问路由，路由守卫检查 token、用户信息是否已加载、目标路由是否命中权限；按钮用指令、Hook 或组件封装判断。

缓存策略：

- token 放安全存储，权限信息可内存 + 本地缓存，但要有版本号或过期时间。
- 用户切换角色、权限变更、收到 401/403 时刷新权限。
- 动态路由要处理刷新页面后的重建。

不能只放前端的部分：

- 接口鉴权、数据范围、敏感操作审批必须由服务端校验。
- 前端权限只负责体验和减少误操作，不能作为安全边界。

关键考点：

- 菜单隐藏不等于权限控制。
- 401 是未认证，403 是已认证但无权限。

常见追问或变体：

- 多租户系统如何设计租户隔离？
- 按钮权限如何避免散落在业务代码里？

## 10. Next.js App Router 缓存策略

参考答案：

商品基础信息变化低，可放在 Server Component 中获取，设置较长 revalidate 或按 tag 失效。库存状态变化快，应动态获取，或用短缓存并在下单、补货后触发失效。用户专属价格依赖登录态，不能静态缓存到公共页面，应基于 cookies/session 在服务端动态获取，或放到 Client Component 调用用户态接口。

设计示例：

- `ProductPage` Server Component 获取商品标题、描述、图片，使用 `revalidate` 或 tag 缓存。
- `StockBadge` 可服务端动态渲染，或客户端轮询/订阅。
- `UserPrice` 放 Client Component 或动态 Server Component，明确 `no-store`，避免串用户。
- 对后台更新商品信息使用 on-demand revalidation。

关键考点：

- 公共数据和用户私有数据必须分开缓存。
- App Router 中要理解静态渲染、动态渲染、数据缓存和路由缓存的边界。

常见追问或变体：

- 使用 cookies 会如何影响动态渲染？
- 如何避免用户 A 看到用户 B 的价格？

## 11. Node.js 高并发接口稳定性

参考答案：

定位先看分层指标：BFF 自身 CPU、内存、事件循环延迟、GC；每个下游服务的耗时、错误率、超时率；连接池排队；网关和负载均衡 502 来源。用 traceId 串起一次请求的 5 个下游调用，找 P95 拉高是单个下游慢、串行调用过多，还是 BFF 主线程阻塞。

治理策略：

- 能并行的下游请求并行，必要时限制并发。
- 每个下游设置独立超时，整体接口设置总 deadline。
- 重试只针对幂等和短暂错误，使用退避和重试预算。
- 非核心数据降级为空值或缓存兜底。
- 配置 HTTP keep-alive、连接池上限和排队上限。
- 对入口加限流和熔断，保护 BFF 和下游。

关键考点：

- 高并发问题通常不是“Node 单线程不行”，而是 I/O、下游、连接池和主线程长任务共同作用。
- 重试可能放大故障，必须有边界。

常见追问或变体：

- 如何监控事件循环延迟？
- BFF 聚合接口如何设计部分失败响应？

## 12. NestJS 鉴权链路设计

参考答案：

推荐链路：

- Guard：解析 JWT、校验登录态和角色/权限，决定请求是否能进入 handler。
- Pipe：做 DTO 参数校验、类型转换和基础格式检查。
- Interceptor：记录耗时、审计日志、统一包装响应，或在 handler 前后增加横切逻辑。
- Exception Filter：统一处理业务异常、鉴权异常、参数异常和未知错误。

JWT 解析可以放在 Auth Guard 中，把用户信息挂到 request；权限校验可以用自定义装饰器声明权限点，再由 Roles/Permission Guard 读取 metadata；参数校验用 `ValidationPipe`；审计日志适合 Interceptor，因为它需要知道请求、用户、耗时和结果。

关键考点：

- Guard 负责“能不能进”，Pipe 负责“参数对不对”，Interceptor 负责“过程增强”，Filter 负责“异常出口”。
- 权限元数据应靠装饰器声明，不要散落在业务函数里。

常见追问或变体：

- Refresh Token 如何设计？
- 多租户权限如何校验 tenantId？

## 13. 数据库接口分页与一致性

参考答案：

`offset/limit` 简单，适合后台小数据分页，但数据量大时 offset 越大扫描成本越高；并发写入时还可能出现重复或漏数据。cursor 分页用稳定排序字段作为游标，适合无限滚动和大数据列表。

接口协议示例：

```http
GET /messages?limit=20&cursor=2026-06-23T10:00:00.000Z_98765
```

返回：

```json
{
  "items": [],
  "nextCursor": "2026-06-23T09:58:00.000Z_98701",
  "hasMore": true
}
```

排序可用 `(created_at desc, id desc)`，游标包含 `created_at` 和 `id`。查询条件是取比当前游标更旧的数据：`created_at < cursorTime OR (created_at = cursorTime AND id < cursorId)`。索引建立联合索引 `(created_at, id)`，根据数据库和查询方向调整顺序。

关键考点：

- cursor 要基于稳定且可唯一排序的字段。
- 只用时间戳可能因同一毫秒多条数据导致丢失。

常见追问或变体：

- 如何支持向上加载新消息？
- 删除消息后 cursor 是否会失效？

## 14. AI 前端流式响应交互

参考答案：

协议选择：

- 纯服务端到前端单向流式输出，SSE 简单稳定。
- 需要双向实时通信或多路事件，WebSocket 更合适。
- 在现代浏览器里也可用 `fetch` + `ReadableStream` 读取流。

前端状态机：

- 用户消息：`sending -> sent -> failed`。
- 助手消息：`pending -> streaming -> done/failed/canceled`。
- 每个 chunk 追加到当前 assistant message，使用节流批量 setState，避免每个 token 都触发重渲染。

能力设计：

- 停止生成用 `AbortController` 或发送 cancel 请求。
- 重试要带上原始上下文和失败消息 id。
- 错误时保留已生成内容并显示可重试状态。
- 首 token 延迟通过连接复用、减少前置 RAG 阻塞、服务端尽早 flush 状态事件优化。

关键考点：

- 流式响应是协议、状态机和渲染性能问题，不只是不断拼字符串。
- Markdown 增量渲染要处理未闭合代码块。

常见追问或变体：

- 如何展示工具调用中的中间状态？
- 如何处理刷新页面后的会话恢复？

## 15. Agent Tool Calling 与 MCP 安全边界

参考答案：

工具注册应包含 name、description、input schema、输出结构、权限级别和是否需要人工确认。参数 schema 要严格，避免自由文本直接进入 SQL、Shell 或内部 API。Agent 调用工具前先由策略层检查用户身份、租户、角色和工具作用域；高风险工具如创建发布单、写数据库必须二次确认。

安全设计：

- 工具最小权限，按用户和场景授权。
- 对工具输入做 schema 校验和业务白名单校验。
- RAG 文档、工单正文等外部内容视为不可信，防止提示词注入诱导调用高危工具。
- 工具调用、参数摘要、操作者、结果、耗时都进入审计日志。
- 对敏感输出做脱敏，避免 Agent 把密钥、隐私数据回显给用户。

前端展示：

- 展示“准备调用工具、等待确认、执行中、成功/失败”的过程。
- 高风险操作展示参数摘要和确认按钮。
- 工具失败时给出可理解错误和重试入口。

关键考点：

- Tool Calling/MCP 扩展的是 Agent 行动能力，也扩大了安全边界。
- 工具描述不能替代权限系统。

常见追问或变体：

- 如何防止 lookalike tool 或恶意同名工具？
- Agent 误调用工具造成副作用时如何回滚？
