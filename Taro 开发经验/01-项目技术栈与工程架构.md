# 项目技术栈与工程架构

## 总体架构

Papertown 的核心技术栈：

- Taro 4.1.11
- React 18
- TypeScript
- Redux Toolkit + react-redux
- SCSS
- axios + `@tarojs/plugin-http`
- `@diezhi/account` 和 `@diezhi/pudge` 等内部业务 SDK
- `@arms/rum-miniapp` / `@arms/rum-browser` 做端侧监控
- `wxml2canvas-2d` 和 `html2canvas-pro` 做分享图能力

这套架构适合“业务状态复杂、跨端差异多、页面需要强适配”的活动型或会员型项目。

## 目录分层

项目主要目录如下：

```text
src/
├── pages/          # 页面与页面私有组件
├── components/     # 跨页面公共组件
├── model/          # Redux Toolkit slice 与 dispatch 封装
├── store/          # store 聚合与类型
├── services/       # API、账号、Passport 请求层
├── hooks/          # 跨页面 hooks
├── utils/          # 工具函数，含平台差异文件
├── constants/      # 常量、枚举、环境、埋点字典
└── types/          # 全局类型与业务类型
```

从 0 到 1 搭项目时，建议先把这些边界定清楚。页面只做编排，复杂状态进 `model`，接口进 `services`，跨端差异进 `utils` 或组件平台文件，公共 UI 进 `components`。

## 项目启动链路

入口在 `src/app.tsx`：

- 先加载 `@/polyfill`，保证内部账号 SDK 依赖的全局能力先完成注入。
- 调用 `initSystem()` 初始化设备静态信息。
- 调用 `initArmsRum()` 初始化生产监控。
- `useLaunch` 中初始化动态设备信息、同步小程序身份、初始化订阅消息、初始化音频。
- `useDidShow` 中加载字体，并补充消息卡片入口埋点。
- 根组件用 Redux `Provider` 包住业务页面。

这个启动链路体现了一个经验：跨页面共享且必须尽早完成的能力，应该在 App 层集中初始化；页面内只消费结果。

## 页面容器

项目所有页面通过 `PaperPage` 作为统一容器，承担：

- 登录态恢复与未登录重定向。
- 设备动态宽高读取。
- `--valid-page-width` 注入。
- H5 PC 不支持提示。
- 高度压缩状态 class 切换。
- 页面内容固定在有效宽度内渲染。

这比每个页面自行处理登录和适配更稳定，也方便后续统一修复兼容问题。

## 状态层设计

Redux store 聚合了：

- `user`：登录、会员、停服、默认引导等用户态。
- `device`：窗口宽高、有效页面宽度、高度压缩状态。
- `loading`：全局请求 loading。
- `audio`：BGM 与点击音效。
- `town`：小镇信息、编辑模式、快照、侧边栏。
- `privacy`：隐私协议。
- `subscribe`：订阅消息设置和场景状态。

状态切片按照业务域拆分，dispatch 再在 `model/*/dispatch.ts` 中封装，页面不用直接知道 action 细节。

## 可复用经验

从 0 到 1 做 Taro 项目时，架构上不要只关注“页面能跑”。更重要的是提前确定这些工程能力：

- App 初始化顺序。
- 页面容器职责。
- 多端差异隔离方式。
- API 请求通道。
- 登录态恢复和失效处理。
- 全局 loading 与错误治理。
- 设备宽高与设计稿适配。
- 业务域状态拆分。
- 小程序分包和包体控制。

## 面试表达

可以这样讲：

> 我会把 Taro 项目拆成页面层、组件层、状态层、服务层、工具层和平台隔离层。页面层只做业务编排，跨页面状态用 Redux Toolkit，接口统一走 Passport 请求封装，登录态恢复、设备尺寸和适配变量放在页面容器中处理。遇到小程序/H5 行为差异，简单分支用 `process.env.TARO_ENV`，复杂能力用 `.h5.ts/.weapp.ts` 拆文件。
