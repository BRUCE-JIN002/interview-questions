# 状态管理与业务数据流

## 为什么用 Redux Toolkit

Papertown 的业务状态跨页面、跨组件、跨生命周期，例如：

- 用户登录态和会员信息。
- 设备宽高和有效页面宽度。
- 全局 loading。
- 小镇信息、编辑模式、素材快照。
- 音频状态。
- 订阅消息设置。
- 停服状态。

这些状态如果都放在页面 `useState` 中，会出现重复请求、状态不一致和页面切换丢失的问题。

## Store 结构

`src/store/index.ts` 聚合：

```ts
reducer: {
  user,
  device,
  loading,
  audio,
  town,
  privacy,
  subscribe,
}
```

经验：按业务域拆 slice，而不是按页面拆。页面可能卸载，业务状态需要保留。

## Slice 与 Dispatch 封装

项目习惯：

- `slice.ts` 定义状态、reducer、selector。
- `dispatch.ts` 封装对 store 的派发。
- 页面和 hook 优先调用 `dispatchXXX`，不直接关心 action 细节。

例子：`town` 模块有 `setTownInfo`、`enterEditMode`、`saveEdit`、`cancelEdit`、`confirmSkinChange` 等 reducer，并在 `dispatch.ts` 中封装。

## 编辑态快照

小镇编辑是典型复杂业务：

- 展示模式进入编辑模式时创建快照。
- 编辑中本地调整素材。
- 保存时把素材拆回 building/decorate。
- 取消时恢复快照。
- 切换皮肤时保留或替换布局。

相关实现：

- `src/model/town/slice.ts`
- `src/model/town/utils.ts`
- `src/pages/town/hooks/useTownCanvas.ts`
- `src/pages/town/components/TownCanvas/index.tsx`

可复用经验：复杂编辑器一定要有“原始快照”和“当前草稿”。不要直接改原始服务端数据，否则取消、重置、比较变化都会变复杂。

## 初始化缓存

`src/model/town/initCache.ts` 把首页初始化接口一次返回的数据缓存起来：

- 活动公告。
- 游戏导航公告。
- 分享补充信息。
- 侧边栏。
- 个人信息。

消费方用 `consumeXXXCache` 读取后清除，避免重复请求和重复消费。

经验：首屏接口如果已经返回多个后续弹窗或模块要用的数据，可以做一次性消费缓存。但要明确消费语义，避免缓存长期不释放。

## Hook 分层

项目把复杂页面逻辑拆进 hook：

- `useTownActions`：弹窗、导航、初始化、公告、分享入口。
- `useTownCanvas`：画布内部状态、滚动位置、素材列表。
- `useCanvasMaterialEdit`：素材编辑动作。
- `usePosterGeneration`：海报生成状态机。
- `useInitialTownLayout`：初始布局数据。

经验：页面组件负责组装 UI，业务流程和状态机尽量下沉到 hook。hook 不只是复用逻辑，也是降低页面复杂度的结构工具。

## 面试表达

> 项目里我会用 Redux Toolkit 管理跨页面状态，比如用户、设备、小镇数据、全局 loading 和订阅消息。复杂编辑类业务会设计快照机制，进入编辑态时保存原始状态，保存时提交，取消时恢复。首屏接口返回的多模块数据会用一次性缓存分发给后续弹窗和页面，避免重复请求。页面本身只负责编排，复杂业务流程拆成 hook。
