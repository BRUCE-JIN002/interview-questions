# Taro 开发经验总览

这套文档从 Papertown 当前代码中提炼，目标不是复述 Taro 官方教程，而是回答两个问题：

1. 如果从 0 到 1 重新构建一个微信小程序 + H5 双端项目，关键技术点应该怎么搭。
2. 如果在面试或实际工作中讲 Taro 项目经验，哪些问题最能体现工程判断、兼容经验和优化能力。

## 项目核心判断

Papertown 是一个 Taro 4 + React 18 + TypeScript 的多端项目，主要运行在微信小程序和 H5。它的复杂度不在页面数量，而在以下几个工程问题：

- 多端构建：同一套业务代码产出 H5 和微信小程序。
- 登录态和接口治理：账号 SDK、业务 Passport 请求、Redux 状态和本地缓存协同。
- 页面适配：以 390px 设计稿为基准，通过 `--valid-page-width` 支撑手机、平板、宽屏 H5。
- H5 视口稳定性：处理 `vh/dvh`、地址栏、键盘、华为浏览器异常高度。
- 小程序平台约束：分包、订阅消息、隐私能力、限频 API、域名白名单。
- Canvas/分享图：小程序用 `wxml2canvas-2d`，H5 用 `html2canvas-pro`，通过平台文件拆分实现。
- 用户体验兜底：全局 loading、停服态、重复导航锁、重复分享生成锁、初始化缓存。

## 建议学习顺序

1. 先看 `01-项目技术栈与工程架构.md`，建立全局地图。
2. 再看 `02-Taro多端构建与目录设计.md`，理解项目如何产出两端。
3. 接着看 `03-页面容器与宽屏适配.md` 和 `04-样式单位与视口高度.md`，这是本项目最有学习价值的适配经验。
4. 然后看 `05-平台差异隔离与兼容边界.md`，建立 Taro 双端开发的风险意识。
5. 继续看 `06-状态管理与业务数据流.md`、`07-服务层登录态与接口治理.md`，理解业务项目如何组织数据流。
6. 最后看弹窗、Canvas、性能、小程序平台能力和面试题库，把项目经验整理成可表达的能力。

## 从 0 到 1 搭建时的主路径

一个可复用的 Taro 项目搭建顺序：

1. 初始化 Taro + React + TypeScript，确定目标端是 `weapp` 和 `h5`。
2. 设计目录：`pages`、`components`、`model`、`store`、`services`、`hooks`、`utils`、`constants`、`types`。
3. 配置 `app.config.ts`：主包页面、分包、预加载、窗口样式。
4. 配置构建：Taro `defineConfig`、环境变量、路径别名、H5 publicPath、微信小程序包体优化。
5. 建立页面容器：统一处理登录态恢复、设备尺寸、PC/H5 兜底、页面宽高变量。
6. 建立状态层：用 Redux Toolkit 管理用户、设备、业务模块、loading、订阅消息等跨页面状态。
7. 建立服务层：统一请求实例、通参、loading、错误、停服态、服务端时间差。
8. 建立跨端组件体系：弹窗、Loading、富文本、图片、分享图、Observer 等。
9. 建立平台隔离策略：少量差异用 `process.env.TARO_ENV`，大差异用 `.h5.ts/.weapp.ts`。
10. 建立验证方式：H5 浏览器实测，小程序开发者工具和真机分别验证。

## 面试表达主线

可以按这条线组织回答：

> 我做过一个 Taro 4 + React 18 的小程序/H5 双端项目。项目不是简单复用页面，而是围绕多端构建、页面适配、登录态、请求治理、Canvas 分享图和小程序平台能力做了一套工程化封装。比如页面宽度统一从 `PaperPage` 注入 `--valid-page-width`，CSS 构建时把 px/vw 转成基于有效宽度的 calc，H5 端再通过 dvh 哨兵和 ResizeObserver 修正地址栏、键盘导致的视口高度问题。平台差异较大的能力，比如 Canvas 分享图、上传、音频，会拆成 `.h5.ts` 和 `.weapp.ts`。业务数据则通过 Redux Toolkit 管理，服务层统一处理登录通参、loading、错误 toast 和停服回退。

这类表达比“我会 Taro 组件和生命周期”更有含金量，因为它体现了完整工程能力。
