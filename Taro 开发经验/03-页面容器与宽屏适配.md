# 页面容器与宽屏适配

## 问题背景

移动端设计稿通常按一个固定宽度设计，本项目设计稿宽度为 390px。但实际运行环境包括：

- iPhone 和 Android 手机。
- 小程序容器。
- H5 浏览器。
- 平板和宽屏设备。
- H5 PC 访问。

如果直接让页面按屏幕宽度拉伸，宽屏上视觉会变形；如果只按 390px 写死，又会在不同设备上出现边界问题。

## 项目方案

项目定义：

- `DESIGN_WIDTH = 390`
- `DESIGN_HEIGHT = 844`
- `DESIGN_RATIO = 390 / 844`
- `MAX_PAGE_RATIO = 0.6`

逻辑在 `src/hooks/useSystemInfo.ts`：

```ts
const calcValidPageWidth = (windowWidth, windowHeight) =>
  windowWidth / windowHeight > MAX_PAGE_RATIO ? Math.floor(windowHeight * DESIGN_RATIO) : windowWidth
```

含义：

- 普通手机：有效页面宽度等于窗口宽度。
- 宽屏设备：按设计稿比例算一个居中的有效宽度，避免横向拉伸。

`PaperPage` 再把 `validPageWidth` 注入为 CSS 变量：

```tsx
<View
  className="screen-box"
  style={{ '--valid-page-width': `${validPageWidth}px` }}
>
```

内部 `.content-box` 使用 `windowHeight` 和 `validPageWidth` 控制实际页面容器。

## 关键经验

页面适配不要只依赖 CSS 媒体查询。Taro 小程序和 H5 的根节点、视口、滚动容器并不完全一致，项目选择在 JS 中统一计算有效宽度，再通过 CSS 变量下发给样式层。

这样做有三个好处：

- JS 内联样式、Canvas 计算、CSS 样式可以使用同一套宽度基准。
- 宽屏和平板适配逻辑集中在 `useSystemInfo` 和 `PaperPage`。
- 后续如果设计稿基准变化，只需要调整核心常量和转换工具。

## 页面容器职责

`PaperPage` 不只是一个布局壳，它还处理：

- 页面 ready 状态。
- 登录态恢复。
- 未登录重定向。
- H5 PC 端拦截和二维码提示。
- 页面高度压缩 class。
- 页面宽高注入。

这类能力集中到页面容器，可以避免每个页面重复写初始化逻辑。

## 从 0 到 1 实现步骤

1. 定义设计稿宽高和最大宽高比阈值。
2. 在全局设备 hook 中读取窗口宽高。
3. 根据宽高比计算 `validPageWidth`。
4. 把设备信息写入全局状态。
5. 页面容器订阅设备信息。
6. 页面容器注入 `--valid-page-width`。
7. CSS 和 JS 动态尺寸都基于这个变量或对应换算函数。

## 面试表达

> 我们没有简单使用 Taro 默认 rpx，而是设计了一个有效页面宽度机制。页面实际宽度由 `windowWidth/windowHeight` 和设计稿比例计算，普通手机铺满，宽屏按设计稿比例居中。`PaperPage` 把这个值注入成 `--valid-page-width`，CSS 构建时把 px/vw 转成基于这个变量的 calc，JS 内联样式用同一个宽度换算函数。这样 H5、小程序、平板宽屏都能保持一致比例。
