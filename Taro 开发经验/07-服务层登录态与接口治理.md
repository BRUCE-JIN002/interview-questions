# 服务层登录态与接口治理

## 服务层分工

项目服务层主要有：

- `src/services/index.ts`：导出服务模块和普通 axios 实例。
- `src/services/passport.ts`：业务 Passport 请求封装。
- `src/services/api.ts`：业务接口方法。
- `src/services/account.ts`：账号登录、缓存恢复、小程序身份绑定。
- `src/services/type.d.ts`：接口类型。

经验：请求层不要散落在页面中。页面调用业务语义方法，如 `getTownInit`、`townSave`，而不是到处拼 URL。

## Passport 请求封装

`businessPassportRequest` 负责：

- 生成请求 ID。
- 控制全局 loading。
- 序列化对象参数。
- 调用 `paperPassport.request`。
- 同步服务端时间差。
- 捕获错误并 toast。
- 识别停服错误码。
- 给错误对象补 `requestInfo.url` 方便排查。

这就是接口治理的核心：把通用问题放在一个请求入口里处理。

## Loading 设计

请求发起时：

- 默认展示全局 loading。
- 可以通过 `hideLoading` 禁用。
- 每个请求用唯一 ID 管理。
- 请求结束后隐藏对应 loading。

经验：全局 loading 不能只用一个 boolean，否则并发请求会互相覆盖。用 requestId 或计数器更稳定。

## 错误与停服

请求失败时：

- 维护状态码会写入 user 状态。
- 触发统一回到小镇页。
- 小镇页监听停服态后清空弹窗、重置 town、停止音频，并回到 loading 页。
- 非停服错误默认 toast，可由接口参数关闭。

经验：停服、登录失效、鉴权失败这类全局状态不适合在每个页面单独处理，应收敛到服务层和全局状态。

## 登录态恢复

`PaperPage` 在 ready 时调用 `initLoginInfoFromStorage`：

- 从本地缓存读取 nid/token。
- 调用账号 SDK 校验 profile。
- 恢复 Redux user 状态。
- 更新 Passport 请求通参。
- 同步 tlog 通参。
- 小程序端登录后绑定 openid/unionid。
- 失败则清理缓存并打开登录弹窗。

经验：登录态恢复不是“读 localStorage 就完了”，必须校验 token 是否有效，并同步请求层和埋点层。

## 小程序身份同步

`syncWeappIdentity`：

- 仅微信小程序端执行。
- 调用 `Taro.login()` 拿 code。
- 用免登录态接口换 openid/unionid。
- 写入 Redux。
- 更新 tlog 通参。
- 用 promise 去重，避免并发重复调用。

这也体现了小程序限频意识：`wx.login` 不应被页面反复触发。

## 面试表达

> 我会把业务接口统一封装成服务层，页面只调用语义化 API。请求入口负责 loading、参数序列化、错误 toast、停服态、服务端时间差和错误排查信息。登录态恢复时不只读缓存，还会调用账号 SDK 校验 token，然后同步 Redux、Passport 通参和埋点通参。小程序 openid/unionid 同步用 promise 去重，避免多处同时调用 `Taro.login()`。
