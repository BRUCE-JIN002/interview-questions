# 样式单位与视口高度

## 为什么不用 Taro 默认 pxTransform

Taro 默认 `px -> rpx` 常以 750px 设计稿为基准，而 Papertown 的设计稿宽度是 390px。项目在 `config/index.ts` 中关闭了小程序默认转换：

```ts
mini: {
  postcss: {
    pxtransform: {
      enable: false,
    },
  },
}
```

然后用自定义 PostCSS 插件统一处理尺寸。

## 自定义单位转换

`.postcssrc.js` 加载：

```js
const plugins = [require('./scripts/postcss-transform-responsive-units')()]

if (process.env.TARO_ENV === 'h5') {
  plugins.push(require('./scripts/postcss-vh-to-dvh')())
}
```

`postcss-transform-responsive-units` 的核心规则：

- `xpx` 转成 `calc(x/390*var(--valid-page-width))`
- `xvw` 转成 `calc(x/100*var(--valid-page-width))`
- 跳过 `env(...)` 内部的 px/vw，避免安全区 fallback 在旧浏览器中失效。

经验：当项目需要适配宽屏时，`vw` 不一定等于设计里的页面宽度。把 `vw` 也转成基于 `--valid-page-width`，可以避免宽屏 H5 被拉得过大。

## JS 内联尺寸

CSS 可以在构建时转换，但 JS 内联样式不经过 PostCSS，所以项目提供 `createResponsivePx`：

```ts
const rpx = createResponsivePx(validPageWidth)
style={{ height: rpx(69) }}
```

规则：

- JSX 内联的宽高不要裸写 `'100px'`。
- `calc(...)` 里的动态基准值也先用 `rpx()` 转换。
- Canvas 或海报缩放时，设计稿值乘以 renderScale 后再转换。

## H5 视口高度问题

移动 H5 的 `100vh` 经常包含地址栏或工具栏，导致页面高度偏大。项目用了两层方案：

1. 构建时：`postcss-vh-to-dvh` 给 `vh` 声明追加 `@supports (height: 1dvh)` 覆盖。
2. 运行时：`vh()` 根据浏览器是否支持 `dvh` 返回 `dvh` 或 `vh`。
3. 像素级布局：`useSystemInfo` 维护可靠 `windowHeight`，`PaperPage` 使用该高度。

## dvh 哨兵方案

`useSystemInfo.ts` 创建一个隐藏元素：

```ts
height:100dvh;
position:fixed;
visibility:hidden;
```

再通过 `ResizeObserver` 监听它的高度变化。这个方案用于处理 iOS 内嵌浏览器中初始 `dvh` 偏大、工具栏动画完成后才修正的问题。

额外还处理了：

- 华为浏览器 SPA 路由切换时 `innerHeight` 异常翻倍。
- 键盘弹出时高度变小但不应重设页面画布。
- 旋转和分屏导致宽度变化时重新计算宽高。
- Android Chrome 键盘收起后 dvh 异步恢复的延迟复核。

## 小程序字号坑

微信小程序部分 Android 机型对小于 10px 的字号会向上取整，导致视觉偏大。经验规则：

- `font-size` 不要小于 10px。
- 需要 8px、9px 等视觉效果时，用 `font-size: 10px` + `transform: scale(...)`。
- 参与 Canvas 截图的元素尤其要注意，小字号会影响海报清晰度。

## 面试表达

> 我们的适配不是简单用 rpx。项目关闭 Taro 默认 pxTransform，基于 390px 设计稿写了 PostCSS 插件，把 px/vw 转成 `calc(... * var(--valid-page-width))`。页面有效宽度由 JS 根据窗口宽高比计算。H5 的高度问题则用 `vh/dvh` 构建时兜底，加上运行时 dvh 哨兵和 ResizeObserver 修正真实可视高度。这样可以同时处理手机、平板、H5 地址栏和键盘场景。
