# 弹窗固定层与交互控制

## 弹窗体系

项目有三层弹窗：

- `Popup`：基础弹窗，支持 top/right/bottom/left/center。
- `BasePopup`：统一业务弹窗骨架和背景图。
- `BlurPopup`：带模糊背景和闪光蒙层的弹窗变体。

新弹窗优先基于 `BasePopup` 或 `BlurPopup`，不要每个业务重新实现一套遮罩、层级、关闭逻辑。

## Popup 基础能力

`Popup` 支持：

- `open`
- `placement`
- `overlay`
- `closeOnClickOverlay`
- `lockScroll`
- `duration`
- `zIndex`
- `overlayStyle`
- `overlayClass`
- `onClose`
- `onTransitionEnd`

经验：基础弹窗应该把“是否展示、位置、遮罩、锁滚动、动画、层级”抽象出来，业务弹窗只关注内容。

## H5 fixed 风险

Taro H5 路由动画可能让 `.taro_page` 带 transform。此时 `position: fixed` 不是相对窗口，而是相对 transform 祖先，弹窗和遮罩容易错位。

项目经验：

- 固定层优先走统一 `Popup`。
- 需要业务背景和关闭按钮时走 `BasePopup`。
- 分享、预览、说明类弹窗走 `BlurPopup`。
- 不在业务组件里随意堆新的 fixed 容器。

## 滚动穿透

小程序中常见写法是 `catchMove` 或 `catchtouchmove`。项目的 `Popup` 在遮罩和弹层上使用 `catchMove={lockScroll}`。

双端经验：

- 小程序可以依赖 `catchMove`。
- H5 需要结合 `preventDefault` 或页面滚动容器处理。
- 弹窗体系里统一处理，比每个弹窗单独处理更可靠。

## 弹窗状态管理

`useTownActions` 用一个对象管理小镇页多个弹窗：

- 商店弹窗。
- 设置弹窗。
- 公告弹窗。
- 信息弹窗。
- 订阅弹窗。
- 消息授权弹窗。
- 分享弹窗。
- 游戏区域弹窗。

它还提供 `closeAllDialogs`，用于停服等全局状态清场。

经验：弹窗多的页面，应有统一状态模型和清场入口。否则全局状态切换时很容易出现弹窗叠加或脏状态。

## 交互防重

项目里有几个防重例子：

- `navigateTo` 用模块级锁避免连续点击重复打开页面。
- 分享生成用 `shareGeneratingRef` 防止重复生成。
- 订阅请求用 `subscribingRef` 防止重复拉起微信订阅弹窗。

经验：移动端用户会快速连点，关键交互必须有防重。不要只依赖按钮 disabled，因为跨组件调用或异步回调也可能重复触发。

## 面试表达

> 我会把弹窗拆成基础层和业务层。基础 `Popup` 处理 open、遮罩、动画、层级和锁滚动，业务弹窗基于 `BasePopup/BlurPopup` 扩展。Taro H5 下 fixed 可能受路由动画 transform 影响，所以固定层不要在业务里散写。多个弹窗页面用统一状态对象管理，并提供 closeAllDialogs 处理停服、退出等全局清场场景。关键交互用 ref 或模块锁防止重复点击。
