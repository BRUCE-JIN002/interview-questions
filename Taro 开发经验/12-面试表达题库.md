# 面试表达题库

## 1. 你们的 Taro 项目技术栈是什么？

答题要点：

- Taro 4 + React 18 + TypeScript。
- Redux Toolkit 管理跨页面状态。
- SCSS + 自定义 PostCSS 做适配。
- axios/Taro HTTP + Passport 请求封装。
- H5 和微信小程序双端构建。
- Canvas 分享图分别用 `wxml2canvas-2d` 和 `html2canvas-pro`。

参考回答：

> 项目是 Taro 4 + React 18 + TypeScript，小程序和 H5 双端。状态层用 Redux Toolkit，接口层封装 Passport 请求，样式上关闭 Taro 默认 pxTransform，改用自定义 PostCSS 基于 390px 设计稿和有效页面宽度适配。分享图能力两端差异较大，所以小程序用 `wxml2canvas-2d`，H5 用 `html2canvas-pro`，通过平台文件拆分统一接口。

## 2. Taro 双端项目怎么处理平台差异？

答题要点：

- 不强行复用所有代码。
- 小差异用 `process.env.TARO_ENV`。
- 大差异用 `.h5.ts/.weapp.ts`。
- Canvas、上传、音频、订阅消息适合拆分。

参考回答：

> 我会先判断差异是轻量逻辑还是能力模型不同。轻量差异，比如某个按钮只在小程序展示，用 `process.env.TARO_ENV`。如果依赖平台 API、渲染结构或生命周期，比如 Canvas、上传、音频、分享弹窗，就拆成 `.h5.ts/.weapp.ts`，对外暴露相同接口，避免平台补丁污染通用代码。

## 3. 你们是怎么做移动端适配的？

答题要点：

- 390px 设计稿。
- `validPageWidth`。
- `PaperPage` 注入 CSS 变量。
- PostCSS 转换 px/vw。
- JS 动态尺寸用 `createResponsivePx`。

参考回答：

> 我们没有直接用 Taro 默认 rpx，而是按 390px 设计稿维护有效页面宽度。页面宽高由 `useSystemInfo` 计算，宽屏时按设计稿比例居中，`PaperPage` 注入 `--valid-page-width`。CSS 构建时把 px/vw 转成基于这个变量的 calc，JS 内联样式用 `createResponsivePx`，这样 CSS、JS 和 Canvas 都用同一套尺寸基准。

## 4. H5 端 100vh 问题怎么解决？

答题要点：

- 移动浏览器地址栏/工具栏会影响 `vh`。
- 构建时追加 `dvh`。
- 运行时 `vh()` 工具。
- `useSystemInfo` 用 dvh 哨兵和 ResizeObserver 修正。
- 键盘弹出不应重设页面画布。

参考回答：

> H5 移动浏览器的 `100vh` 可能包含地址栏，所以项目做了构建时和运行时两层处理。CSS 构建时给 vh 追加 `@supports(height:1dvh)` 的 dvh 覆盖；JS 内联样式用 `vh()` 自动选择 vh/dvh；像素级布局通过 `useSystemInfo` 维护真实 `windowHeight`，并用隐藏的 100dvh 哨兵加 ResizeObserver 处理 iOS 工具栏动画后的高度修正，同时识别键盘导致的高度压缩。

## 5. 业务接口层怎么设计？

答题要点：

- 页面不直接拼 URL。
- API map 和语义方法。
- 统一请求入口处理 loading、错误、停服、通参。
- 登录态恢复后同步 Passport 通参。

参考回答：

> 页面只调用语义化 API，比如 `getTownInit`、`townSave`。底层统一走 `businessPassportRequest`，它负责 loading、参数序列化、错误 toast、停服态、服务端时间差和 requestInfo。登录态恢复成功后会同步 Redux、账号 SDK、Passport 通参和埋点通参，这样接口调用方不用重复处理这些横切逻辑。

## 6. Redux 在项目里解决了什么问题？

答题要点：

- 跨页面状态。
- 设备尺寸、登录态、业务数据、loading、订阅消息。
- 复杂编辑态用快照。
- selector 降低无关重渲染。

参考回答：

> Redux Toolkit 主要管理跨页面和跨生命周期状态，比如 user、device、loading、town、subscribe。小镇编辑态会在进入编辑时创建快照，保存时提交，取消时恢复，避免直接修改原始服务端数据。页面通过 selector 订阅必要字段，复杂流程拆到 hook 里。

## 7. 分享图怎么做跨端？

答题要点：

- 统一 `CanvasTemplate` 接口。
- 小程序 `wxml2canvas-2d`。
- H5 `html2canvas-pro`。
- H5 clone DOM 修补 background-image 和精灵图。
- 预生成两版海报并缓存。

参考回答：

> 分享图对外是统一 `generateImg`。小程序端用 `wxml2canvas-2d` 原生组件 draw 后导出临时文件，H5 用 `html2canvas-pro` 截 DOM。H5 截图前会在 clone DOM 里把 background-image 替换成 img，并提取精灵图首帧保证清晰度。分享弹窗会预生成带用户信息和不带用户信息两版，全部就绪后再显示，切换时直接用缓存，避免闪烁。

## 8. 小程序订阅消息要注意什么？

答题要点：

- 主开关。
- 模板 itemSettings。
- 永久拒绝。
- 授权结果上报。
- 限频。

参考回答：

> 订阅消息不能只调 `requestSubscribeMessage`。需要先查主开关，再判断模板是否被用户永久拒绝；如果永久拒绝，直接引导去设置页。发起订阅后要上报用户选择，并在用户当次勾选“总是保持以上选择”且拒绝时再次查询设置，引导开启。`getSetting` 和订阅请求也要避免高频调用。

## 9. 如何控制小程序包体和首屏性能？

答题要点：

- 主包只放首屏。
- 分包。
- 预加载。
- 初始化缓存。
- 避免重复请求。

参考回答：

> 小程序端主包只保留小镇首页，用户、webview、徽章、名片设置等放分包，并对高频分包配置 preloadRule。首屏接口返回的公告、导航、分享信息和个人信息会放到一次性缓存里，后续弹窗消费，避免重复请求。构建上也开启 main package 优化。

## 10. 你从这个项目里总结的 Taro 开发原则是什么？

参考回答：

> 第一，Taro 复用的是业务逻辑，不是强行抹平所有平台能力。第二，适配要有统一尺寸基准，不能 CSS、JS、Canvas 各算各的。第三，登录态、请求、loading、错误、停服这类横切逻辑必须收敛到服务层。第四，复杂页面要用状态机、快照和 hook 拆解。第五，小程序平台约束要前置考虑，比如域名白名单、限频 API、隐私合规和 Canvas 资源格式。
