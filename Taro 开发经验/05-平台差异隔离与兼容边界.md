# 平台差异隔离与兼容边界

## 核心原则

Taro 的价值是复用业务逻辑，不是抹平所有平台差异。真正做项目时，需要明确哪些代码可以通用，哪些必须平台隔离。

项目采用两种方式：

- 少量差异：`process.env.TARO_ENV` 条件判断。
- 大差异：`.h5.ts` / `.weapp.ts` 文件拆分。

## 哪些能力适合文件拆分

项目中的例子：

- Canvas 分享图：H5 用 DOM 截图，小程序用原生组件。
- QRCode：H5 和小程序实现不同。
- 分享弹窗：H5 和小程序交互能力不同。
- 上传：小程序上传和 H5 上传 API 不同。
- 音频管理：浏览器和小程序音频能力不同。
- 埋点：H5 和小程序上报环境不同。

判断标准：如果平台差异会影响依赖、渲染结构、生命周期、API 参数或错误处理，就不要塞在一个文件里。

## 高风险 Taro API

需要提前评估：

- `catchMove` / `catchtouchmove`：小程序有效，H5 不等价。
- `createSelectorQuery` 获取 canvas node：H5 拿不到小程序式 node。
- `createAnimation`：H5 支持有限。
- `onPageScroll` / `onReachBottom`：H5 可能需要监听 `.taro_page`。
- `Taro.preload`：H5 不可用。
- `getCurrentPages().route`：H5 缺少 route 字段。
- `setTabBarBadge`：H5 可能不实时更新。
- Canvas API：两端能力差异大。

## CSS 兼容边界

Taro 双端样式中要避免：

- 标签选择器，如 `View {}`、`Image {}`。
- 只把 CSS 变量挂在 `:root`，小程序页面树继承不可靠。
- H5 里滥用 `position: fixed`，路由动画可能让 `.taro_page` 带 transform，导致 fixed 相对祖先定位。
- 依赖 `-webkit-overflow-scrolling: touch` 的固定层，可能触发 z-index 问题。

Papertown 的项目内解法是优先复用统一 `Popup/BasePopup/BlurPopup` 承载固定层和弹窗。

## 事件兼容

阻止滚动穿透的通用写法：

```tsx
<View
  onTouchMove={(e) => {
    e.preventDefault()
    e.stopPropagation()
  }}
  catchMove={true}
/>
```

项目中 `Popup` 使用 `catchMove` 锁定背景滚动，小程序端有效；涉及 H5 时仍需结合真实场景检查。

## 从 0 到 1 的风险评估清单

写方案前先问：

- 这个能力依赖小程序专属 API 吗？
- H5 是否有等价 API？
- 是否涉及 Canvas、音频、上传、订阅消息、隐私接口？
- CSS 是否依赖 fixed、滚动容器、根变量、标签选择器？
- 文件拆分是否比运行时分支更清晰？
- 是否需要真机验证？

## 面试表达

> 我对 Taro 的理解是“复用业务，不强行复用平台能力”。比如 Canvas 分享图、小程序订阅、上传和音频这类能力，两端 API 差异大，我会用 `.h5.ts/.weapp.ts` 拆分。对于轻量差异，比如某个按钮只在微信端展示，可以用 `process.env.TARO_ENV`。CSS 上也会规避标签选择器、fixed 定位和根变量继承问题，因为 Taro 编译到 H5 和小程序后的真实 DOM/WXML 不一样。
